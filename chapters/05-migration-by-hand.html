5  Migration by Hand

The previous chapter described protocols as pairs of theories and promised that migration falls out automatically. That’s true, but abstractions are only useful if you can trace them through a concrete example. In this chapter, we will migrate a real schema change—adding a tags field to a Bluesky post—step by step, by hand, watching every piece of the machinery at work.

By the end, you will be able to look at any schema change and predict exactly what the migration engine will do. That predictability is the whole point: migration should never be surprising.

5.1 The Setup

We start with a minimal ATProto schema for a post. Version 1 has two fields: text (a string containing the post body) and createdAt (a datetime timestamp).

import { Panproto } from '@panproto/core';

const panproto = await Panproto.init();
const atproto = panproto.protocol('atproto');

// start snippet schema-v1
const schemaV1 = atproto.schema()
  .vertex('post', 'record', { nsid: 'app.bsky.feed.post' })
  .vertex('post:body', 'object')
  .vertex('post:body.text', 'string')
  .vertex('post:body.createdAt', 'string')
  .edge('post', 'post:body', 'record-schema')
  .edge('post:body', 'post:body.text', 'prop', { name: 'text' })
  .edge('post:body', 'post:body.createdAt', 'prop', { name: 'createdAt' })
  .build();
// end snippet schema-v1

// start snippet schema-v2
const schemaV2 = atproto.schema()
  .vertex('post', 'record', { nsid: 'app.bsky.feed.post' })
  .vertex('post:body', 'object')
  .vertex('post:body.text', 'string')
  .vertex('post:body.createdAt', 'string')
  .vertex('post:body.tags', 'array')
  .vertex('post:body.tags:item', 'string')
  .edge('post', 'post:body', 'record-schema')
  .edge('post:body', 'post:body.text', 'prop', { name: 'text' })
  .edge('post:body', 'post:body.createdAt', 'prop', { name: 'createdAt' })
  .edge('post:body', 'post:body.tags', 'prop', { name: 'tags' })
  .edge('post:body.tags', 'post:body.tags:item', 'items')
  .build();
// end snippet schema-v2

// start snippet migration
const migration = panproto.migration(schemaV1, schemaV2)
  .map('post', 'post')
  .map('post:body', 'post:body')
  .map('post:body.text', 'post:body.text')
  .map('post:body.createdAt', 'post:body.createdAt')
  .compile();
// end snippet migration

// start snippet lift
const inputRecord = {
  text: 'Hello, world!',
  createdAt: '2024-01-15T12:00:00Z',
};

const result = migration.lift(inputRecord);
// result.data: { text: 'Hello, world!', createdAt: '2024-01-15T12:00:00Z' }
// The tags field is absent (not mapped from v1).
// end snippet lift

This schema, viewed through the lens of Chapter 4, is a model of the ATProto schema theory: colimit(ThGraph, ThConstraint, ThMulti). Concretely, it’s a directed graph with:

  • Vertices: Post, Text, CreatedAt (each annotated with a kind—record, string, datetime).
  • Edges: text : Post → Text, createdAt : Post → CreatedAt (each annotated with multiplicity and constraints).

Now suppose we want version 2, which adds a tags field—an array of strings that let users categorize their posts.

  .edge('post:body', 'post:body.createdAt', 'prop', { name: 'createdAt' })
  .build();
// end snippet schema-v1

// start snippet schema-v2
const schemaV2 = atproto.schema()
  .vertex('post', 'record', { nsid: 'app.bsky.feed.post' })
  .vertex('post:body', 'object')
  .vertex('post:body.text', 'string')
  .vertex('post:body.createdAt', 'string')

The v2 schema has an additional vertex (Tags, with kind array<string>) and an additional edge (tags : Post → Tags, with multiplicity optional). Everything else is unchanged.

The question is: given a record conforming to v2, how do we produce a record conforming to v1?

NoteDirection of migration

Notice that we are migrating backward: from v2 data to v1 compatibility. This is the common direction in practice. A client that only understands v1 receives a v2 record and needs to make sense of it. The forward direction (v1 data to v2) requires defaults or explicit construction of the new fields, which is a different operation (the left adjoint \(\Sigma_F\) in the functor world, or a fill operation in the W-type world). We focus on the backward direction here.

5.2 The Migration Morphism

A migration from v1 to v2 is a schema morphism: a structure-preserving map from the v1 schema graph to the v2 schema graph. In panproto, this means specifying a vertex map and an edge map.

const migration = panproto.migration(schemaV1, schemaV2)
  .map('post', 'post')
  .map('post:body', 'post:body')
  .map('post:body.text', 'post:body.text')
  .map('post:body.createdAt', 'post:body.createdAt')
  .compile();

Let’s read this carefully:

Vertex map: each v1 vertex is mapped to the v2 vertex it corresponds to.

  • Post ↦ Post — the post record type is the same in both versions.
  • Text ↦ Text — the text type is the same.
  • CreatedAt ↦ CreatedAt — the datetime type is the same.

Edge map: each v1 edge is mapped to the v2 edge it corresponds to.

  • text ↦ text — the text field is the same.
  • createdAt ↦ createdAt — the createdAt field is the same.

Notice what is not in the migration: the Tags vertex and the tags edge. They exist in v2 but have no preimage in v1. This is the structural encoding of “v2 added a new field that v1 doesn’t know about.”

The migration morphism is an injection from the v1 schema into the v2 schema. Not all migrations are injections—you can merge vertices, collapse edges, remap types—but the “add a field” pattern always produces an injection.

5.3 What the Maps Mean

It’s worth dwelling on the semantics of these maps, because they carry more information than they might appear to at first glance.

The vertex map \(\phi_V : V_1 \to V_2\) says: “for each type in the old schema, here is the corresponding type in the new schema.” If a v1 vertex maps to a v2 vertex, the engine knows that data anchored to that vertex type can survive the migration. If a v2 vertex has no preimage under \(\phi_V\), the engine knows that data anchored to that vertex type must be introduced (in the forward direction) or discarded (in the backward direction).

The edge map \(\phi_E : E_1 \to E_2\) says: “for each field/relationship in the old schema, here is the corresponding field/relationship in the new schema.” This map must be compatible with the vertex map: if edge \(e_1\) goes from \(u_1\) to \(v_1\), then \(\phi_E(e_1)\) must go from \(\phi_V(u_1)\) to \(\phi_V(v_1)\). This compatibility condition ensures that the migration is structure-preserving—it doesn’t scramble the graph topology.

Together, \((\phi_V, \phi_E)\) form a graph homomorphism from the v1 schema to the v2 schema. In the language of (Spivak 2012), this is a functor between the schemas viewed as categories (though for ATProto, which uses a graph rather than a category as its schema theory, the precise statement is a graph homomorphism).

5.4 Compiling and Lifting

The raw migration morphism is a declaration of intent. Before it can be applied to data, it must be compiled: the engine pre-computes the data structures needed for efficient migration.

const inputRecord = {
  text: 'Hello, world!',
  createdAt: '2024-01-15T12:00:00Z',
};

const result = migration.lift(inputRecord);
// result.data: { text: 'Hello, world!', createdAt: '2024-01-15T12:00:00Z' }
// The tags field is absent (not mapped from v1).

The compile step analyzes the migration morphism and produces a compiled migration object. This object contains:

  • The surviving vertex set: the set of v2 vertices that are in the image of \(\phi_V\). In our example: {Post, Text, CreatedAt}.
  • The surviving edge set: the set of v2 edges that are in the image of \(\phi_E\). In our example: {text, createdAt}.
  • Remap tables: lookup structures mapping v2 vertices/edges back to their v1 preimages.
  • Constraint compatibility data: pre-checked results for whether constraints tighten or relax across the migration (more on this in Chapter 6).

The lift function takes a compiled migration and a v2 record, and produces a v1 record. “Lift” is the term we use for applying a migration to data in the backward direction (the term comes from the lifting property in category theory, though the analogy is loose).

Let’s trace through what lift does on a concrete record.

5.5 Tracing the Lift

Suppose we have a v2 record:

{
  "text": "Hello, world!",
  "createdAt": "2025-01-15T12:00:00Z",
  "tags": ["greeting", "test"]
}

This record, viewed as a W-type instance, is a tree:

The root node is anchored to the Post vertex. It has three children, each anchored to a different vertex and reached via a different edge.

Now the engine applies the lift. Because ATProto uses W-type instances, it runs the tree surgery pipeline:

5.5.1 Step 1: Signature Restriction

Remove nodes anchored to vertices not in the surviving vertex set. The surviving set is {Post, Text, CreatedAt}. The Tags vertex is not in this set, so the node anchored to Tags is removed:

5.5.2 Step 2: Reachability BFS

Starting from the root, walk the tree to find all surviving nodes that are still reachable. In this simple case, both remaining children are direct children of the root, so they’re trivially reachable. No nodes are lost.

In more complex migrations—where intermediate nodes are removed—this step can discover orphaned subtrees that need to be either discarded or reattached via ancestor contraction.

5.5.3 Step 3: Ancestor Contraction

When a removed node has surviving descendants, those descendants need to be reattached to the nearest surviving ancestor. In our example, the removed Tags node has no surviving descendants (it was a leaf), so contraction is a no-op.

We will see contraction do real work in Chapter 7, where we handle nested schema changes.

5.5.4 Step 4: Edge Resolution

Relabel edges according to the migration’s edge map. Since our edge map is the identity on surviving edges (text ↦ text, createdAt ↦ createdAt), the labels don’t change.

Edge resolution becomes interesting when the migration renames fields, or when ancestor contraction introduces new parent-child relationships that need to be labeled.

5.5.5 Step 5: Fan Reconstruction

Rebuild the children list for each surviving node. The root’s fan (its set of outgoing edges and their targets) is now {text → Text, createdAt → CreatedAt}, which is exactly the fan described by the v1 schema.

The result is a v1-compatible record:

{
  "text": "Hello, world!",
  "createdAt": "2025-01-15T12:00:00Z"
}

The tags field has been cleanly removed. The surviving fields are untouched.

5.6 What Happened Under the Hood

Let’s step back and name what just occurred. The migration morphism \((\phi_V, \phi_E) : S_1 \hookrightarrow S_2\) is an injection from the v1 schema into the v2 schema. The lift operation took a v2 instance (a tree) and produced a v1 instance (a smaller tree) by:

  1. Restricting to the image of \(\phi_V\) (keeping only the types that v1 knows about).
  2. Checking reachability (making sure nothing got disconnected).
  3. Contracting past removed intermediaries (not needed here).
  4. Resolving edge labels (trivial here).
  5. Rebuilding the fan structure (trivial here).

This five-step pipeline is the restriction operator for W-type instances that we mentioned in Chapter 4. For this particular migration—adding a leaf field—the pipeline reduces to “delete the new node.” But the same pipeline handles arbitrarily complex schema changes: renaming fields, restructuring nested objects, collapsing hierarchies, splitting types. The steps are always the same; only the work done in each step varies.

5.7 The Same Pattern in SQL

To build intuition across protocol boundaries, let’s see the same “add a column” migration in SQL, where the instance theory is ThFunctor instead of ThWType.

v1 schema (as a category):

  • Objects: posts (a table)
  • No non-identity morphisms (no foreign keys in this simple example)

v2 schema:

  • Objects: posts (same table, now with an additional tags column)

Migration: the functor \(F : \mathcal{S}_1 \to \mathcal{S}_2\) maps posts ↦ posts.

Instance: a v2 instance \(I : \mathcal{S}_2 \to \mathbf{Set}\) assigns to posts the set of all rows:

id text created_at tags
1 “Hello, world!” 2025-01-15 {“greeting”, “test”}
2 “Another post” 2025-01-16 {}

Lift (\(\Delta_F\)): the restricted instance \(\Delta_F(I) = I \circ F\) assigns to posts the same set of rows, but viewed through the v1 schema, which doesn’t include the tags column. The result is the projection:

id text created_at
1 “Hello, world!” 2025-01-15
2 “Another post” 2025-01-16

The tags column is dropped because v1’s schema doesn’t have a morphism corresponding to it.

NoteSame migration, different mechanics

The ATProto and SQL versions of “add a field” produce the same observable result: the new field is dropped when migrating backward. But the mechanism is different. ATProto prunes a tree node (W-type restriction). SQL projects away a column (functor precomposition). The migration morphism is the same kind of object in both cases—a structure-preserving map between schemas—but the restriction operator is determined by the instance theory.

This is the two-parameter architecture at work: the schema morphism is protocol-independent, but the data operation is protocol-specific, derived from the instance theory.

5.8 Composing Migrations

Migrations compose. If you have a migration \(m_{12} : S_1 \to S_2\) and a migration \(m_{23} : S_2 \to S_3\), their composite \(m_{13} = m_{23} \circ m_{12} : S_1 \to S_3\) is also a valid migration.

For our running example, suppose v3 adds a langs field (an array of language codes) on top of v2:

  • \(m_{12}\) : v1 → v2 (adds tags)
  • \(m_{23}\) : v2 → v3 (adds langs)
  • \(m_{13} = m_{23} \circ m_{12}\) : v1 → v3 (adds both tags and langs)

The composite migration’s vertex map sends each v1 vertex through both maps: \(\phi^{13}_V(v) = \phi^{23}_V(\phi^{12}_V(v))\). The edge map composes similarly.

Lifting through the composite is equivalent to lifting through each step in sequence:

\[\text{lift}(m_{13}, \text{record}_3) = \text{lift}(m_{12}, \text{lift}(m_{23}, \text{record}_3))\]

For functor instances, this is automatic: \(\Delta_{F \circ G} = \Delta_G \circ \Delta_F\) (precomposition is contravariantly functorial). For W-type instances, the engine must verify this property for each migration pair, but in practice it holds whenever both migrations pass the existence checks of Chapter 6.

Composability is what makes migration scalable. You don’t need a migration from every old version to every new version. You maintain a chain \(S_1 \to S_2 \to \cdots \to S_n\), and the engine composes as needed.

5.9 Migrations That Do More Than Add Fields

Adding a field is the simplest possible migration. Let’s briefly sketch what happens for more complex schema changes, to give you a sense of the pipeline’s range.

5.9.1 Renaming a Field

Suppose v2 renames text to body. The vertex map still sends Text ↦ Text (the type hasn’t changed), but the edge map sends text ↦ body. During edge resolution (step 4), the engine relabels the body edge back to text, producing a record that uses the old field name. The data is untouched; only the label changes.

5.9.2 Restructuring: Extracting a Nested Object

Suppose v1 has a flat post with text, createdAt, and authorName. v2 extracts authorName into a nested author object with a name field. The vertex map sends v1’s AuthorName to v2’s Name (inside the Author object). The edge map sends v1’s authorName to the composed path author.name.

During ancestor contraction (step 3), the engine recognizes that the Author vertex is an intermediate node not in v1’s schema. It contracts the path Post → Author → Name to Post → AuthorName, preserving the data while flattening the structure.

5.9.3 Removing a Required Field

Suppose v2 removes the createdAt field entirely. The vertex map doesn’t include CreatedAt, and the edge map doesn’t include createdAt. This migration might not exist—if v1 requires createdAt (it’s not optional), then there’s no valid way to produce a v1 record without it. The engine detects this during compilation and reports an existence failure. Chapter 6 covers the precise conditions.

5.10 The Lift as a Natural Transformation

For readers with a category theory background, there is a clean way to state what lift does. The lift operation is the component of a natural transformation between data presheaves, induced by the migration morphism.

In the functor case, this is literal. A migration \(F : \mathcal{S}_1 \to \mathcal{S}_2\) induces a functor \(\Delta_F : \mathbf{Set}^{\mathcal{S}_2} \to \mathbf{Set}^{\mathcal{S}_1}\) between presheaf categories. This functor sends each \(\mathcal{S}_2\)-instance to an \(\mathcal{S}_1\)-instance by precomposition. The lift of a single record is the action of \(\Delta_F\) on a representable presheaf (Spivak 2012).

In the W-type case, the connection is less direct but still meaningful. The tree surgery pipeline computes the action of a restriction functor on the polynomial functor associated with the schema. The mathematical details involve the theory of polynomial functors and their morphisms (Gambino and Hyland 2004; Abbott et al. 2005), which we defer to Appendix C.

For practical purposes, you don’t need the category theory to use panproto effectively. The compile + lift API is self-contained. But if you want to verify that a complex migration is correct, the categorical perspective gives you the tools: check that the morphism is a valid graph homomorphism, and the rest follows by functoriality.

5.11 A Note on Terminology

Different communities use different words for the operations we’ve described:

panproto’s terminology is closest to Litt et al. (2022)’s Cambria project, which was an important practical inspiration. The key difference is that Cambria hardcodes its schema language and migration semantics, while panproto parameterizes both by theories.

5.12 Summary

  • A migration is a structure-preserving map (vertex map + edge map) from an old schema to a new schema.
  • The migration is compiled to pre-compute surviving sets and remap tables.
  • The lift operation applies the compiled migration to a record, producing a record conforming to the old schema.
  • For W-type instances (ATProto, JSON, GraphQL), lift runs a 5-step tree surgery pipeline: signature restriction, reachability BFS, ancestor contraction, edge resolution, fan reconstruction.
  • For functor instances (SQL, CQL), lift is precomposition: pull rows through the vertex map.
  • Migrations compose: you maintain a chain of migrations, and the engine composes as needed.
  • The same migration morphism structure works across protocols; only the lift implementation varies, determined by the instance theory.

The raw migration morphism is a declaration of intent. Before it can be applied to data, it must be *compiled*: the engine pre-computes the data structures needed for efficient migration.

- **Remap tables**: lookup structures mapping v2 vertices/edges back to their v1 preimages.
- **Constraint compatibility data**: pre-checked results for whether constraints tighten or relax across the migration (more on this in Chapter 6).

The `lift` function takes a compiled migration and a v2 record, and produces a v1 record. "Lift" is the term we use for applying a migration to data in the backward direction (the term comes from the lifting property in category theory, though the analogy is loose).

Let's trace through what `lift` does on a concrete record.


## Tracing the Lift

Suppose we have a v2 record:

```json
{
  "text": "Hello, world!",
  "createdAt": "2025-01-15T12:00:00Z",
  "tags": ["greeting", "test"]
}
```

This record, viewed as a W-type instance, is a tree:

```{mermaid}
%%| label: fig-v2-tree
%%| fig-cap: "The v2 record as a W-type tree."
graph TD
    R["Post (root)"]
    T["Text: 'Hello, world!'"]
    C["CreatedAt: '2025-01-15T12:00:00Z'"]
    G["Tags: ['greeting', 'test']"]
    R -->|text| T
    R -->|createdAt| C
    R -->|tags| G
```

The root node is anchored to the `Post` vertex. It has three children, each anchored to a different vertex and reached via a different edge.

Now the engine applies the lift. Because ATProto uses W-type instances, it runs the tree surgery pipeline:

### Step 1: Signature Restriction

Remove nodes anchored to vertices *not* in the surviving vertex set. The surviving set is `{Post, Text, CreatedAt}`. The `Tags` vertex is not in this set, so the node anchored to `Tags` is removed:

```{mermaid}
%%| label: fig-after-restriction
%%| fig-cap: "After signature restriction: the Tags node is removed."
graph TD
    R["Post (root)"]
    T["Text: 'Hello, world!'"]
    C["CreatedAt: '2025-01-15T12:00:00Z'"]
    R -->|text| T
    R -->|createdAt| C
```

### Step 2: Reachability BFS

Starting from the root, walk the tree to find all surviving nodes that are still reachable. In this simple case, both remaining children are direct children of the root, so they're trivially reachable. No nodes are lost.

In more complex migrations---where intermediate nodes are removed---this step can discover orphaned subtrees that need to be either discarded or reattached via ancestor contraction.

### Step 3: Ancestor Contraction

When a removed node has surviving descendants, those descendants need to be reattached to the nearest surviving ancestor. In our example, the removed `Tags` node has no surviving descendants (it was a leaf), so contraction is a no-op.

We will see contraction do real work in Chapter 7, where we handle nested schema changes.

### Step 4: Edge Resolution

Relabel edges according to the migration's edge map. Since our edge map is the identity on surviving edges (`text ↦ text`, `createdAt ↦ createdAt`), the labels don't change.

Edge resolution becomes interesting when the migration renames fields, or when ancestor contraction introduces new parent-child relationships that need to be labeled.

### Step 5: Fan Reconstruction

Rebuild the children list for each surviving node. The root's fan (its set of outgoing edges and their targets) is now `{text → Text, createdAt → CreatedAt}`, which is exactly the fan described by the v1 schema.

The result is a v1-compatible record:

```json
{
  "text": "Hello, world!",
  "createdAt": "2025-01-15T12:00:00Z"
}
```

The `tags` field has been cleanly removed. The surviving fields are untouched.


## What Happened Under the Hood

Let's step back and name what just occurred. The migration morphism $(\phi_V, \phi_E) : S_1 \hookrightarrow S_2$ is an injection from the v1 schema into the v2 schema. The lift operation took a v2 instance (a tree) and produced a v1 instance (a smaller tree) by:

1. Restricting to the image of $\phi_V$ (keeping only the types that v1 knows about).
2. Checking reachability (making sure nothing got disconnected).
3. Contracting past removed intermediaries (not needed here).
4. Resolving edge labels (trivial here).
5. Rebuilding the fan structure (trivial here).

This five-step pipeline is the *restriction operator* for W-type instances that we mentioned in Chapter 4. For this particular migration---adding a leaf field---the pipeline reduces to "delete the new node." But the same pipeline handles arbitrarily complex schema changes: renaming fields, restructuring nested objects, collapsing hierarchies, splitting types. The steps are always the same; only the work done in each step varies.


## The Same Pattern in SQL

To build intuition across protocol boundaries, let's see the same "add a column" migration in SQL, where the instance theory is ThFunctor instead of ThWType.

**v1 schema** (as a category):

- Objects: `posts` (a table)
- No non-identity morphisms (no foreign keys in this simple example)

**v2 schema**:

- Objects: `posts` (same table, now with an additional `tags` column)

**Migration**: the functor $F : \mathcal{S}_1 \to \mathcal{S}_2$ maps `posts ↦ posts`.

**Instance**: a v2 instance $I : \mathcal{S}_2 \to \mathbf{Set}$ assigns to `posts` the set of all rows:

| id | text | created_at | tags |
|----|------|------------|------|
| 1 | "Hello, world!" | 2025-01-15 | {"greeting", "test"} |
| 2 | "Another post" | 2025-01-16 | {} |

**Lift** ($\Delta_F$): the restricted instance $\Delta_F(I) = I \circ F$ assigns to `posts` the *same* set of rows, but viewed through the v1 schema, which doesn't include the `tags` column. The result is the projection:

| id | text | created_at |
|----|------|------------|
| 1 | "Hello, world!" | 2025-01-15 |
| 2 | "Another post" | 2025-01-16 |

The `tags` column is dropped because v1's schema doesn't have a morphism corresponding to it.

:::{.callout-note}
## Same migration, different mechanics

The ATProto and SQL versions of "add a field" produce the same observable result: the new field is dropped when migrating backward. But the *mechanism* is different. ATProto prunes a tree node (W-type restriction). SQL projects away a column (functor precomposition). The migration morphism is the same kind of object in both cases---a structure-preserving map between schemas---but the restriction operator is determined by the instance theory.

This is the two-parameter architecture at work: the schema morphism is protocol-independent, but the data operation is protocol-specific, derived from the instance theory.
:::


## Composing Migrations

Migrations compose. If you have a migration $m_{12} : S_1 \to S_2$ and a migration $m_{23} : S_2 \to S_3$, their composite $m_{13} = m_{23} \circ m_{12} : S_1 \to S_3$ is also a valid migration.

For our running example, suppose v3 adds a `langs` field (an array of language codes) on top of v2:

- $m_{12}$ : v1 → v2 (adds `tags`)
- $m_{23}$ : v2 → v3 (adds `langs`)
- $m_{13} = m_{23} \circ m_{12}$ : v1 → v3 (adds both `tags` and `langs`)

The composite migration's vertex map sends each v1 vertex through both maps: $\phi^{13}_V(v) = \phi^{23}_V(\phi^{12}_V(v))$. The edge map composes similarly.

Lifting through the composite is equivalent to lifting through each step in sequence:

$$\text{lift}(m_{13}, \text{record}_3) = \text{lift}(m_{12}, \text{lift}(m_{23}, \text{record}_3))$$

For functor instances, this is automatic: $\Delta_{F \circ G} = \Delta_G \circ \Delta_F$ (precomposition is contravariantly functorial). For W-type instances, the engine must verify this property for each migration pair, but in practice it holds whenever both migrations pass the existence checks of Chapter 6.

Composability is what makes migration *scalable*. You don't need a migration from every old version to every new version. You maintain a chain $S_1 \to S_2 \to \cdots \to S_n$, and the engine composes as needed.


## Migrations That Do More Than Add Fields

Adding a field is the simplest possible migration. Let's briefly sketch what happens for more complex schema changes, to give you a sense of the pipeline's range.

### Renaming a Field

Suppose v2 renames `text` to `body`. The vertex map still sends `Text ↦ Text` (the type hasn't changed), but the edge map sends `text ↦ body`. During edge resolution (step 4), the engine relabels the `body` edge back to `text`, producing a record that uses the old field name. The data is untouched; only the label changes.

### Restructuring: Extracting a Nested Object

Suppose v1 has a flat post with `text`, `createdAt`, and `authorName`. v2 extracts `authorName` into a nested `author` object with a `name` field. The vertex map sends v1's `AuthorName` to v2's `Name` (inside the `Author` object). The edge map sends v1's `authorName` to the *composed path* `author.name`.

During ancestor contraction (step 3), the engine recognizes that the `Author` vertex is an intermediate node not in v1's schema. It contracts the path `Post → Author → Name` to `Post → AuthorName`, preserving the data while flattening the structure.

### Removing a Required Field

Suppose v2 removes the `createdAt` field entirely. The vertex map doesn't include `CreatedAt`, and the edge map doesn't include `createdAt`. This migration *might not exist*---if v1 requires `createdAt` (it's not optional), then there's no valid way to produce a v1 record without it. The engine detects this during compilation and reports an existence failure. Chapter 6 covers the precise conditions.


## The Lift as a Natural Transformation

For readers with a category theory background, there is a clean way to state what `lift` does. The lift operation is the component of a natural transformation between data presheaves, induced by the migration morphism.

In the functor case, this is literal. A migration $F : \mathcal{S}_1 \to \mathcal{S}_2$ induces a functor $\Delta_F : \mathbf{Set}^{\mathcal{S}_2} \to \mathbf{Set}^{\mathcal{S}_1}$ between presheaf categories. This functor sends each $\mathcal{S}_2$-instance to an $\mathcal{S}_1$-instance by precomposition. The lift of a single record is the action of $\Delta_F$ on a representable presheaf [@spivak2012].

In the W-type case, the connection is less direct but still meaningful. The tree surgery pipeline computes the action of a restriction functor on the polynomial functor associated with the schema. The mathematical details involve the theory of polynomial functors and their morphisms [@gambino2004; @abbott2005], which we defer to [Appendix C](../appendices/C-guide-to-the-literature.qmd).

For practical purposes, you don't need the category theory to use panproto effectively. The `compile` + `lift` API is self-contained. But if you want to verify that a complex migration is correct, the categorical perspective gives you the tools: check that the morphism is a valid graph homomorphism, and the rest follows by functoriality.


## A Note on Terminology

Different communities use different words for the operations we've described:

| panproto | CQL / Spivak | Cambria | SQL |
|----------|-------------|---------|-----|
| Migration morphism | Schema functor | Lens | ALTER TABLE + mapping |
| Compile | (implicit) | Compile lens | Query planning |
| Lift (backward) | $\Delta_F$ | Convert | View / projection |
| Fill (forward) | $\Sigma_F$ / $\Pi_F$ | (manual) | INSERT with defaults |

: Terminology across communities. {#tbl-terminology}

panproto's terminology is closest to @litt2022's Cambria project, which was an important practical inspiration. The key difference is that Cambria hardcodes its schema language and migration semantics, while panproto parameterizes both by theories.


## Summary

- A **migration** is a structure-preserving map (vertex map + edge map) from an old schema to a new schema.
- The migration is **compiled** to pre-compute surviving sets and remap tables.
- The **lift** operation applies the compiled migration to a record, producing a record conforming to the old schema.
- For **W-type instances** (ATProto, JSON, GraphQL), lift runs a 5-step tree surgery pipeline: signature restriction, reachability BFS, ancestor contraction, edge resolution, fan reconstruction.
- For **functor instances** (SQL, CQL), lift is precomposition: pull rows through the vertex map.
- Migrations **compose**: you maintain a chain of migrations, and the engine composes as needed.
- The same **migration morphism** structure works across protocols; only the **lift implementation** varies, determined by the instance theory.

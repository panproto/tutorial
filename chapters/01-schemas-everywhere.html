1  Schemas Everywhere

2 Schemas Everywhere

You already know what a schema is. You write them constantly—every time you define a database table, declare a protobuf message, or sketch out a GraphQL type. Schemas are the skeletons of structured data: they say what fields exist, what types those fields carry, and (sometimes) what constraints hold.

What you might not have noticed is how many schema languages there are, and how eerily similar they look once you squint past the syntax. This chapter makes that observation concrete. We’ll take a single, deliberately boring entity—a blog post—and write its schema in nine different formats drawn from panproto’s coverage of 54 protocols. Then we’ll show how panproto can construct the same schema using its TypeScript SDK, regardless of which protocol you’re targeting.

2.1 The Running Example: A Blog Post

Our blog post has three fields:

Field Type Required?
text string yes
createdAt datetime (string) yes
tags array of strings no

That’s it. No nested objects, no foreign keys, no clever tricks. The point isn’t the complexity of the entity—it’s the variety of ways we can describe it.

2.2 Five Representations

2.2.1 JSON Schema

JSON Schema is the lingua franca of HTTP APIs. You’ve seen it in OpenAPI specs, in configuration file validators, and in editor autocompletion engines.

{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id": "https://example.com/schemas/post.json",
  "type": "object",
  "required": ["text", "createdAt"],
  "properties": {
    "text": {
      "type": "string",
      "maxLength": 3000
    },
    "createdAt": {
      "type": "string",
      "format": "date-time"
    },
    "tags": {
      "type": "array",
      "items": {
        "type": "string"
      },
      "maxItems": 8
    }
  }
}

A few things to note: the top-level construct is an object with named properties. Each property has a type. Some properties appear in the required array. Additional constraints like maxLength and maxItems sit alongside the type declarations.

2.2.2 Protocol Buffers

Protobuf takes a different approach. Where JSON Schema is declarative and JSON-native, Protobuf is a compiled IDL designed for efficient binary serialization.

syntax = "proto3";

package example.blog;

import "google/protobuf/timestamp.proto";

message Post {
  string text = 1;
  google.protobuf.Timestamp created_at = 2;
  repeated string tags = 3;
}

The top-level construct is a message with numbered fields. Each field has a type and a field number (used for wire encoding). The repeated keyword marks tags as a list. There’s no direct equivalent of required in proto3—all fields have default values.

Notice what’s missing compared to JSON Schema: no maxLength, no maxItems. Protobuf doesn’t have a built-in constraint language. If you want validation, you reach for something like protoc-gen-validate or Buf’s protovalidate.

2.2.3 SQL DDL

SQL’s Data Definition Language is the oldest schema language most developers encounter. It predates JSON, Protobuf, and GraphQL by decades.

CREATE TABLE posts (
    id          BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    text        VARCHAR(3000)   NOT NULL,
    created_at  TIMESTAMPTZ     NOT NULL DEFAULT now(),
    tags        TEXT[]
);

Here the top-level construct is a table with named columns. Each column has a data type and optional constraints (NOT NULL, PRIMARY KEY, DEFAULT). The TEXT[] type for tags is PostgreSQL’s array syntax—other databases handle this differently (a separate post_tags join table, for instance).

SQL brings something the previous two formats lack: a notion of identity (PRIMARY KEY) and defaults (DEFAULT now()). These aren’t just descriptions of shape—they’re operational semantics baked into the schema.

2.2.4 GraphQL SDL

GraphQL’s Schema Definition Language is designed for API surfaces. It describes what a client can query, not how data is stored.

"""A blog post."""
type Post {
  text: String!
  createdAt: DateTime!
  tags: [String!]
}

The top-level construct is a type with named fields. Each field has a return type. The ! suffix means non-nullable (GraphQL’s version of required). [String!] means “a nullable list of non-nullable strings”—note how nullability is part of the type system, not a separate constraint.

GraphQL doesn’t have maxLength either. Validation happens in resolvers or directives, not in the SDL itself.

2.2.5 ATProto Lexicon

ATProto (the protocol behind Bluesky) uses Lexicon, a JSON-based schema language that borrows from JSON Schema but adds protocol-specific concepts like NSIDs and record types.

{
  "lexicon": 1,
  "id": "com.example.blog.post",
  "defs": {
    "main": {
      "type": "record",
      "description": "A blog post.",
      "key": "tid",
      "record": {
        "type": "object",
        "required": ["text", "createdAt"],
        "properties": {
          "text": {
            "type": "string",
            "maxLength": 3000,
            "maxGraphemes": 300
          },
          "createdAt": {
            "type": "string",
            "format": "datetime"
          },
          "tags": {
            "type": "array",
            "items": {
              "type": "string"
            },
            "maxLength": 8
          }
        }
      }
    }
  }
}

Lexicon looks a lot like JSON Schema, and that’s intentional—it was designed to be familiar. But notice the differences: there’s a record wrapper with a key strategy (tid means “timestamp-based ID”). The NSID (com.example.blog.post) serves as both an identifier and a namespace. maxGraphemes is an ATProto-specific constraint that counts Unicode grapheme clusters rather than code points.

2.3 Beyond Five: The Full Spectrum

Five formats illustrate the pattern, but panproto covers 54 protocols. To give a sense of the breadth, here are four more representations of the same blog post—drawn from serialization, type system, API specification, and document-exchange traditions respectively.

2.3.1 Avro

Apache Avro is the schema language of the Hadoop ecosystem and a common choice for Kafka message contracts. Schemas are themselves JSON documents, and types are defined recursively.

{
  "type": "record",
  "name": "Post",
  "namespace": "com.example.blog",
  "doc": "A blog post.",
  "fields": [
    {
      "name": "text",
      "type": "string",
      "doc": "Post body, maximum 3000 characters."
    },
    {
      "name": "createdAt",
      "type": {
        "type": "long",
        "logicalType": "timestamp-millis"
      }
    },
    {
      "name": "tags",
      "type": {
        "type": "array",
        "items": "string"
      },
      "default": []
    }
  ]
}

Avro’s top-level construct is a record with a list of fields. The type is either a primitive name ("string") or a nested type definition. Avro’s logicalType mechanism layers semantic meaning—here, a timestamp interpretation—on top of a physical primitive (long). The default on tags makes the field optional in the sense that a message without it is still schema-valid.

2.3.2 TypeScript

TypeScript interfaces are schema-like artifacts even though TypeScript is a programming language rather than a dedicated schema language. They’re the de facto contract layer for JavaScript ecosystems, and many tools (Zod, io-ts, TypeBox) treat them as first-class schema targets.

/** A blog post. */
export interface Post {
  /** Post body, maximum 3000 characters. */
  text: string;
  createdAt: string; // ISO 8601 date-time
  tags?: string[];
}

The top-level construct is an interface with named members. Types are TypeScript’s own type expressions. The ? suffix marks a member as optional (the TypeScript analog of “not required”). Comments carry the constraint information that the type system itself cannot express—maxLength and maxItems have no native representation in a TypeScript interface, though they can be encoded via branded types or validation library schemas.

2.3.3 OpenAPI

OpenAPI 3.1 is the dominant standard for describing REST APIs. A schema appears in two places: in the path item (describing the operation) and in the components section (defining reusable types). Here is the path for creating a post, plus the schema component it references.

paths:
  /posts:
    post:
      summary: Create a blog post
      requestBody:
        required: true
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/Post'
      responses:
        '201':
          description: Post created

components:
  schemas:
    Post:
      type: object
      required:
        - text
        - createdAt
      properties:
        text:
          type: string
          maxLength: 3000
        createdAt:
          type: string
          format: date-time
        tags:
          type: array
          items:
            type: string
          maxItems: 8

The schema portion of OpenAPI is essentially JSON Schema (OpenAPI 3.1 adopted the JSON Schema 2020-12 vocabulary), so the components/schemas/Post block looks identical to the JSON Schema example. What OpenAPI adds is the path layer: operations, HTTP methods, request bodies, and responses. The schema language describes shapes; the path layer describes the protocol contract around those shapes.

2.3.4 XML Schema Definition (XSD)

XSD is the W3C schema language for XML documents. It predates JSON Schema by more than a decade and remains the dominant schema mechanism in enterprise integration, document-exchange standards, and SOAP-based web services.

<?xml version="1.0" encoding="UTF-8"?>
<xs:schema xmlns:xs="http://www.w3.org/2001/XMLSchema"
           targetNamespace="https://example.com/blog"
           xmlns:blog="https://example.com/blog"
           elementFormDefault="qualified">

  <xs:element name="post" type="blog:PostType"/>

  <xs:complexType name="PostType">
    <xs:sequence>
      <xs:element name="text">
        <xs:simpleType>
          <xs:restriction base="xs:string">
            <xs:maxLength value="3000"/>
          </xs:restriction>
        </xs:simpleType>
      </xs:element>
      <xs:element name="createdAt" type="xs:dateTime"/>
      <xs:element name="tags" minOccurs="0">
        <xs:complexType>
          <xs:sequence>
            <xs:element name="tag" type="xs:string"
                        minOccurs="0" maxOccurs="8"/>
          </xs:sequence>
        </xs:complexType>
      </xs:element>
    </xs:sequence>
  </xs:complexType>

</xs:schema>

XSD’s vocabulary is more elaborate than any of the formats above. The top-level construct is a complexType containing a sequence of elements. Constraints are expressed as restrictions on base types—note how maxLength requires nesting a simpleType > restriction around the element type. Arrays are modeled as a containing element with repeated child elements (maxOccurs="8"), a fundamentally different representation from the type: array approach used by JSON-family formats. Namespaces pervade everything.

The verbosity is not accidental. XSD was designed for document exchange between organizations, where precise, version-stable, toolchain-independent contracts matter more than developer ergonomics.

2.4 The Structural Rhyme

Step back and look at these nine representations side by side. Despite the syntactic differences, they share a common anatomy:

Concept JSON Schema Protobuf SQL DDL GraphQL SDL ATProto Lexicon Avro TypeScript OpenAPI (schema) XSD
Container object message table type record > object record interface object complexType > sequence
Named member property field column field property field member property element
Member type type type data type return type type type type type type / simpleType
Required marker required[] (always set) NOT NULL ! suffix required[] (no default) (no ?) required[] minOccurs=“1”
Length constraint maxLength (external) VARCHAR(n) (external) maxLength (doc only) (doc only) maxLength xs:maxLength
List marker type: array repeated ARRAY / table [T] syntax type: array type: array T[] type: array maxOccurs > 1

Every row in that table describes the same structural idea wearing different clothes. A container holds named members. Each member has a type. Some members are required. Some have constraints.

This isn’t a coincidence. These formats all describe the same kind of thing: structured data with named, typed components. The differences are real—SQL has foreign keys, Protobuf has field numbers, GraphQL has nullability as a type feature, XSD has namespace machinery—but the similarities dominate.

2.5 Building Schemas with panproto

panproto takes this observation seriously. Instead of treating each schema language as a special case, it provides a single API that can target multiple protocols. You describe your intent once; panproto maps it to the target format’s native constructs.

Here’s what that looks like in practice using the panproto TypeScript SDK.

2.5.1 ATProto

The ATProto schema builder produces a Lexicon-compatible record definition. The same blog post entity, expressed through panproto’s API:

import { Panproto } from '@panproto/core';

// start snippet init
const panproto = await Panproto.init();
const atproto = panproto.protocol('atproto');
// end snippet init

// start snippet build-schema
const postSchema = atproto.schema()
  .vertex('post', 'record', { nsid: 'app.bsky.feed.post' })
  .vertex('post:body', 'object')
  .vertex('post:body.text', 'string')
  .vertex('post:body.createdAt', 'string')
  .edge('post', 'post:body', 'record-schema')
  .edge('post:body', 'post:body.text', 'prop', { name: 'text' })
  .edge('post:body', 'post:body.createdAt', 'prop', { name: 'createdAt' })
  .constraint('post:body.text', 'maxLength', '3000')
  .constraint('post:body.text', 'maxGraphemes', '300')
  .constraint('post:body.createdAt', 'format', 'datetime')
  .build();
// end snippet build-schema

The builder API mirrors the structure we saw in the Lexicon JSON above, but you get type checking, autocompletion, and—crucially—the ability to compose schemas programmatically. You’re not concatenating JSON strings; you’re building a typed data structure.

2.5.2 SQL

The SQL schema builder targets DDL. Same entity, different protocol:

import { Panproto } from '@panproto/core';

const panproto = await Panproto.init();

// start snippet build-schema
const sql = panproto.protocol('sql');

const postTable = sql.schema()
  .vertex('posts', 'table')
  .vertex('posts.id', 'integer')
  .vertex('posts.text', 'string')
  .vertex('posts.created_at', 'string')
  .edge('posts', 'posts.id', 'column', { name: 'id' })
  .edge('posts', 'posts.text', 'column', { name: 'text' })
  .edge('posts', 'posts.created_at', 'column', { name: 'created_at' })
  .constraint('posts.text', 'maxLength', '3000')
  .build();
// end snippet build-schema

The API surface is deliberately similar. You’re still declaring a container with named, typed members. The builder knows about SQL-specific concepts (column types, constraints, defaults) just as the ATProto builder knows about Lexicon concepts (NSIDs, key strategies, grapheme limits).

2.5.3 What Changed, What Didn’t

Compare the two code snippets. The structure is the same: create a builder, declare a container, add typed fields with constraints. The vocabulary differs: record vs. table, property vs. column, maxGraphemes vs. VARCHAR. And the output differs: one produces Lexicon JSON, the other produces SQL DDL.

This is the core value proposition of panproto: you learn the pattern once, then apply it across protocols. When a new schema language appears (and new ones appear regularly), you don’t start from scratch—you map its concepts to the pattern you already know.

2.6 What We’re Looking At

Let’s name what we’ve been seeing, without getting formal yet.

All nine schema formats describe vertices connected by edges:

  • In JSON Schema, the vertices are types (object, string, array) and the edges are properties linking an object to its field types.
  • In Protobuf, the vertices are messages and scalar types, and the edges are fields.
  • In SQL, the vertices are tables and data types, and the edges are columns (linking a table to the type of each column). Foreign keys add edges between tables.
  • In GraphQL, the vertices are types and the edges are fields.
  • In ATProto Lexicon, the vertices are records, objects, and primitive types, and the edges are properties.
  • In Avro, the vertices are records, arrays, and primitive types, and the edges are fields. Logical types annotate a vertex without changing the graph structure.
  • In TypeScript interfaces, the vertices are interfaces and primitive types, and the edges are members. Optional members (?) carry their own edge annotation.
  • In OpenAPI, the schema layer is structurally identical to JSON Schema; the additional path layer introduces operation vertices whose edges point into the schema graph via $ref.
  • In XSD, the vertices are complexTypes, simpleTypes, and built-in primitive types. The edges are element declarations within a sequence, and restrictions on simpleTypes add constraint annotations to edges.

Sometimes the edges carry extra information: a field number in Protobuf, a NOT NULL constraint in SQL, a maxLength in JSON Schema and Lexicon, a minOccurs/maxOccurs pair in XSD. But the fundamental picture is the same: a graph of types connected by typed relationships.

The differences between schema languages come down to:

  1. What kinds of vertices are allowed. SQL has tables and views. Protobuf has messages and enums. GraphQL has types, interfaces, unions, and enums. XSD has complexTypes, simpleTypes, and element declarations as first-class vertices.
  2. What kinds of edges are allowed. SQL columns are different from Protobuf fields, which are different from GraphQL fields, which are different from XSD element declarations within a sequence.
  3. What constraints can be attached. JSON Schema has a rich constraint vocabulary (maxLength, pattern, minimum). Protobuf has almost none built in. SQL constraints are operational (they’re enforced at write time). XSD has a structured restriction vocabulary that can express the same constraints as JSON Schema but requires significantly more nesting to do so.

These are not superficial syntactic differences. They reflect genuine design choices about what each format is optimized for. But they are parametric differences—variations on a common structural theme, not fundamentally different kinds of things.

That structural theme is what panproto is built on. The next chapter gives it a name.

2.7 Looking Ahead

We’ve established that schema languages share a common graph-like structure, and that panproto exploits this commonality. But we’ve been hand-waving about what “graph-like” means. The word “graph” is doing a lot of work, and it turns out that different schema languages need different kinds of graphs.

In Chapter 3, we’ll make this precise. We’ll see that some schemas are plain directed graphs, others are multigraphs, and still others need hypergraphs. These aren’t just academic distinctions—they determine what operations are well-defined on a schema, and they explain why single-protocol tools can’t easily generalize.

But first, take a moment to appreciate the view from here: nine different schema languages, nine different communities, nine different ecosystems—all describing the same blog post, all using the same structural pattern. And these nine are a sample: panproto’s full coverage spans 54 protocols, from serialization formats like MessagePack and CBOR to interface definition languages like Thrift and WSDL to domain-specific schema systems like FHIR and JSON-LD. The fact that a single structural account reaches across all of them is not an accident. It’s a consequence of what schemas are, and understanding that is the goal of this tutorial.

# Schemas Everywhere

You already know what a schema is. You write them constantly—every time you
define a database table, declare a protobuf message, or sketch out a GraphQL
type. Schemas are the skeletons of structured data: they say what fields exist,
what types those fields carry, and (sometimes) what constraints hold.

What you might not have noticed is how *many* schema languages there are, and
how eerily similar they look once you squint past the syntax. This chapter makes
that observation concrete. We'll take a single, deliberately boring entity—a
blog post—and write its schema in nine different formats drawn from panproto's
coverage of 54 protocols. Then we'll show how panproto can construct the same
schema using its TypeScript SDK, regardless of which protocol you're targeting.

## The Running Example: A Blog Post

Our blog post has three fields:

| Field       | Type              | Required? |
|-------------|-------------------|-----------|
| `text`      | string            | yes       |
| `createdAt` | datetime (string) | yes       |
| `tags`      | array of strings  | no        |

That's it. No nested objects, no foreign keys, no clever tricks. The point
isn't the complexity of the entity—it's the variety of ways we can describe it.


## Five Representations

### JSON Schema

JSON Schema is the lingua franca of HTTP APIs. You've seen it in OpenAPI specs,
in configuration file validators, and in editor autocompletion engines.

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id": "https://example.com/schemas/post.json",
  "type": "object",
  "required": ["text", "createdAt"],
  "properties": {
    "text": {
      "type": "string",
      "maxLength": 3000
    },
    "createdAt": {
      "type": "string",
      "format": "date-time"
    },
    "tags": {
      "type": "array",
      "items": {
        "type": "string"
      },
      "maxItems": 8
    }
  }
}
```

A few things to note: the top-level construct is an **object** with named
**properties**. Each property has a **type**. Some properties appear in the
**required** array. Additional constraints like `maxLength` and `maxItems` sit
alongside the type declarations.

### Protocol Buffers

Protobuf takes a different approach. Where JSON Schema is declarative and
JSON-native, Protobuf is a compiled IDL designed for efficient binary
serialization.

```protobuf
syntax = "proto3";

package example.blog;

import "google/protobuf/timestamp.proto";

message Post {
  string text = 1;
  google.protobuf.Timestamp created_at = 2;
  repeated string tags = 3;
}
```

The top-level construct is a **message** with numbered **fields**. Each field
has a **type** and a **field number** (used for wire encoding). The `repeated`
keyword marks `tags` as a list. There's no direct equivalent of `required` in
proto3—all fields have default values.

Notice what's missing compared to JSON Schema: no `maxLength`, no `maxItems`.
Protobuf doesn't have a built-in constraint language. If you want validation,
you reach for something like `protoc-gen-validate` or Buf's `protovalidate`.

### SQL DDL

SQL's Data Definition Language is the oldest schema language most developers
encounter. It predates JSON, Protobuf, and GraphQL by decades.

```sql
CREATE TABLE posts (
    id          BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    text        VARCHAR(3000)   NOT NULL,
    created_at  TIMESTAMPTZ     NOT NULL DEFAULT now(),
    tags        TEXT[]
);
```

Here the top-level construct is a **table** with named **columns**. Each column
has a **data type** and optional **constraints** (`NOT NULL`, `PRIMARY KEY`,
`DEFAULT`). The `TEXT[]` type for tags is PostgreSQL's array syntax—other
databases handle this differently (a separate `post_tags` join table, for
instance).

SQL brings something the previous two formats lack: a notion of **identity**
(`PRIMARY KEY`) and **defaults** (`DEFAULT now()`). These aren't just
descriptions of shape—they're operational semantics baked into the schema.

### GraphQL SDL

GraphQL's Schema Definition Language is designed for API surfaces. It describes
what a client can query, not how data is stored.

```graphql
"""A blog post."""
type Post {
  text: String!
  createdAt: DateTime!
  tags: [String!]
}
```

The top-level construct is a **type** with named **fields**. Each field has a
**return type**. The `!` suffix means non-nullable (GraphQL's version of
`required`). `[String!]` means "a nullable list of non-nullable strings"—note
how nullability is part of the type system, not a separate constraint.

GraphQL doesn't have `maxLength` either. Validation happens in resolvers or
directives, not in the SDL itself.

### ATProto Lexicon

ATProto (the protocol behind Bluesky) uses Lexicon, a JSON-based schema
language that borrows from JSON Schema but adds protocol-specific concepts like
NSIDs and record types.

```json
{
  "lexicon": 1,
  "id": "com.example.blog.post",
  "defs": {
    "main": {
      "type": "record",
      "description": "A blog post.",
      "key": "tid",
      "record": {
        "type": "object",
        "required": ["text", "createdAt"],
        "properties": {
          "text": {
            "type": "string",
            "maxLength": 3000,
            "maxGraphemes": 300
          },
          "createdAt": {
            "type": "string",
            "format": "datetime"
          },
          "tags": {
            "type": "array",
            "items": {
              "type": "string"
            },
            "maxLength": 8
          }
        }
      }
    }
  }
}
```

Lexicon looks a lot like JSON Schema, and that's intentional—it was designed to
be familiar. But notice the differences: there's a **record** wrapper with a
`key` strategy (`tid` means "timestamp-based ID"). The NSID
(`com.example.blog.post`) serves as both an identifier and a namespace.
`maxGraphemes` is an ATProto-specific constraint that counts Unicode grapheme
clusters rather than code points.


## Beyond Five: The Full Spectrum

Five formats illustrate the pattern, but panproto covers 54 protocols. To
give a sense of the breadth, here are four more representations of the same
blog post—drawn from serialization, type system, API specification, and
document-exchange traditions respectively.

### Avro

Apache Avro is the schema language of the Hadoop ecosystem and a common choice
for Kafka message contracts. Schemas are themselves JSON documents, and types
are defined recursively.

```json
{
  "type": "record",
  "name": "Post",
  "namespace": "com.example.blog",
  "doc": "A blog post.",
  "fields": [
    {
      "name": "text",
      "type": "string",
      "doc": "Post body, maximum 3000 characters."
    },
    {
      "name": "createdAt",
      "type": {
        "type": "long",
        "logicalType": "timestamp-millis"
      }
    },
    {
      "name": "tags",
      "type": {
        "type": "array",
        "items": "string"
      },
      "default": []
    }
  ]
}
```

Avro's top-level construct is a **record** with a list of **fields**. The
`type` is either a primitive name (`"string"`) or a nested type definition.
Avro's `logicalType` mechanism layers semantic meaning—here, a timestamp
interpretation—on top of a physical primitive (`long`). The `default` on
`tags` makes the field optional in the sense that a message without it is still
schema-valid.

### TypeScript

TypeScript interfaces are schema-like artifacts even though TypeScript is a
programming language rather than a dedicated schema language. They're the
de facto contract layer for JavaScript ecosystems, and many tools (Zod, io-ts,
TypeBox) treat them as first-class schema targets.

```typescript
/** A blog post. */
export interface Post {
  /** Post body, maximum 3000 characters. */
  text: string;
  createdAt: string; // ISO 8601 date-time
  tags?: string[];
}
```

The top-level construct is an **interface** with named **members**. Types are
TypeScript's own type expressions. The `?` suffix marks a member as optional
(the TypeScript analog of "not required"). Comments carry the constraint
information that the type system itself cannot express—`maxLength` and
`maxItems` have no native representation in a TypeScript interface, though they
can be encoded via branded types or validation library schemas.

### OpenAPI

OpenAPI 3.1 is the dominant standard for describing REST APIs. A schema
appears in two places: in the **path item** (describing the operation) and in
the **components** section (defining reusable types). Here is the path for
creating a post, plus the schema component it references.

```yaml
paths:
  /posts:
    post:
      summary: Create a blog post
      requestBody:
        required: true
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/Post'
      responses:
        '201':
          description: Post created

components:
  schemas:
    Post:
      type: object
      required:
        - text
        - createdAt
      properties:
        text:
          type: string
          maxLength: 3000
        createdAt:
          type: string
          format: date-time
        tags:
          type: array
          items:
            type: string
          maxItems: 8
```

The schema portion of OpenAPI is essentially JSON Schema (OpenAPI 3.1 adopted
the JSON Schema 2020-12 vocabulary), so the `components/schemas/Post` block
looks identical to the JSON Schema example. What OpenAPI adds is the **path
layer**: operations, HTTP methods, request bodies, and responses. The schema
language describes shapes; the path layer describes the protocol contract around
those shapes.

### XML Schema Definition (XSD)

XSD is the W3C schema language for XML documents. It predates JSON Schema by
more than a decade and remains the dominant schema mechanism in enterprise
integration, document-exchange standards, and SOAP-based web services.

```xml
<?xml version="1.0" encoding="UTF-8"?>
<xs:schema xmlns:xs="http://www.w3.org/2001/XMLSchema"
           targetNamespace="https://example.com/blog"
           xmlns:blog="https://example.com/blog"
           elementFormDefault="qualified">

  <xs:element name="post" type="blog:PostType"/>

  <xs:complexType name="PostType">
    <xs:sequence>
      <xs:element name="text">
        <xs:simpleType>
          <xs:restriction base="xs:string">
            <xs:maxLength value="3000"/>
          </xs:restriction>
        </xs:simpleType>
      </xs:element>
      <xs:element name="createdAt" type="xs:dateTime"/>
      <xs:element name="tags" minOccurs="0">
        <xs:complexType>
          <xs:sequence>
            <xs:element name="tag" type="xs:string"
                        minOccurs="0" maxOccurs="8"/>
          </xs:sequence>
        </xs:complexType>
      </xs:element>
    </xs:sequence>
  </xs:complexType>

</xs:schema>
```

XSD's vocabulary is more elaborate than any of the formats above. The
top-level construct is a **complexType** containing a **sequence** of
**elements**. Constraints are expressed as **restrictions** on base types—note
how `maxLength` requires nesting a `simpleType > restriction` around the
element type. Arrays are modeled as a containing element with repeated child
elements (`maxOccurs="8"`), a fundamentally different representation from the
`type: array` approach used by JSON-family formats. Namespaces pervade
everything.

The verbosity is not accidental. XSD was designed for document exchange
between organizations, where precise, version-stable, toolchain-independent
contracts matter more than developer ergonomics.


## The Structural Rhyme

Step back and look at these nine representations side by side. Despite the
syntactic differences, they share a common anatomy:

| Concept            | JSON Schema   | Protobuf       | SQL DDL         | GraphQL SDL   | ATProto Lexicon  | Avro          | TypeScript    | OpenAPI (schema) | XSD                    |
|--------------------|---------------|----------------|-----------------|---------------|------------------|---------------|---------------|------------------|------------------------|
| Container          | object        | message        | table           | type          | record > object  | record        | interface     | object           | complexType > sequence |
| Named member       | property      | field          | column          | field         | property         | field         | member        | property         | element                |
| Member type        | type          | type           | data type       | return type   | type             | type          | type          | type             | type / simpleType      |
| Required marker    | required[]    | (always set)   | NOT NULL        | `!` suffix    | required[]       | (no default)  | (no `?`)      | required[]       | minOccurs="1"          |
| Length constraint  | maxLength     | (external)     | VARCHAR(n)      | (external)    | maxLength        | (doc only)    | (doc only)    | maxLength        | xs:maxLength           |
| List marker        | type: array   | repeated       | ARRAY / table   | `[T]` syntax  | type: array      | type: array   | `T[]`         | type: array      | maxOccurs > 1          |

Every row in that table describes the same structural idea wearing different
clothes. A **container** holds **named members**. Each member has a **type**.
Some members are **required**. Some have **constraints**.

This isn't a coincidence. These formats all describe the same kind of thing:
structured data with named, typed components. The differences are real—SQL has
foreign keys, Protobuf has field numbers, GraphQL has nullability as a type
feature, XSD has namespace machinery—but the similarities dominate.

## What We're Looking At

Let's name what we've been seeing, without getting formal yet.

All nine schema formats describe **vertices** connected by **edges**:

- In JSON Schema, the vertices are types (`object`, `string`, `array`) and the
  edges are properties linking an object to its field types.
- In Protobuf, the vertices are messages and scalar types, and the edges are
  fields.
- In SQL, the vertices are tables and data types, and the edges are columns
  (linking a table to the type of each column). Foreign keys add edges between
  tables.
- In GraphQL, the vertices are types and the edges are fields.
- In ATProto Lexicon, the vertices are records, objects, and primitive types,
  and the edges are properties.
- In Avro, the vertices are records, arrays, and primitive types, and the edges
  are fields. Logical types annotate a vertex without changing the graph
  structure.
- In TypeScript interfaces, the vertices are interfaces and primitive types, and
  the edges are members. Optional members (`?`) carry their own edge annotation.
- In OpenAPI, the schema layer is structurally identical to JSON Schema; the
  additional path layer introduces operation vertices whose edges point into the
  schema graph via `$ref`.
- In XSD, the vertices are complexTypes, simpleTypes, and built-in primitive
  types. The edges are element declarations within a sequence, and restrictions
  on simpleTypes add constraint annotations to edges.

Sometimes the edges carry extra information: a field number in Protobuf, a
`NOT NULL` constraint in SQL, a `maxLength` in JSON Schema and Lexicon, a
`minOccurs`/`maxOccurs` pair in XSD. But the
fundamental picture is the same: **a graph of types connected by typed
relationships**.

The differences between schema languages come down to:

1. **What kinds of vertices are allowed.** SQL has tables and views. Protobuf
   has messages and enums. GraphQL has types, interfaces, unions, and enums.
   XSD has complexTypes, simpleTypes, and element declarations as first-class
   vertices.
2. **What kinds of edges are allowed.** SQL columns are different from Protobuf
   fields, which are different from GraphQL fields, which are different from XSD
   element declarations within a sequence.
3. **What constraints can be attached.** JSON Schema has a rich constraint
   vocabulary (`maxLength`, `pattern`, `minimum`). Protobuf has almost none
   built in. SQL constraints are operational (they're enforced at write time).
   XSD has a structured restriction vocabulary that can express the same
   constraints as JSON Schema but requires significantly more nesting to do so.

These are not superficial syntactic differences. They reflect genuine design
choices about what each format is optimized for. But they are **parametric**
differences—variations on a common structural theme, not fundamentally different
kinds of things.

That structural theme is what panproto is built on. The next chapter gives it a
name.


## Looking Ahead

We've established that schema languages share a common graph-like structure,
and that panproto exploits this commonality. But we've been hand-waving about
what "graph-like" means. The word "graph" is doing a lot of work, and it turns
out that different schema languages need different *kinds* of graphs.

In @sec-shape-of-schemas, we'll make this precise. We'll see that some schemas
are plain directed graphs, others are multigraphs, and still others need
hypergraphs. These aren't just academic distinctions—they determine what
operations are well-defined on a schema, and they explain why single-protocol
tools can't easily generalize.

But first, take a moment to appreciate the view from here: nine different schema
languages, nine different communities, nine different ecosystems—all describing
the same blog post, all using the same structural pattern. And these nine are a
sample: panproto's full coverage spans 54 protocols, from serialization formats
like MessagePack and CBOR to interface definition languages like Thrift and
WSDL to domain-specific schema systems like FHIR and JSON-LD. The fact that a
single structural account reaches across all of them is not an accident. It's a
consequence of what schemas *are*, and understanding that is the goal of this
tutorial.

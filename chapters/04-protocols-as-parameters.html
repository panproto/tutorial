4  Protocols as Parameters

In the previous chapter, we saw that a theory defines an interface: a collection of sorts, operations, and axioms that any conforming model must satisfy. Theories let us talk about “categories,” “graphs,” “constraints,” and so on without committing to a particular representation. In this chapter, we put that machinery to work by asking a sharper question: what is a protocol?

The answer in panproto is surprisingly concise. A protocol is a pair of theories—one controlling the shape of schemas, and one controlling the shape of data. Everything else—migration, diffing, breaking-change detection, bidirectional lenses—is derived from that pair by mechanical reasoning. You do not write protocol-specific logic. You declare two theories and let the engine do the rest.

panproto currently supports 54 protocols organized into five theory groups, each defined by a characteristic schema theory and instance theory combination.

4.1 The Two-Parameter Architecture

Every protocol that panproto supports is described by exactly two parameters:

  1. Schema theory: the algebraic theory whose models are schemas. This determines what “a schema” looks like—whether it has vertices and edges (a graph), vertices and hyperedges (a hypergraph), constraints, interfaces, or some combination of these.

  2. Instance theory: the algebraic theory whose models are data instances conforming to a schema. This determines what “a record” or “a row” looks like once you have a schema in hand.

The following table catalogues all 54 protocols and their decomposition into this two-parameter form. The protocols are grouped by their shared (schema theory, instance theory) signature; within each group every row is an instance of the same theoretical machinery. Each row is a protocol fully characterized by its schema theory, its instance theory, and a restriction operator that implements migration.

A few things to notice:

  • Schema theories are colimits. Most real-world protocols combine several concerns—graphs with constraints, multigraph structure with interfaces. The schema theory is assembled by taking a colimit (a structured union) of simpler component theories. This is exactly the colimit construction from Chapter 3.

  • Instance theories cluster into three families. There are functor-based instances (ThFunctor), tree-based instances (ThWType), and flat instances (ThFlat). This is the deepest structural split in the table, and it determines how migration works at the data level. ThFlat is best understood as a degenerate case of ThWType restricted to depth-1 trees.

  • The restriction operator is determined by the instance theory. Functor instances always migrate via precomposition (\(\Delta_F\)). W-type instances always migrate via the signature-restriction pipeline. Flat instances use the simpler field-projection operator, which is the depth-1 specialization of tree surgery. You don’t choose a migration strategy; it follows from the theory.

4.2 Three Families of Instances

The instance theory is where the rubber meets the road. It determines what “data” means for a given schema, and therefore what “migrating data” means. Let’s look at the three families in detail.

4.2.1 Set-Valued Functors

The functor model comes from the categorical databases tradition, developed most thoroughly by Spivak (2012) and formalized in the CQL system by Schultz et al. (2017).

The idea is direct. A schema is a category \(\mathcal{C}\)—objects are tables (or types), morphisms are foreign keys (or relationships). An instance of that schema is a functor \(I : \mathcal{C} \to \mathbf{Set}\). In concrete terms:

  • Each table \(T\) in the schema maps to a set \(I(T)\)—the rows of that table.
  • Each foreign key \(f : T_1 \to T_2\) maps to a function \(I(f) : I(T_1) \to I(T_2)\)—the lookup that sends each row of \(T_1\) to its corresponding row in \(T_2\).

This is a faithful model of relational data. The functoriality condition (\(I(g \circ f) = I(g) \circ I(f)\)) ensures that following two foreign keys in sequence gives the same result whether you compose them first or chase them one at a time. Referential integrity is built into the mathematics.

Migration under the functor model is precomposition. Given a schema morphism \(F : \mathcal{C}_1 \to \mathcal{C}_2\) (a functor mapping the old schema into the new one), and an instance \(I : \mathcal{C}_2 \to \mathbf{Set}\) living over the new schema, you get an instance \(\Delta_F(I) = I \circ F : \mathcal{C}_1 \to \mathbf{Set}\) over the old schema. In operational terms: for each old table \(T\), you look up which new table \(F(T)\) it maps to, and pull back that table’s rows. Foreign keys compose automatically.

This is the pullback migration functor \(\Delta_F\) from (Spivak 2012). It has left and right adjoints (\(\Sigma_F\) and \(\Pi_F\)) that handle the “push data forward” direction, but for now, \(\Delta_F\) is the one that matters for understanding the restriction column of the protocol table.

4.2.2 W-Types: Tree-Shaped Data

The functor model works beautifully for relational data, but most modern protocols—ATProto, GraphQL, JSON Schema, TypeScript, and the majority of the 54 protocols panproto supports—traffic in tree-shaped data. A Bluesky post is not a row in a table; it is a JSON document with nested objects, arrays, and references. The right mathematical model for this kind of data is a W-type (wellfounded tree), studied by Gambino and Hyland (2004) in the context of dependent type theory and related to the containers of Abbott et al. (2005).

A W-type instance over a schema is a finitely-branching tree where:

  • Each node is anchored to a schema vertex. The root might be anchored to a Post vertex, a child to a Text vertex, a sibling to a CreatedAt vertex.
  • Each branch (parent-to-child edge) is labeled by a schema edge. The edge from Post to Text might be labeled text, the edge from Post to CreatedAt might be labeled createdAt.
  • The tree is wellfounded. No infinite descending chains. Every path from root to leaf is finite.

This is exactly the shape of a JSON document, an ATProto record, or a GraphQL response. The schema tells you which node types exist and which edges connect them. The instance is a particular tree that conforms to those constraints.

Migration under the W-type model is more involved than precomposition, because you can’t just “pull back” a tree through a functor. Instead, the engine runs a multi-step pipeline:

  1. Signature restriction: remove nodes anchored to vertices not in the image of the migration map.
  2. Reachability BFS: starting from the root, find all nodes still reachable after the pruning.
  3. Ancestor contraction: when intermediate nodes are removed, reconnect their children to the nearest surviving ancestor.
  4. Edge resolution: relabel branches according to the migration’s edge map, resolving ambiguities where necessary.
  5. Fan reconstruction: rebuild the branching structure of the tree to maintain wellfoundedness.

We will walk through this pipeline concretely in Chapter 7. For now, the key point is that this pipeline is derived from the theory pair. The engine knows it needs to do tree surgery (rather than precomposition) because the instance theory is ThWType rather than ThFunctor.

4.2.3 Flat Instances: Depth-1 Trees

ThFlat is not an independent theory so much as a disciplined restriction of ThWType. When the schema is a simple graph (no parallel edges, no recursive types) and instances are constrained to trees of depth at most 1, the full W-type machinery reduces to a much simpler operation: field projection.

A flat instance over a simple-graph schema is a collection of key-value records where each key is a schema edge and each value is a leaf datum. Because the tree never nests beyond depth 1, contraction never fires, reachability is trivial, and edge resolution degenerates to a straightforward lookup. The restriction operator collapses to: keep the fields whose schema edges appear in the image of the migration map; discard the rest.

This simplification is not merely computational. ThFlat captures a genuine semantic boundary: protocols like Protobuf, Avro, and Thrift are deliberately flat. They are designed for high-throughput serialization where recursive nesting would impose unacceptable overhead. Treating ThFlat as a proper theory (rather than a naming convention) ensures that panproto can enforce the depth-1 constraint at the theory level and reject schemas that would violate it.

The relationship is:

\[\text{ThFlat} \hookrightarrow \text{ThWType}\]

where the inclusion is a theory morphism that restricts the tree-height axioms. A ThFlat instance is a ThWType instance; the converse does not hold.

4.3 Comparing the Three Models

It’s worth pausing to see these three instance theories side by side, because the choice among them is the central architectural decision that distinguishes protocol families.

None of these models is “better” in the abstract. The functor model has superior compositional properties—migration functors compose, and the adjoint triple gives you round-tripping for free. But the functor model demands that your data is tabular, which is a poor fit for document-oriented protocols. The W-type model handles trees natively, at the cost of more complex migration logic and fewer automatic guarantees. ThFlat threads a middle path: it gives up the generality of arbitrary nesting in exchange for a migration operator so simple it is nearly free to compute.

panproto’s contribution is that you don’t have to choose one model for all protocols. Each protocol declares its own instance theory, and the engine adapts accordingly. SQL gets precomposition. ATProto gets tree surgery. Protobuf gets field projection. The outer framework—schema morphisms, existence checks, breaking-change detection—works uniformly across all three.

4.4 Building a Protocol from Theories

To make this concrete, let’s look at how a protocol is actually defined in panproto’s TypeScript API.

import { Panproto } from '@panproto/core';

const panproto = await Panproto.init();

// start snippet define-protocol
const tomlConfig = panproto.defineProtocol({
  name: 'toml-config',
  schemaTheory: 'ThConstrainedGraph',
  instanceTheory: 'ThWType',
  edgeRules: [
    { edgeKind: 'key', srcKinds: ['section'], tgtKinds: [] },
    { edgeKind: 'subsection', srcKinds: ['section'], tgtKinds: ['section'] },
  ],
  objKinds: ['section'],
  constraintSorts: ['required', 'deprecated'],
});
// end snippet define-protocol

// start snippet use-protocol
const configSchema = tomlConfig.schema()
  .vertex('root', 'section')
  .vertex('root.database', 'section')
  .vertex('root.database.host', 'string')
  .vertex('root.database.port', 'integer')
  .edge('root', 'root.database', 'subsection', { name: 'database' })
  .edge('root.database', 'root.database.host', 'key', { name: 'host' })
  .edge('root.database', 'root.database.port', 'key', { name: 'port' })
  .constraint('root.database.host', 'required', 'true')
  .build();
// end snippet use-protocol

That’s it. A protocol is a record with two fields: a schema theory and an instance theory. The engine reads the theory structure—what sorts exist, what operations connect them, what axioms constrain them—and derives:

  • The migration type: what data a migration must carry (vertex map, edge map, constraint map, …).
  • The restriction operator: how to apply a migration to an instance (precomposition, tree surgery, or field projection).
  • The existence checks: which conditions a proposed migration must satisfy to be valid (Chapter 6).
  • The breaking-change classifier: which migrations are safe and which are breaking (Chapter 9).
NoteThe key insight

Adding a new schema language to panproto means writing a few lines of data. You are not writing a parser, a differ, a migration engine, or a breaking-change detector. You declare two theories, and the engine derives everything else.

4.5 Why Colimits for Schema Theories?

You may have noticed that most schema theories in Table 4.1 are colimits of simpler theories. This is not an accident. Real-world protocols are assembled from orthogonal concerns:

  • Graph structure: vertices and directed edges (ThGraph).
  • Constraint structure: invariants on vertex/edge data (ThConstraint).
  • Multiplicity structure: arrays, optional fields, repeated edges (ThMulti).
  • Interface structure: abstract types implemented by concrete ones (ThInterface).
  • Hyperedge structure: edges connecting more than two vertices simultaneously (ThHyperEdge).
  • Metadata structure: out-of-band annotation layers attached to the schema or instance (ThMeta).

These concerns are largely independent of one another. A protocol might need graphs and constraints but not interfaces (ATProto). Another might need graphs, constraints, multiplicity, and interfaces (GraphQL). Still another might need the full multigraph combination plus a metadata layer, as in the tree-plus-metadata protocols of Group E. The colimit construction lets you assemble exactly the combination you need, with the categorical machinery ensuring that the pieces fit together coherently.

For example, ATProto’s schema theory is:

\[\text{ThATProtoSchema} = \text{colim}(\text{ThGraph}, \text{ThConstraint}, \text{ThMulti})\]

This says: an ATProto schema is a structure that is simultaneously a graph, a constrained structure, and a multiplicity-aware structure, with the overlapping parts (the shared sorts and operations) identified. The colimit handles the identification automatically.

SQL’s schema theory replaces ThGraph with ThHypergraph (because SQL tables have multi-column keys and compound foreign keys) and drops ThMulti (because SQL doesn’t have native array columns in the same sense):

\[\text{ThSQLSchema} = \text{colim}(\text{ThHypergraph}, \text{ThConstraint})\]

GraphQL adds ThInterface on top of ATProto’s combination:

\[\text{ThGraphQLSchema} = \text{colim}(\text{ThGraph}, \text{ThConstraint}, \text{ThMulti}, \text{ThInterface})\]

Group E protocols—XML/XSD, HTML, Neo4j, and their kin—add ThMeta to the constrained multigraph base. This captures the fact that these formats carry an additional annotation layer (XML attributes, HTML metadata, Neo4j property maps) that sits alongside the tree structure rather than inside it:

\[\text{ThXMLSchema} = \text{colim}(\text{ThGraph}, \text{ThConstraint}, \text{ThMulti}, \text{ThMeta})\]

The instance theory for Group E protocols is correspondingly \(\text{colim}(\text{ThWType}, \text{ThMeta})\): a wellfounded tree together with a metadata functor that annotates nodes with out-of-band data. Migration for these protocols runs the standard W-type pipeline on the tree component and a separate restriction pass on the metadata component, with the two passes coupled at the nodes they share.

The pattern is modular and extensible. If tomorrow a new protocol emerges that needs graph structure, constraints, and some novel “versioning” concern, you write ThVersioning as a new theory and take the colimit with the existing pieces. The engine will derive migration, existence checks, and breaking-change detection for the new combination without any protocol-specific code.

4.6 The Restrict Column

The rightmost column of Table 4.1, labeled “Restrict,” names the operation that applies a migration to data. This column is determined by the instance theory—you never choose it independently.

4.6.1 Precomposition (\(\Delta_F\))

When the instance theory is ThFunctor, the restriction operator is precomposition. Given:

  • A schema morphism \(F : \mathcal{S}_1 \to \mathcal{S}_2\)
  • An instance \(I : \mathcal{S}_2 \to \mathbf{Set}\)

the restricted instance is \(\Delta_F(I) = I \circ F\). This is the pullback functor from (Spivak 2012).

In practice, precomposition is almost trivially simple to implement. For each vertex \(v\) in \(\mathcal{S}_1\), look up \(F(v)\) in \(\mathcal{S}_2\) and take the data from that vertex. For each edge \(e\) in \(\mathcal{S}_1\), look up \(F(e)\) and use that function to chase foreign keys. The functoriality of \(F\) guarantees that everything is consistent.

4.6.2 Signature Restriction + BFS + Contraction

When the instance theory is ThWType, the restriction operator is a multi-step pipeline. We gave a brief overview in Section 4.2.2; Chapter 7 provides the full algorithmic detail. The key steps are:

  1. Signature restriction: discard tree nodes whose anchor vertex is not in the image of the migration’s vertex map.
  2. Reachability BFS: from the root, walk the tree to find which surviving nodes are still connected.
  3. Ancestor contraction: if a removed node had children that survived, reattach those children to the nearest surviving ancestor.
  4. Edge resolution: relabel parent-child edges according to the migration’s edge map. If the schema allows parallel edges (multigraphs), this step may require resolver data to disambiguate.
  5. Fan reconstruction: rebuild the node’s children list to reflect the new edge labels and ordering.

This pipeline is more expensive than precomposition, but it handles the structural complexity of tree-shaped data. Every step is necessary: skip reachability and you’ll keep orphaned subtrees; skip contraction and you’ll lose data that should survive; skip resolution and you’ll have ambiguous edge labels.

4.6.3 Field Projection

When the instance theory is ThFlat, the restriction operator reduces to field projection: keep the fields that the migration maps, drop the rest. As noted in Section 4.2.3, this is the depth-1 degenerate case of the W-type pipeline—the BFS is trivial (root plus leaves only), contraction never fires, and fan reconstruction is a simple list filter. Group C protocols (Protobuf, Avro, Thrift, Cap’n Proto, FlatBuffers, ASN.1, Bond, Redis, HCL) all use this operator.

4.7 From Table to Engine

The protocol table is not just a reference chart. It is a specification that the engine reads at startup. When you instantiate panproto for a particular protocol, the engine:

  1. Parses the schema theory to determine what sorts (Vertex, Edge, Constraint, …) exist and what operations connect them.
  2. Parses the instance theory to determine the data representation (functor, W-type, or flat).
  3. Derives the migration type from the schema theory’s sorts and operations. A migration must carry a map for each sort in the schema theory.
  4. Derives the restriction operator from the instance theory. Functor → precomposition. W-type → tree surgery. Flat → field projection.
  5. Derives existence checks from the schema theory’s axioms (Chapter 6).
  6. Derives the breaking-change classifier from the interaction of schema theory and instance theory (Chapter 9).

Steps 3–6 are fully automatic. The protocol author provides the two theories; the engine provides the rest.

This is the sense in which panproto is protocol-parametric. The protocol is a parameter—literally, a pair of theory values passed to the engine. Swap the parameter, and you get a different protocol’s migration system, with correct semantics, for free.

4.8 Protocols as a Design Space

The two-parameter architecture gives us a way to think about the design space of all 54 protocols. Each point in the space is a (schema theory, instance theory) pair; the five theory groups correspond to five clusters of points.

This framing surfaces some natural questions:

  • What separates Group A from Group D? Both use ThWType instances over constrained multigraph schemas. The distinguishing feature is ThInterface: Group D protocols treat abstract types (interfaces, traits, type classes) as first-class schema citizens, whereas Group A protocols do not.
  • What makes Group E distinct from Group A? Group E adds ThMeta to the schema colimit and correspondingly lifts the instance theory to \(\text{colim}(\text{ThWType}, \text{ThMeta})\). The metadata layer is not just an annotation convention; it participates in migration.
  • What lives between Group B and Group A? A protocol with hypergraph schemas but W-type instances—perhaps a document database with compound keys and nested values. panproto can accommodate such a protocol by combining ThHypergraph, ThConstraint, and ThWType in the obvious way.
  • What lies beyond Group D? Add a ThVersioning theory to the schema colimit, and you get a protocol with built-in schema versioning at the theory level. The engine will derive migration and breaking-change detection for it automatically.
  • Can you mix instance theories? In principle, a protocol could use functor instances for some schema sorts and W-type instances for others. panproto doesn’t support this yet, but the theory permits it.

The design space is open. panproto gives you the coordinates.

4.9 Historical Context

The idea that data migration can be parameterized by algebraic structure has roots in several traditions.

Spivak (2012) showed that functorial data migration—using functors between categories as schema morphisms—gives a clean, compositional account of SQL-style migration. The CQL project (Schultz et al. 2017) turned this into a practical tool for relational databases, demonstrating that the category-theoretic framework could handle real workloads.

The W-type perspective comes from dependent type theory. Gambino and Hyland (2004) showed that wellfounded trees (W-types) are the natural inductive types in categories with dependent products. Abbott et al. (2005) connected this to the theory of containers, which views tree-shaped data as polynomial functors. panproto uses this connection: a W-type instance is a section of a polynomial functor determined by the schema.

The insight that these traditions can be unified under a common framework—that functor instances, W-type instances, and flat instances all arise as models of different instance theories, with migration derived from the theory structure, and that 54 real-world protocols fall cleanly into five theory groups under this classification—is, as far as we know, original to panproto.

4.10 Looking Ahead

We now have the vocabulary to describe any protocol as a point in a two-dimensional space, and a catalogue of 54 protocols organized into five theory groups that illustrate how that space is populated. In the next chapter, we’ll use this vocabulary to do something concrete: migrate data by hand, step by step, watching the restriction operator at work on a real schema change. This will ground the abstractions of this chapter in the specifics of ATProto records and SQL rows.

4.11 Summary

  • A protocol in panproto is a pair: (schema theory, instance theory). panproto currently supports 54 protocols organized into five theory groups.
  • Schema theories are assembled as colimits of simpler theories capturing orthogonal concerns (graph structure, constraints, multiplicity, interfaces, metadata, hyperedges).
  • Instance theories fall into three families: set-valued functors (Group B, 8 protocols), W-types (Groups A, D, E; 38 protocols total), and flat instances (Group C, 9 protocols). ThFlat is the depth-1 degenerate case of ThWType.
  • The restriction operator—how migration acts on data—is determined by the instance theory. Functors use precomposition (\(\Delta_F\)). W-types use a multi-step tree surgery pipeline. Flat instances use field projection, which is the depth-1 specialization of tree surgery.
  • Group E protocols extend the W-type machinery with a metadata theory (\(\text{colim}(\text{ThWType}, \text{ThMeta})\)), enabling migration of both tree structure and out-of-band annotation in a single pass.
  • Everything else is derived. Migration types, existence checks, and breaking-change classifiers are mechanically produced from the two theories.
  • The protocol is a parameter, not hardcoded logic. Adding a new protocol means declaring two theories, not writing an engine.

A few things to notice:

- **Schema theories are colimits.** Most real-world protocols combine several concerns---graphs with constraints, multigraph structure with interfaces. The schema theory is assembled by taking a colimit (a structured union) of simpler component theories. This is exactly the colimit construction from Chapter 3.

- **Instance theories cluster into three families.** There are functor-based instances (ThFunctor), tree-based instances (ThWType), and flat instances (ThFlat). This is the deepest structural split in the table, and it determines how migration works at the data level. ThFlat is best understood as a degenerate case of ThWType restricted to depth-1 trees.

- **The restriction operator is determined by the instance theory.** Functor instances always migrate via precomposition ($\Delta_F$). W-type instances always migrate via the signature-restriction pipeline. Flat instances use the simpler field-projection operator, which is the depth-1 specialization of tree surgery. You don't choose a migration strategy; it follows from the theory.


## Three Families of Instances

The instance theory is where the rubber meets the road. It determines what "data" means for a given schema, and therefore what "migrating data" means. Let's look at the three families in detail.

### Set-Valued Functors {#sec-functor-instances}

The functor model comes from the categorical databases tradition, developed most thoroughly by @spivak2012 and formalized in the CQL system by @schultz2017.

The idea is direct. A schema is a category $\mathcal{C}$---objects are tables (or types), morphisms are foreign keys (or relationships). An *instance* of that schema is a functor $I : \mathcal{C} \to \mathbf{Set}$. In concrete terms:

- Each table $T$ in the schema maps to a set $I(T)$---the rows of that table.
- Each foreign key $f : T_1 \to T_2$ maps to a function $I(f) : I(T_1) \to I(T_2)$---the lookup that sends each row of $T_1$ to its corresponding row in $T_2$.

This is a faithful model of relational data. The functoriality condition ($I(g \circ f) = I(g) \circ I(f)$) ensures that following two foreign keys in sequence gives the same result whether you compose them first or chase them one at a time. Referential integrity is built into the mathematics.

**Migration under the functor model** is precomposition. Given a schema morphism $F : \mathcal{C}_1 \to \mathcal{C}_2$ (a functor mapping the old schema into the new one), and an instance $I : \mathcal{C}_2 \to \mathbf{Set}$ living over the new schema, you get an instance $\Delta_F(I) = I \circ F : \mathcal{C}_1 \to \mathbf{Set}$ over the old schema. In operational terms: for each old table $T$, you look up which new table $F(T)$ it maps to, and pull back that table's rows. Foreign keys compose automatically.

This is the pullback migration functor $\Delta_F$ from [@spivak2012]. It has left and right adjoints ($\Sigma_F$ and $\Pi_F$) that handle the "push data forward" direction, but for now, $\Delta_F$ is the one that matters for understanding the restriction column of the protocol table.

### W-Types: Tree-Shaped Data {#sec-wtype-instances}

The functor model works beautifully for relational data, but most modern protocols---ATProto, GraphQL, JSON Schema, TypeScript, and the majority of the 54 protocols panproto supports---traffic in *tree-shaped* data. A Bluesky post is not a row in a table; it is a JSON document with nested objects, arrays, and references. The right mathematical model for this kind of data is a *W-type* (wellfounded tree), studied by @gambino2004 in the context of dependent type theory and related to the *containers* of @abbott2005.

A W-type instance over a schema is a finitely-branching tree where:

- **Each node is anchored to a schema vertex.** The root might be anchored to a `Post` vertex, a child to a `Text` vertex, a sibling to a `CreatedAt` vertex.
- **Each branch (parent-to-child edge) is labeled by a schema edge.** The edge from `Post` to `Text` might be labeled `text`, the edge from `Post` to `CreatedAt` might be labeled `createdAt`.
- **The tree is wellfounded.** No infinite descending chains. Every path from root to leaf is finite.

This is exactly the shape of a JSON document, an ATProto record, or a GraphQL response. The schema tells you which node types exist and which edges connect them. The instance is a particular tree that conforms to those constraints.

**Migration under the W-type model** is more involved than precomposition, because you can't just "pull back" a tree through a functor. Instead, the engine runs a multi-step pipeline:

1. **Signature restriction**: remove nodes anchored to vertices not in the image of the migration map.
2. **Reachability BFS**: starting from the root, find all nodes still reachable after the pruning.
3. **Ancestor contraction**: when intermediate nodes are removed, reconnect their children to the nearest surviving ancestor.
4. **Edge resolution**: relabel branches according to the migration's edge map, resolving ambiguities where necessary.
5. **Fan reconstruction**: rebuild the branching structure of the tree to maintain wellfoundedness.

We will walk through this pipeline concretely in [Chapter 7](07-lifting-data.qmd). For now, the key point is that this pipeline is *derived* from the theory pair. The engine knows it needs to do tree surgery (rather than precomposition) because the instance theory is ThWType rather than ThFunctor.

### Flat Instances: Depth-1 Trees {#sec-flat-instances}

ThFlat is not an independent theory so much as a disciplined restriction of ThWType. When the schema is a simple graph (no parallel edges, no recursive types) and instances are constrained to trees of depth at most 1, the full W-type machinery reduces to a much simpler operation: **field projection**.

A flat instance over a simple-graph schema is a collection of key-value records where each key is a schema edge and each value is a leaf datum. Because the tree never nests beyond depth 1, contraction never fires, reachability is trivial, and edge resolution degenerates to a straightforward lookup. The restriction operator collapses to: keep the fields whose schema edges appear in the image of the migration map; discard the rest.

This simplification is not merely computational. ThFlat captures a genuine semantic boundary: protocols like Protobuf, Avro, and Thrift are deliberately flat. They are designed for high-throughput serialization where recursive nesting would impose unacceptable overhead. Treating ThFlat as a proper theory (rather than a naming convention) ensures that panproto can enforce the depth-1 constraint at the theory level and reject schemas that would violate it.

The relationship is:

$$\text{ThFlat} \hookrightarrow \text{ThWType}$$

where the inclusion is a theory morphism that restricts the tree-height axioms. A ThFlat instance is a ThWType instance; the converse does not hold.


## Comparing the Three Models

It's worth pausing to see these three instance theories side by side, because the choice among them is the central architectural decision that distinguishes protocol families.

| Aspect | Functor (ThFunctor) | W-Type (ThWType) | Flat (ThFlat) |
|--------|---------------------|-------------------|----------------|
| Data shape | Flat tables of rows | Nested trees of nodes | Depth-1 key-value records |
| Schema role | Category (objects + morphisms) | Graph (vertices + edges) | Simple graph (no parallel edges) |
| Instance role | Functor to **Set** | Wellfounded tree | Depth-1 tree |
| Migration | Precomposition ($\Delta_F$) | Tree surgery pipeline | Field projection |
| Canonical users | SQL, CQL, Parquet | ATProto, GraphQL, TypeScript | Protobuf, Avro, Thrift |
| Compositionality | Functorial (automatic) | Requires reachability analysis | Trivial (projection) |
| Adjoints available | Yes ($\Sigma_F$, $\Pi_F$) | Not in general | Not in general |
| Relationship | Independent | Generalizes ThFlat | Special case of ThWType |

: Comparing the three instance theory families. {#tbl-instance-comparison}

None of these models is "better" in the abstract. The functor model has superior compositional properties---migration functors compose, and the adjoint triple gives you round-tripping for free. But the functor model demands that your data is tabular, which is a poor fit for document-oriented protocols. The W-type model handles trees natively, at the cost of more complex migration logic and fewer automatic guarantees. ThFlat threads a middle path: it gives up the generality of arbitrary nesting in exchange for a migration operator so simple it is nearly free to compute.

panproto's contribution is that you don't have to choose one model for all protocols. Each protocol declares its own instance theory, and the engine adapts accordingly. SQL gets precomposition. ATProto gets tree surgery. Protobuf gets field projection. The outer framework---schema morphisms, existence checks, breaking-change detection---works uniformly across all three.

## Why Colimits for Schema Theories?

You may have noticed that most schema theories in @tbl-protocol-table are colimits of simpler theories. This is not an accident. Real-world protocols are assembled from orthogonal concerns:

- **Graph structure**: vertices and directed edges (ThGraph).
- **Constraint structure**: invariants on vertex/edge data (ThConstraint).
- **Multiplicity structure**: arrays, optional fields, repeated edges (ThMulti).
- **Interface structure**: abstract types implemented by concrete ones (ThInterface).
- **Hyperedge structure**: edges connecting more than two vertices simultaneously (ThHyperEdge).
- **Metadata structure**: out-of-band annotation layers attached to the schema or instance (ThMeta).

These concerns are largely independent of one another. A protocol might need graphs and constraints but not interfaces (ATProto). Another might need graphs, constraints, multiplicity, *and* interfaces (GraphQL). Still another might need the full multigraph combination *plus* a metadata layer, as in the tree-plus-metadata protocols of Group E. The colimit construction lets you assemble exactly the combination you need, with the categorical machinery ensuring that the pieces fit together coherently.

For example, ATProto's schema theory is:

$$\text{ThATProtoSchema} = \text{colim}(\text{ThGraph}, \text{ThConstraint}, \text{ThMulti})$$

This says: an ATProto schema is a structure that is simultaneously a graph, a constrained structure, and a multiplicity-aware structure, with the overlapping parts (the shared sorts and operations) identified. The colimit handles the identification automatically.

SQL's schema theory replaces ThGraph with ThHypergraph (because SQL tables have multi-column keys and compound foreign keys) and drops ThMulti (because SQL doesn't have native array columns in the same sense):

$$\text{ThSQLSchema} = \text{colim}(\text{ThHypergraph}, \text{ThConstraint})$$

GraphQL adds ThInterface on top of ATProto's combination:

$$\text{ThGraphQLSchema} = \text{colim}(\text{ThGraph}, \text{ThConstraint}, \text{ThMulti}, \text{ThInterface})$$

Group E protocols---XML/XSD, HTML, Neo4j, and their kin---add ThMeta to the constrained multigraph base. This captures the fact that these formats carry an additional annotation layer (XML attributes, HTML metadata, Neo4j property maps) that sits alongside the tree structure rather than inside it:

$$\text{ThXMLSchema} = \text{colim}(\text{ThGraph}, \text{ThConstraint}, \text{ThMulti}, \text{ThMeta})$$

The instance theory for Group E protocols is correspondingly $\text{colim}(\text{ThWType}, \text{ThMeta})$: a wellfounded tree together with a metadata functor that annotates nodes with out-of-band data. Migration for these protocols runs the standard W-type pipeline on the tree component and a separate restriction pass on the metadata component, with the two passes coupled at the nodes they share.

The pattern is modular and extensible. If tomorrow a new protocol emerges that needs graph structure, constraints, and some novel "versioning" concern, you write ThVersioning as a new theory and take the colimit with the existing pieces. The engine will derive migration, existence checks, and breaking-change detection for the new combination without any protocol-specific code.


## The Restrict Column

The rightmost column of @tbl-protocol-table, labeled "Restrict," names the operation that applies a migration to data. This column is *determined by* the instance theory---you never choose it independently.

### Precomposition ($\Delta_F$)

When the instance theory is ThFunctor, the restriction operator is precomposition. Given:

- A schema morphism $F : \mathcal{S}_1 \to \mathcal{S}_2$
- An instance $I : \mathcal{S}_2 \to \mathbf{Set}$

the restricted instance is $\Delta_F(I) = I \circ F$. This is the pullback functor from [@spivak2012].

In practice, precomposition is almost trivially simple to implement. For each vertex $v$ in $\mathcal{S}_1$, look up $F(v)$ in $\mathcal{S}_2$ and take the data from that vertex. For each edge $e$ in $\mathcal{S}_1$, look up $F(e)$ and use that function to chase foreign keys. The functoriality of $F$ guarantees that everything is consistent.

### Signature Restriction + BFS + Contraction

When the instance theory is ThWType, the restriction operator is a multi-step pipeline. We gave a brief overview in @sec-wtype-instances; [Chapter 7](07-lifting-data.qmd) provides the full algorithmic detail. The key steps are:

1. **Signature restriction**: discard tree nodes whose anchor vertex is not in the image of the migration's vertex map.
2. **Reachability BFS**: from the root, walk the tree to find which surviving nodes are still connected.
3. **Ancestor contraction**: if a removed node had children that survived, reattach those children to the nearest surviving ancestor.
4. **Edge resolution**: relabel parent-child edges according to the migration's edge map. If the schema allows parallel edges (multigraphs), this step may require resolver data to disambiguate.
5. **Fan reconstruction**: rebuild the node's children list to reflect the new edge labels and ordering.

This pipeline is more expensive than precomposition, but it handles the structural complexity of tree-shaped data. Every step is necessary: skip reachability and you'll keep orphaned subtrees; skip contraction and you'll lose data that should survive; skip resolution and you'll have ambiguous edge labels.

### Field Projection

When the instance theory is ThFlat, the restriction operator reduces to field projection: keep the fields that the migration maps, drop the rest. As noted in @sec-flat-instances, this is the depth-1 degenerate case of the W-type pipeline---the BFS is trivial (root plus leaves only), contraction never fires, and fan reconstruction is a simple list filter. Group C protocols (Protobuf, Avro, Thrift, Cap'n Proto, FlatBuffers, ASN.1, Bond, Redis, HCL) all use this operator.


## From Table to Engine

The protocol table is not just a reference chart. It is a *specification* that the engine reads at startup. When you instantiate panproto for a particular protocol, the engine:

1. **Parses the schema theory** to determine what sorts (Vertex, Edge, Constraint, ...) exist and what operations connect them.
2. **Parses the instance theory** to determine the data representation (functor, W-type, or flat).
3. **Derives the migration type** from the schema theory's sorts and operations. A migration must carry a map for each sort in the schema theory.
4. **Derives the restriction operator** from the instance theory. Functor → precomposition. W-type → tree surgery. Flat → field projection.
5. **Derives existence checks** from the schema theory's axioms (Chapter 6).
6. **Derives the breaking-change classifier** from the interaction of schema theory and instance theory (Chapter 9).

Steps 3--6 are fully automatic. The protocol author provides the two theories; the engine provides the rest.

This is the sense in which panproto is *protocol-parametric*. The protocol is a parameter---literally, a pair of theory values passed to the engine. Swap the parameter, and you get a different protocol's migration system, with correct semantics, for free.

This framing surfaces some natural questions:

- **What separates Group A from Group D?** Both use ThWType instances over constrained multigraph schemas. The distinguishing feature is ThInterface: Group D protocols treat abstract types (interfaces, traits, type classes) as first-class schema citizens, whereas Group A protocols do not.
- **What makes Group E distinct from Group A?** Group E adds ThMeta to the schema colimit and correspondingly lifts the instance theory to $\text{colim}(\text{ThWType}, \text{ThMeta})$. The metadata layer is not just an annotation convention; it participates in migration.
- **What lives between Group B and Group A?** A protocol with hypergraph schemas but W-type instances---perhaps a document database with compound keys and nested values. panproto can accommodate such a protocol by combining ThHypergraph, ThConstraint, and ThWType in the obvious way.
- **What lies beyond Group D?** Add a ThVersioning theory to the schema colimit, and you get a protocol with built-in schema versioning at the theory level. The engine will derive migration and breaking-change detection for it automatically.
- **Can you mix instance theories?** In principle, a protocol could use functor instances for some schema sorts and W-type instances for others. panproto doesn't support this yet, but the theory permits it.

The design space is open. panproto gives you the coordinates.


## Historical Context

The idea that data migration can be parameterized by algebraic structure has roots in several traditions.

@spivak2012 showed that functorial data migration---using functors between categories as schema morphisms---gives a clean, compositional account of SQL-style migration. The CQL project [@schultz2017] turned this into a practical tool for relational databases, demonstrating that the category-theoretic framework could handle real workloads.

The W-type perspective comes from dependent type theory. @gambino2004 showed that wellfounded trees (W-types) are the natural inductive types in categories with dependent products. @abbott2005 connected this to the theory of containers, which views tree-shaped data as polynomial functors. panproto uses this connection: a W-type instance is a section of a polynomial functor determined by the schema.

The insight that these traditions can be unified under a common framework---that functor instances, W-type instances, and flat instances all arise as models of different instance theories, with migration derived from the theory structure, and that 54 real-world protocols fall cleanly into five theory groups under this classification---is, as far as we know, original to panproto.

## Summary

- A **protocol** in panproto is a pair: (schema theory, instance theory). panproto currently supports **54 protocols** organized into **five theory groups**.
- **Schema theories** are assembled as colimits of simpler theories capturing orthogonal concerns (graph structure, constraints, multiplicity, interfaces, metadata, hyperedges).
- **Instance theories** fall into three families: **set-valued functors** (Group B, 8 protocols), **W-types** (Groups A, D, E; 38 protocols total), and **flat instances** (Group C, 9 protocols). ThFlat is the depth-1 degenerate case of ThWType.
- The **restriction operator**---how migration acts on data---is determined by the instance theory. Functors use precomposition ($\Delta_F$). W-types use a multi-step tree surgery pipeline. Flat instances use field projection, which is the depth-1 specialization of tree surgery.
- **Group E** protocols extend the W-type machinery with a metadata theory ($\text{colim}(\text{ThWType}, \text{ThMeta})$), enabling migration of both tree structure and out-of-band annotation in a single pass.
- **Everything else is derived.** Migration types, existence checks, and breaking-change classifiers are mechanically produced from the two theories.
- The protocol is a *parameter*, not hardcoded logic. Adding a new protocol means declaring two theories, not writing an engine.

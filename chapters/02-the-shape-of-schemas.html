3 The Shape of Schemas

In the previous chapter, we looked at a blog post schema written in five different formats and noticed that they all describe vertices connected by edges. That was a useful intuition, but it was imprecise. The word “graph” can mean many things, and the differences matter.

This chapter makes the graph-theoretic structure of schemas precise. We’ll see that schema languages don’t all use the same kind of graph. Some need multigraphs. Some need hypergraphs. Some need constraints layered on top. Understanding which graph structure a schema language requires is the first step toward understanding what panproto actually parameterizes over.

3.1 Directed Graphs: The Simplest Case

A directed graph consists of:

  • A set of vertices (also called nodes)
  • A set of edges (also called arrows), where each edge has a source vertex and a target vertex

That’s it. Two operations—src and tgt—take an edge and return its source and target vertices, respectively. This is the simplest graph structure, and it’s the one most developers picture when they hear “graph.”

Consider a minimal JSON Schema:

{
  "type": "object",
  "properties": {
    "name": { "type": "string" }
  }
}

This describes a directed graph with two vertices (object and string) and one edge (name, pointing from object to string). The src of the edge is the object type; the tgt is the string type.

Many simple schemas fit this pattern. A GraphQL type with scalar fields is a directed graph. A Protobuf message with only primitive-typed fields is a directed graph. The directed graph captures the most basic structural relationship: “this type has a field of that type.”

Graph transformation theory provides a rigorous vocabulary for working with these structures. The foundational framework, developed in (Ehrig et al. 2006), treats graphs as algebraic objects and defines operations (morphisms, pushouts, pullbacks) that let you compose, decompose, and transform graphs in principled ways. This vocabulary turns out to be exactly what we need for schema operations—migration, merging, diffing—across protocols.

The origins of this approach go back further, to the graph grammars introduced by (Ehrig et al. 1973). That early work established that graph-based structures could be manipulated by formal rewriting rules, much like string grammars in formal language theory. The key insight was that structure-preserving transformations on graphs could be characterized algebraically—an insight that panproto inherits directly.

3.2 Multigraphs: When One Edge Isn’t Enough

A directed graph has a limitation: between any pair of vertices, there’s at most one edge (in each direction). For many schemas, this is fine. But some schemas need multiple distinct edges between the same pair of vertices.

Consider an ATProto Lexicon record with two string fields:

{
  "type": "object",
  "properties": {
    "displayName": { "type": "string", "maxLength": 640 },
    "description": { "type": "string", "maxLength": 2560 }
  }
}

In graph terms, the object vertex has two edges pointing to the string vertex: one labeled displayName and one labeled description. A plain directed graph can’t represent this—it would collapse the two edges into one. We need a multigraph: a directed graph that allows parallel edges between the same pair of vertices.

A directed multigraph consists of:

  • A set of vertices
  • A set of edges, where each edge has a source vertex, a target vertex, and an identity distinct from other edges (even those with the same source and target)

The distinction between a graph and a multigraph might seem pedantic, but it has real consequences for tooling. If your internal representation is a simple adjacency matrix (which can’t represent parallel edges), you’ll silently collapse distinct fields that happen to share a type. panproto uses multigraphs as its baseline representation precisely to avoid this class of bugs.

Here’s what our ATProto profile schema looks like as a multigraph:

Both edges connect object to string, but they are distinct: they carry different labels and different constraints. A tool that represents schemas as simple graphs would need some workaround—perhaps making each field its own vertex, or using an adjacency list with labels. A multigraph handles this naturally.

3.3 Hypergraphs: When Edges Connect More Than Two Things

Directed graphs (and multigraphs) have edges that connect exactly two vertices: a source and a target. But some schema constructs are inherently multi-ary— they involve relationships among three, four, or more entities simultaneously.

The canonical example is a SQL foreign key.

CREATE TABLE posts (
    id          BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    text        VARCHAR(3000) NOT NULL,
    author_id   BIGINT NOT NULL,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE authors (
    id          BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name        VARCHAR(255) NOT NULL
);

ALTER TABLE posts
    ADD CONSTRAINT fk_posts_author
    FOREIGN KEY (author_id) REFERENCES authors (id);

The foreign key fk_posts_author doesn’t connect two things—it connects four:

  1. The source table (posts)
  2. The source column (author_id)
  3. The referenced table (authors)
  4. The referenced column (id)

You can’t faithfully represent this as a binary edge without losing information. If you draw an edge from posts to authors, you’ve lost the column-level detail. If you draw an edge from author_id to id, you’ve lost the table-level context. The foreign key is a single constraint that simultaneously involves all four entities.

This is exactly what hypergraphs are for.

A hypergraph consists of:

  • A set of vertices
  • A set of hyperedges, where each hyperedge connects an ordered sequence of vertices (not just two)

A hyperedge can connect any number of vertices—two, three, four, or more. Binary edges are a special case. This makes hypergraphs strictly more expressive than directed graphs or multigraphs.

The theory of hyperedge replacement grammars, developed by (Drewes et al. 1997), provides a formal framework for manipulating hypergraphs with the same rigor that graph grammars provide for ordinary graphs. In this framework, hyperedges can be “replaced” by more complex subhypergraphs according to production rules—a mechanism that maps naturally to schema expansion operations like denormalization or view materialization.

Here’s our SQL schema with its foreign key as a hypergraph:

The dashed node fk_posts_author is the hyperedge. It has four tentacles, each connecting to a different vertex in the schema. This can’t be reduced to binary edges without either losing information or introducing artificial intermediate vertices.

3.3.1 Why Hyperedges Matter for Tooling

If your schema representation only supports binary edges, you have two options for foreign keys:

  1. Lose information. Draw an edge from posts to authors and hope the column-level detail is recoverable from context. This works until you have composite foreign keys (multiple columns), at which point it breaks.

  2. Introduce intermediate vertices. Create a “foreign key” vertex with binary edges to each of the four participants. This works, but it means your graph has vertices that don’t correspond to anything in the original schema. Every operation (traversal, transformation, serialization) must now handle these phantom vertices correctly.

Hypergraphs avoid both problems. The foreign key is a first-class hyperedge, connecting exactly the vertices it involves. No information lost, no phantoms introduced.

This isn’t just about foreign keys. Composite unique constraints in SQL, oneof declarations in Protobuf, union types in GraphQL—these are all multi-ary relationships that benefit from hypergraph representation.

3.4 Constraints: Decorating the Graph

So far we’ve talked about the shape of the graph: what kinds of vertices and edges exist, how they connect. But schemas carry more information than just connectivity. They have constraints: maxLength, NOT NULL, DEFAULT, pattern, minimum, maxItems, and dozens more.

Constraints don’t change the graph structure—they decorate it. A maxLength constraint on a string property doesn’t add a new vertex or edge; it attaches metadata to an existing edge (the property edge connecting an object to a string type).

We can model this cleanly by distinguishing between:

  • The underlying graph (vertices and edges/hyperedges), which captures the structural skeleton of the schema
  • Constraint annotations on vertices and edges, which carry the domain-specific validation rules

This separation is useful because graph-structural operations (like merging two schemas or computing a diff) can often ignore constraints, while constraint-specific operations (like checking whether one schema is a subtype of another) can work with the graph structure as given.

Different schema languages support different constraints:

Constraint JSON Schema Protobuf SQL DDL GraphQL ATProto Lexicon
maxLength yes no VARCHAR(n) no yes
pattern (regex) yes no CHECK no no
minimum / maximum yes no CHECK no yes
NOT NULL required[] (default set) NOT NULL ! required[]
DEFAULT default default value DEFAULT no default
UNIQUE no no UNIQUE no no
PRIMARY KEY no no PRIMARY KEY no no
format yes no data type scalar yes
maxItems yes no no no yes (maxLength)

The constraint vocabulary varies wildly across schema languages, but the mechanism—annotations on a graph—is uniform. panproto represents constraints as typed metadata on edges and vertices, allowing protocol-specific constraints to coexist with structural information.

3.5 A Taxonomy of Schema Structures

We can now classify schema languages by the graph structure they require:

Schema Language Minimum Graph Structure Theory Group Why
JSON Schema Multigraph + constraints A Parallel properties to same type; rich constraint set
Protobuf Simple graph (Multigraph) C Parallel fields to same type; few built-in constraints
SQL DDL Hypergraph + constraints B Foreign keys are multi-ary; CHECK, NOT NULL, etc.
GraphQL SDL Multigraph + constraints D Parallel fields; nullability as constraint
ATProto Lexicon Multigraph + constraints A Parallel properties; maxLength, maxGraphemes, etc.
Avro Simple graph (Multigraph) C Parallel fields; Group C simple graph
TypeScript Multigraph + constraints D Interfaces, generics; typed graph
Cassandra CQL Hypergraph + constraints B Compound partition keys; multi-ary key structure
XML/XSD Multigraph + constraints + metadata E Element/attribute duality; meta-level annotations
OpenAPI Multigraph + constraints A Path parameters, schemas; rich constraint vocabulary

Notice that no schema language we’ve examined requires only a simple directed graph. The moment you allow two fields of the same type (which is… always), you need at least a multigraph. SQL’s and Cassandra’s compound keys push the requirement to hypergraphs. XML/XSD’s element-attribute duality introduces a meta-level that requires annotations beyond ordinary constraints.

These ten protocols are a representative sample. panproto organizes its full set of 54 protocols into 5 theory groups based on the graph structure they require—from Group C (simple multigraphs, e.g. Protobuf, Avro, Thrift) through Group E (multigraphs with constraint and metadata layers, e.g. XML/XSD, HTML). The taxonomy extends uniformly to all 54 protocols; see ?sec-protocol-catalog for the complete catalog with theory group assignments.

This taxonomy explains a pattern you’ve probably noticed in the ecosystem: schema tools tend to be protocol-specific. Buf is excellent for Protobuf. Atlas is excellent for SQL. GraphQL Code Generator is excellent for GraphQL. But none of them generalize across protocols.

3.6 Why Single-Protocol Tools Can’t Generalize

The reason is structural, not incidental. Each tool hardcodes a specific graph type:

  • Buf assumes a multigraph structure (messages with fields). It understands Protobuf’s field numbering, package namespaces, and options. But it has no representation for hyperedges, so it can’t natively express SQL foreign keys.

  • Atlas assumes a hypergraph structure (tables with columns and multi-ary constraints). It understands SQL’s type system, migration semantics, and constraint checking. But its vertex types are SQL-specific (tables, views, indexes), not Protobuf-specific (messages, enums, services).

  • GraphQL Code Generator assumes a multigraph with GraphQL’s specific vertex types (Object, Interface, Union, Enum, Scalar). It can’t represent Protobuf’s field numbering or SQL’s CHECK constraints.

Each tool is a perfect fit for its target protocol and a poor fit for everything else. This isn’t a failure of engineering—it’s a consequence of fixing the graph structure at design time.

panproto takes a different approach: it parameterizes over the graph structure. Instead of hardcoding “the schema is a multigraph with these specific vertex and edge types,” panproto lets you specify what kinds of vertices, edges, and constraints your schema language needs. The same core algorithms—diffing, merging, migration planning, code generation—work across any graph structure that fits the framework.

This parameterization is not free. It requires a mathematical framework expressive enough to describe multigraphs, hypergraphs, and constrained variants in a uniform way. That framework exists: it comes from the categorical treatment of schemas developed by (Spivak 2012), who showed that database schemas can be viewed as categories (with tables as objects and foreign keys as morphisms) and that schema mappings are functors between those categories. panproto generalizes this functorial perspective beyond relational databases to arbitrary schema languages.

The key insight from Spivak’s work is that once you view schemas as categorical structures, you get a rich toolkit for free: limits and colimits give you schema merging and intersection, functors give you schema mappings and translations, and natural transformations give you data migration along those mappings. These aren’t metaphors—they’re precise mathematical constructions with well-defined properties.

NoteThe Parameterization Principle

Single-protocol tools fix the graph structure and optimize for a specific schema language. panproto parameterizes the graph structure and provides uniform algorithms that work across schema languages. This is the fundamental architectural difference—and it’s what makes cross-protocol schema operations possible.

3.7 From Graphs to Theories

We’ve established that schema languages have graph-theoretic structure, and that different languages need different kinds of graphs. But we’ve been describing these graph structures informally—listing their components (vertices, edges, operations) in prose.

There’s a precise way to describe a graph structure: you specify its sorts (the kinds of things that exist), its operations (the functions between those sorts), and its equations (the laws those operations obey). A directed graph, for instance, has two sorts (Vertex and Edge), two operations (src and tgt, both from Edge to Vertex), and no equations. A multigraph has the same description—the “multi” comes from not requiring edges to be uniquely determined by their source and target.

This sort-operation-equation description is called a theory, and the things that satisfy a theory are called models. A specific ATProto schema is a model of the multigraph theory. A specific SQL schema is a model of the hypergraph-with-constraints theory.

If the language of sorts, operations, and equations sounds suspiciously like an abstract algebra course, that’s because it is. The next chapter introduces the specific algebraic framework panproto uses—Generalized Algebraic Theories—and shows how the Rust implementation represents theories and models as concrete data structures. The math will earn its keep: it’s what makes panproto’s parameterization possible.

3.8 Summary

This chapter established three ideas:

  1. Schema languages have graph-theoretic structure. Types are vertices; fields, columns, and properties are edges. This is not a metaphor—it’s a precise structural description.

  2. Different schema languages need different kinds of graphs. Multigraphs for parallel fields (most languages). Hypergraphs for multi-ary constraints (SQL). Constraints as annotations on any of these.

  3. Single-protocol tools hardcode a specific graph structure. This is why Buf can’t do SQL migrations and Atlas can’t lint Protobuf. panproto parameterizes over the graph structure instead.

The question now is: how do you precisely describe a graph structure so that software can work with it generically? That’s the subject of the next chapter.

## Hypergraphs: When Edges Connect More Than Two Things

Directed graphs (and multigraphs) have edges that connect exactly two vertices:
a source and a target. But some schema constructs are inherently **multi-ary**—
they involve relationships among three, four, or more entities simultaneously.

The canonical example is a SQL **foreign key**.

```sql
CREATE TABLE posts (
    id          BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    text        VARCHAR(3000) NOT NULL,
    author_id   BIGINT NOT NULL,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE authors (
    id          BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name        VARCHAR(255) NOT NULL
);

ALTER TABLE posts
    ADD CONSTRAINT fk_posts_author
    FOREIGN KEY (author_id) REFERENCES authors (id);
```

The foreign key `fk_posts_author` doesn't connect two things—it connects
**four**:

1. The **source table** (`posts`)
2. The **source column** (`author_id`)
3. The **referenced table** (`authors`)
4. The **referenced column** (`id`)

You can't faithfully represent this as a binary edge without losing information.
If you draw an edge from `posts` to `authors`, you've lost the column-level
detail. If you draw an edge from `author_id` to `id`, you've lost the
table-level context. The foreign key is a single constraint that simultaneously
involves all four entities.

This is exactly what **hypergraphs** are for.

A **hypergraph** consists of:

- A set of **vertices**
- A set of **hyperedges**, where each hyperedge connects an ordered sequence of
  vertices (not just two)

A hyperedge can connect any number of vertices—two, three, four, or more. Binary
edges are a special case. This makes hypergraphs strictly more expressive than
directed graphs or multigraphs.

The theory of hyperedge replacement grammars, developed by [@drewes1997],
provides a formal framework for manipulating hypergraphs with the same rigor
that graph grammars provide for ordinary graphs. In this framework, hyperedges
can be "replaced" by more complex subhypergraphs according to production
rules—a mechanism that maps naturally to schema expansion operations like
denormalization or view materialization.

Here's our SQL schema with its foreign key as a hypergraph:


The dashed node `fk_posts_author` is the hyperedge. It has four tentacles, each

edges without either losing information or introducing artificial intermediate
vertices.

### Why Hyperedges Matter for Tooling

If your schema representation only supports binary edges, you have two options
for foreign keys:

1. **Lose information.** Draw an edge from `posts` to `authors` and hope the
   column-level detail is recoverable from context. This works until you have
   composite foreign keys (multiple columns), at which point it breaks.

2. **Introduce intermediate vertices.** Create a "foreign key" vertex with
   binary edges to each of the four participants. This works, but it means your
   graph has vertices that don't correspond to anything in the original schema.
   Every operation (traversal, transformation, serialization) must now handle
   these phantom vertices correctly.

Hypergraphs avoid both problems. The foreign key is a first-class hyperedge,
connecting exactly the vertices it involves. No information lost, no phantoms
introduced.

This isn't just about foreign keys. Composite unique constraints in SQL,
`oneof` declarations in Protobuf, union types in GraphQL—these are all
multi-ary relationships that benefit from hypergraph representation.


## Constraints: Decorating the Graph

So far we've talked about the *shape* of the graph: what kinds of vertices and
edges exist, how they connect. But schemas carry more information than just
connectivity. They have **constraints**: `maxLength`, `NOT NULL`, `DEFAULT`,
`pattern`, `minimum`, `maxItems`, and dozens more.

Constraints don't change the graph structure—they decorate it. A `maxLength`
constraint on a string property doesn't add a new vertex or edge; it attaches
metadata to an existing edge (the property edge connecting an object to a
string type).

We can model this cleanly by distinguishing between:

- **The underlying graph** (vertices and edges/hyperedges), which captures the
  structural skeleton of the schema
- **Constraint annotations** on vertices and edges, which carry the
  domain-specific validation rules

This separation is useful because graph-structural operations (like merging two
schemas or computing a diff) can often ignore constraints, while
constraint-specific operations (like checking whether one schema is a subtype of
another) can work with the graph structure as given.

Different schema languages support different constraints:

| Constraint        | JSON Schema | Protobuf      | SQL DDL       | GraphQL   | ATProto Lexicon |
|-------------------|-------------|---------------|---------------|-----------|-----------------|
| maxLength         | yes         | no            | VARCHAR(n)    | no        | yes             |
| pattern (regex)   | yes         | no            | CHECK         | no        | no              |
| minimum / maximum | yes         | no            | CHECK         | no        | yes             |
| NOT NULL          | required[]  | (default set) | NOT NULL      | `!`       | required[]      |
| DEFAULT           | default     | default value | DEFAULT       | no        | default         |
| UNIQUE            | no          | no            | UNIQUE        | no        | no              |
| PRIMARY KEY       | no          | no            | PRIMARY KEY   | no        | no              |
| format            | yes         | no            | data type     | scalar    | yes             |
| maxItems          | yes         | no            | no            | no        | yes (maxLength) |

The constraint vocabulary varies wildly across schema languages, but the
*mechanism*—annotations on a graph—is uniform. panproto represents constraints
as typed metadata on edges and vertices, allowing protocol-specific constraints
to coexist with structural information.

|-------------------|--------------------------------------|--------------|--------------------------------------------------------------|
| JSON Schema       | Multigraph + constraints             | A            | Parallel properties to same type; rich constraint set        |
| Protobuf          | Simple graph (Multigraph)            | C            | Parallel fields to same type; few built-in constraints       |
| SQL DDL           | Hypergraph + constraints             | B            | Foreign keys are multi-ary; CHECK, NOT NULL, etc.            |
| GraphQL SDL       | Multigraph + constraints             | D            | Parallel fields; nullability as constraint                   |
| ATProto Lexicon   | Multigraph + constraints             | A            | Parallel properties; maxLength, maxGraphemes, etc.           |
| Avro              | Simple graph (Multigraph)            | C            | Parallel fields; Group C simple graph                        |
| TypeScript        | Multigraph + constraints             | D            | Interfaces, generics; typed graph                            |
| Cassandra CQL     | Hypergraph + constraints             | B            | Compound partition keys; multi-ary key structure             |
| XML/XSD           | Multigraph + constraints + metadata  | E            | Element/attribute duality; meta-level annotations            |
| OpenAPI           | Multigraph + constraints             | A            | Path parameters, schemas; rich constraint vocabulary         |

Notice that no schema language we've examined requires only a simple directed
graph. The moment you allow two fields of the same type (which is... always),
you need at least a multigraph. SQL's and Cassandra's compound keys push the
requirement to hypergraphs. XML/XSD's element-attribute duality introduces a
meta-level that requires annotations beyond ordinary constraints.

These ten protocols are a representative sample. panproto organizes its full
set of 54 protocols into 5 theory groups based on the graph structure they
require—from Group C (simple multigraphs, e.g. Protobuf, Avro, Thrift) through
Group E (multigraphs with constraint and metadata layers, e.g. XML/XSD, HTML).
The taxonomy extends uniformly to all 54 protocols; see
@sec-protocol-catalog for the complete catalog with theory group assignments.

This taxonomy explains a pattern you've probably noticed in the ecosystem:
schema tools tend to be protocol-specific. **Buf** is excellent for Protobuf.
**Atlas** is excellent for SQL. **GraphQL Code Generator** is excellent for
GraphQL. But none of them generalize across protocols.


## Why Single-Protocol Tools Can't Generalize

The reason is structural, not incidental. Each tool hardcodes a specific graph
type:

- **Buf** assumes a multigraph structure (messages with fields). It understands
  Protobuf's field numbering, package namespaces, and options. But it has no
  representation for hyperedges, so it can't natively express SQL foreign keys.

- **Atlas** assumes a hypergraph structure (tables with columns and multi-ary
  constraints). It understands SQL's type system, migration semantics, and
  constraint checking. But its vertex types are SQL-specific (tables, views,
  indexes), not Protobuf-specific (messages, enums, services).

- **GraphQL Code Generator** assumes a multigraph with GraphQL's specific
  vertex types (Object, Interface, Union, Enum, Scalar). It can't represent
  Protobuf's field numbering or SQL's CHECK constraints.

Each tool is a perfect fit for its target protocol and a poor fit for everything
else. This isn't a failure of engineering—it's a consequence of fixing the graph
structure at design time.

panproto takes a different approach: it **parameterizes** over the graph
structure. Instead of hardcoding "the schema is a multigraph with these specific
vertex and edge types," panproto lets you specify what kinds of vertices, edges,
and constraints your schema language needs. The same core algorithms—diffing,
merging, migration planning, code generation—work across any graph structure
that fits the framework.

This parameterization is not free. It requires a mathematical framework
expressive enough to describe multigraphs, hypergraphs, and constrained variants
in a uniform way. That framework exists: it comes from the categorical treatment
of schemas developed by [@spivak2012], who showed that database schemas can be
viewed as categories (with tables as objects and foreign keys as morphisms) and
that schema mappings are functors between those categories. panproto generalizes
this functorial perspective beyond relational databases to arbitrary schema
languages.

The key insight from Spivak's work is that once you view schemas as categorical
structures, you get a rich toolkit for free: limits and colimits give you schema
merging and intersection, functors give you schema mappings and translations,
and natural transformations give you data migration along those mappings. These
aren't metaphors—they're precise mathematical constructions with well-defined
properties.

:::{.callout-note}
## The Parameterization Principle

Single-protocol tools fix the graph structure and optimize for a specific
schema language. panproto parameterizes the graph structure and provides uniform
algorithms that work across schema languages. This is the fundamental
architectural difference—and it's what makes cross-protocol schema operations
possible.
:::


## From Graphs to Theories

We've established that schema languages have graph-theoretic structure, and that
different languages need different kinds of graphs. But we've been describing
these graph structures informally—listing their components (vertices, edges,
operations) in prose.

There's a precise way to describe a graph structure: you specify its
**sorts** (the kinds of things that exist), its **operations** (the functions
between those sorts), and its **equations** (the laws those operations obey). A
directed graph, for instance, has two sorts (Vertex and Edge), two operations
(src and tgt, both from Edge to Vertex), and no equations. A multigraph has the
same description—the "multi" comes from not requiring edges to be uniquely
determined by their source and target.

This sort-operation-equation description is called a **theory**, and the things
that satisfy a theory are called **models**. A specific ATProto schema is a
model of the multigraph theory. A specific SQL schema is a model of the
hypergraph-with-constraints theory.

If the language of sorts, operations, and equations sounds suspiciously like an
abstract algebra course, that's because it is. The next chapter introduces the
specific algebraic framework panproto uses—Generalized Algebraic Theories—and
shows how the Rust implementation represents theories and models as concrete
data structures. The math will earn its keep: it's what makes panproto's
parameterization possible.


## Summary

This chapter established three ideas:

1. **Schema languages have graph-theoretic structure.** Types are vertices;
   fields, columns, and properties are edges. This is not a metaphor—it's a
   precise structural description.

2. **Different schema languages need different kinds of graphs.** Multigraphs
   for parallel fields (most languages). Hypergraphs for multi-ary constraints
   (SQL). Constraints as annotations on any of these.

3. **Single-protocol tools hardcode a specific graph structure.** This is why
   Buf can't do SQL migrations and Atlas can't lint Protobuf. panproto
   parameterizes over the graph structure instead.

The question now is: how do you precisely describe a graph structure so that
software can work with it generically? That's the subject of the next chapter.

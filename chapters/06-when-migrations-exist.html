6  When Migrations Exist

Not every proposed vertex map and edge map constitutes a valid migration. Some schema changes are structurally incoherent—they ask the engine to do something impossible, like satisfy a tighter constraint with no new data, or resolve an ambiguous edge label with no disambiguating information. The engine must detect these situations before attempting to lift data, because a migration that silently produces garbage is worse than one that refuses to run.

In most migration tools, the validity checks are hardcoded: a fixed list of rules, written by hand, specific to one protocol. panproto takes a different approach. The existence conditions are derived from the schema theory and instance theory. The engine inspects the theory’s sorts, operations, and axioms, and assembles the appropriate checks automatically. This is why the same engine can validate migrations for SQL (where you need foreign key consistency) and ATProto (where you need reachability in a tree).

This chapter explains how that derivation works and demonstrates it through four worked examples of migrations that fail.

6.1 Theory-Derived Conditions

The engine determines which existence checks to apply by inspecting the schema theory’s signature—its sorts and operations. Each sort or structural feature triggers a specific check. The checks are cumulative: a schema theory with more sorts requires more checks.

Here is the mapping from theory features to existence conditions:

6.1.1 Kind Consistency

Trigger: the schema theory has a Kind sort (a sort classifying vertices by their data type).

Condition: for every vertex \(v\) in the source schema, \(\text{kind}(v)\) must be compatible with \(\text{kind}(\phi_V(v))\) in the target schema. “Compatible” means the kinds are equal, or the source kind is a subtype of the target kind according to the protocol’s subtyping rules.

Intuition: you can’t map a string vertex to a union{string, integer} vertex if the protocol doesn’t support implicit widening. The kinds must match (or the target must be strictly more general, in protocols that support subsumption).

Protocols affected: all protocols have some notion of kind. The strictness of “compatible” varies: Protobuf is strict (kinds must be equal), ATProto allows some subtyping (string is compatible with union{string}), GraphQL respects interface subtyping.

6.1.2 Edge Compatibility

Trigger: the schema theory has an Edge sort with src and tgt operations (operations assigning source and target vertices to each edge).

Condition: the edge map must be compatible with the vertex map. Formally, for every edge \(e : u \to v\) in the source schema:

\[\text{src}(\phi_E(e)) = \phi_V(\text{src}(e)) \quad \text{and} \quad \text{tgt}(\phi_E(e)) = \phi_V(\text{tgt}(e))\]

This is the naturality condition: the edge map and vertex map must form a graph homomorphism.

Intuition: if field text goes from Post to Text in v1, and you map Post ↦ Post and Text ↦ Text, then you must map text to an edge that also goes from Post to Text. You can’t map it to an edge from Post to Integer—that would break the graph structure.

Protocols affected: all protocols with edges (which is all of them—edges encode fields and relationships).

6.1.3 Constraint Compatibility

Trigger: the schema theory has a Constraint sort (a sort attaching invariants to vertices or edges).

Condition: for every constraint \(c\) on a source vertex or edge, the corresponding constraint in the target must be at least as permissive. That is, the set of values satisfying the target constraint must be a superset of the values satisfying the source constraint.

More precisely, if \(c_1\) constrains vertex \(v\) in the source schema, and \(c_2\) constrains \(\phi_V(v)\) in the target schema, then the engine checks:

\[\text{sat}(c_2) \supseteq \text{sat}(c_1)\]

where \(\text{sat}(c)\) is the set of values satisfying constraint \(c\).

Intuition: if the source schema says maxLength: 3000 and the target says maxLength: 300, then a valid source value of length 1000 would violate the target constraint. The migration can’t guarantee that all source data fits in the target’s tighter constraint, so it fails.

The direction here matters. We are checking whether target data can be restricted to the source schema. A tighter target constraint is fine (all target values already satisfy the source constraint). A looser target constraint is the problem: the source expects tight bounds, but the target allows values that would violate them.

Wait—let’s be precise. In the backward direction (lifting v2 data to v1), we need v2 data to satisfy v1’s constraints. So the check is: for each source (v1) constraint, every value that satisfies the target (v2) constraint must also satisfy the source constraint. This means \(\text{sat}(c_2) \subseteq \text{sat}(c_1)\)—the target must be tighter or equal to the source.

For the forward direction, the condition reverses. panproto checks the appropriate direction based on which lift direction you’re compiling.

Protocols affected: ATProto, SQL, Protobuf, GraphQL, JSON Schema—every protocol with constraints.

6.1.4 Pushout Complement Uniqueness

Trigger: the schema theory allows parallel edges (the multigraph condition from ThMulti).

Condition: when ancestor contraction creates a new parent-child relationship (because an intermediate node was removed), there must be a unique way to label the new edge. If the schema allows multiple edges between the same pair of vertices, the engine cannot determine which label to use without additional resolver data.

This condition comes from the theory of algebraic graph transformation. In the double-pushout (DPO) approach (Ehrig et al. 2006), a graph rewriting rule can only be applied when a pushout complement exists, and that complement is unique. The pushout complement existence condition—also called the gluing condition—ensures that the rewriting doesn’t create dangling edges or ambiguous identifications.

In panproto’s setting, the analogous condition arises during ancestor contraction. When a removed intermediate vertex \(w\) has a parent edge \(e_1 : u \to w\) and a child edge \(e_2 : w \to v\), contraction needs to create a direct edge \(e' : u \to v\). If the target schema has multiple edges from \(\phi_V(u)\) to \(\phi_V(v)\), the choice of \(e'\) is ambiguous.

Intuition: imagine a schema where a Person has two edges to Address: homeAddress and workAddress. If a migration removes an intermediate ContactInfo node, the engine needs to reconnect Person to Address directly, but which label should it use? Without resolver data, the migration is ambiguous and therefore invalid.

Protocols affected: ATProto, GraphQL, JSON Schema (all protocols with ThMulti in the schema theory).

6.1.5 Signature Coherence and Simultaneity

Trigger: the schema theory has a HyperEdge sort (edges connecting more than two vertices simultaneously).

Condition: when a migration maps the vertices of a hyperedge, all vertices in the hyperedge must be mapped, or none of them. You cannot partially map a hyperedge—that would create a dangling hyperedge connecting some mapped vertices and some unmapped ones.

This is the simultaneity condition for hypergraph morphisms, discussed by Barr and Wells (1990) in the context of sketch morphisms. A sketch morphism must preserve the distinguished cones and cocones of the sketch, which requires that the underlying graph homomorphism respect the hypergraph structure.

Intuition: in SQL, a compound foreign key (author_id, post_id) → (users.id, posts.id) is a hyperedge connecting three tables simultaneously. If you map users and posts but drop the join table, the foreign key becomes incoherent. Either all participants in the hyperedge survive, or the hyperedge itself must be explicitly removed.

Protocols affected: SQL (the primary user of ThHyperEdge).

6.1.6 Reachability

Trigger: the instance theory uses W-types (ThWType).

Condition: after applying the vertex map, the root vertex of every instance must still be reachable. More precisely, for every path from the root to a leaf in the source schema, the image of that path under the migration must still be a valid path in the target schema (possibly shortened by contraction, but never broken).

This is a global condition: it cannot be checked vertex-by-vertex. The engine performs a reachability analysis on the target schema, restricted to the image of the migration, starting from the image of the root vertex.

Intuition: if a migration removes a vertex that lies on every path from the root to some subtree, that entire subtree becomes unreachable. The data it contains cannot be preserved. If the subtree contains required fields, the migration is invalid.

Protocols affected: ATProto, GraphQL, JSON Schema (all protocols with W-type instances).

6.2 The Check Pipeline

The engine assembles these checks into a pipeline based on the theory structure. The pipeline runs during compile and reports all failures at once (it does not short-circuit on the first failure, because developers need to see the full picture).

The pipeline is data-driven: the engine reads the theory and builds the pipeline at initialization time. Adding a new sort to a theory automatically adds the corresponding check. Removing a sort removes the check. There is no hardcoded list of protocol-specific rules.

6.3 Worked Example 1: Constraint Tightening

Let’s see a migration that fails the constraint compatibility check.

import { Panproto } from '@panproto/core';

const panproto = await Panproto.init();
const atproto = panproto.protocol('atproto');

// start snippet constraint-tightening
const schemaOld = atproto.schema()
  .vertex('post', 'record', { nsid: 'app.bsky.feed.post' })
  .vertex('post:body', 'object')
  .vertex('post:body.text', 'string')
  .edge('post', 'post:body', 'record-schema')
  .edge('post:body', 'post:body.text', 'prop', { name: 'text' })
  .constraint('post:body.text', 'maxLength', '3000')
  .build();

const schemaNew = atproto.schema()
  .vertex('post', 'record', { nsid: 'app.bsky.feed.post' })
  .vertex('post:body', 'object')
  .vertex('post:body.text', 'string')
  .edge('post', 'post:body', 'record-schema')
  .edge('post:body', 'post:body.text', 'prop', { name: 'text' })
  .constraint('post:body.text', 'maxLength', '300') // Tightened!
  .build();

const migBuilder = panproto.migration(schemaOld, schemaNew)
  .map('post', 'post')
  .map('post:body', 'post:body')
  .map('post:body.text', 'post:body.text');

const report = panproto.checkExistence(schemaOld, schemaNew, migBuilder);
// report.valid === false
// report.errors[0].kind === 'constraint-tightened'
// end snippet constraint-tightening

The scenario: v1 allows post text up to 3000 characters (maxLength: 3000). v2 tightens this to 300 characters (maxLength: 300). The migration maps v1’s Text vertex to v2’s Text vertex.

Why it fails: We are compiling a backward migration (lifting v2 data to v1 compatibility). A v2 record has text with at most 300 characters, which is within v1’s 3000-character limit. So backward migration is actually fine—the target (v2) constraint is tighter, which means all v2 data satisfies v1’s constraint.

But now consider the forward direction: migrating v1 data to v2 compatibility. A v1 record could have text with 1000 characters, which violates v2’s 300-character limit. The engine detects this: sat(maxLength: 300) ⊄ sat(maxLength: 3000).

The engine reports:

ExistenceError: Constraint tightening on vertex Text
  Source constraint: maxLength ≤ 3000
  Target constraint: maxLength ≤ 300
  Direction: forward
  Reason: source data may violate target constraint
NoteConstraint direction matters

Constraint tightening blocks forward migration but permits backward migration. Constraint loosening blocks backward migration but permits forward migration. The engine tracks the migration direction and applies the check accordingly. This is why panproto compiles forward and backward migrations separately—they have different existence conditions.

6.4 Worked Example 2: Kind Inconsistency

Consider a migration that maps a string vertex to a union{string, integer} vertex.

In ATProto’s schema theory, vertex kinds form a lattice. The kind string is not a subtype of union{string, integer}—rather, string is a component of the union, but the kinds themselves are different. An ATProto record anchored to a union{string, integer} vertex can hold either a string or an integer; a record anchored to a string vertex can only hold a string.

The migration maps StringField ↦ UnionField in the vertex map. The engine checks kind consistency:

kind(StringField) = string
kind(φ_V(StringField)) = kind(UnionField) = union{string, integer}
string ≠ union{string, integer}

The engine then consults the protocol’s subtyping rules. ATProto permits widening from a concrete type to a union containing that type in the forward direction (a string value is a valid union value). But in the backward direction, a union value might be an integer, which is not a valid string.

So this migration:

  • Passes kind consistency for the forward direction (string embeds into union{string, integer}).
  • Fails kind consistency for the backward direction (union{string, integer} does not embed into string).

The engine reports:

ExistenceError: Kind inconsistency on vertex StringField
  Source kind: string
  Target kind: union{string, integer}
  Direction: backward
  Reason: target kind is not a subkind of source kind

The fix depends on your intent. If you want backward compatibility, you need a resolver—a function that handles the integer case (e.g., by converting it to a string representation, or by signaling an error). Resolvers are covered in Chapter 8.

6.5 Worked Example 3: Required Field Missing

This is perhaps the most common migration failure in practice. Consider v1 with a required name field and v2 without it. A migration maps v2’s vertices into v1, but name has no preimage—no v2 vertex maps to v1’s Name vertex.

In the backward direction (lifting v2 data to v1), the engine needs to produce a v1 record with a name field. But v2 data doesn’t contain a name. If name is optional in v1, this is fine—the engine produces a record with name absent. But if name is required, the engine cannot produce a valid v1 record.

The check works as follows:

  1. The engine identifies all v1 vertices and edges that are not in the image of the migration.
  2. For each unmapped vertex, it checks whether the vertex is required (reachable from the root via a path of required edges).
  3. If an unmapped vertex is required, the migration fails.

This check is a combination of edge compatibility and reachability. The engine uses the schema theory’s constraint sort (to determine which edges are required vs. optional) and, for W-type instances, performs a reachability analysis from the root.

ExistenceError: Required field missing
  Unmapped vertex: Name (kind: string)
  Required path: Post → name → Name
  Direction: backward
  Reason: no source vertex maps to Name, and Name is reachable
          via a required path from the root

The fix is to provide a default value for the missing field. In panproto, defaults are specified as part of the migration morphism:

// Adding a default makes the migration valid
const migration = {
  vertexMap: { /* ... existing maps ... */ },
  edgeMap: { /* ... existing maps ... */ },
  defaults: {
    Name: { value: "Unknown" }
  }
};

With the default in place, the lift operation fills in "Unknown" for the missing name field, and the migration passes.

6.6 Worked Example 4: Pushout Complement Ambiguity

This example requires a multigraph schema—one with parallel edges between the same pair of vertices. ATProto, GraphQL, and JSON Schema all allow this (via ThMulti in the schema theory).

Consider a schema with:

  • Vertices: Person, ContactInfo, Address
  • Edges: contact : Person → ContactInfo, location : ContactInfo → Address

And a v2 schema that removes ContactInfo, connecting Person directly to Address via two edges:

  • Vertices: Person, Address
  • Edges: homeAddress : Person → Address, workAddress : Person → Address

The migration maps Person ↦ Person and Address ↦ Address, but ContactInfo has no preimage (it’s been removed). During ancestor contraction, the engine needs to create a direct edge from Person to Address to replace the two-hop path Person → ContactInfo → Address.

But there are two edges from Person to Address in v2: homeAddress and workAddress. Which one should the contracted edge map to?

The engine cannot decide. This is the pushout complement uniqueness failure—the contraction produces an ambiguous edge.

ExistenceError: Pushout complement ambiguity
  Contracted path: Person → ContactInfo → Address
  Candidate edges: homeAddress, workAddress
  Direction: backward
  Reason: ancestor contraction produces ambiguous edge label;
          multiple edges from Person to Address in target schema

This failure has its roots in algebraic graph transformation theory. Ehrig et al. (2006) showed that DPO graph rewriting requires the pushout complement to exist and be unique. When it’s not unique, the rewriting is ambiguous. Löwe (1993) studied the single-pushout (SPO) approach, which resolves ambiguity by making a default choice (typically deletion), but panproto follows the DPO tradition and requires explicit disambiguation.

The fix is to provide resolver data in the migration morphism:

const migration = {
  vertexMap: { Person: "Person", Address: "Address" },
  edgeMap: { /* ... */ },
  resolvers: {
    // When contracting Person → ContactInfo → Address,
    // use the homeAddress edge
    "Person→ContactInfo→Address": "homeAddress"
  }
};

With the resolver, the engine knows which edge to use, and the migration passes. Resolvers are discussed more fully in Chapter 8.

6.7 Why Derivation Matters

You might wonder: is the theory-derived approach actually better than a hardcoded list of checks? After all, the checks themselves are not complicated—constraint compatibility, kind consistency, and so on are individually straightforward.

The advantage is extensibility and correctness. Consider what happens when you add a new protocol.

With hardcoded checks: you study the new protocol’s semantics, identify the relevant existence conditions, write the checks by hand, and hope you haven’t missed any. If the protocol has unusual features (like hyperedges, or a novel subtyping relation), you need to invent new checks from scratch. The check suite grows linearly with the number of protocols, and cross-protocol consistency is a manual audit.

With theory-derived checks: you write the schema theory and instance theory for the new protocol. The engine reads the theory’s sorts and operations, matches them to the catalogue of known check generators, and assembles the pipeline automatically. If the theory has a Constraint sort, the constraint check fires. If the theory has parallel edges, the pushout complement check fires. You get exactly the checks that the theory requires, no more and no less.

This is a direct consequence of the framework described by Lack and Sobociński (2005): in adhesive categories (which include the categories of graphs and hypergraphs that panproto works with), the existence of pushout complements is decidable, and the conditions are determined by the categorical structure. panproto mechanizes this decidability.

NoteThe key insight

Other tools hardcode their existence checks. panproto derives them from the theory. This is why the same engine works for SQL (where you need foreign key consistency and hyperedge simultaneity) and ATProto (where you need tree reachability and pushout complement uniqueness). The checks are not the same, but the mechanism that produces them is.

6.8 Existence vs. Uniqueness

There is a subtlety we’ve been glossing over. The checks above determine whether a migration exists—whether there is any valid way to lift data through the proposed morphism. But in some cases, a migration can exist without being unique: there might be multiple valid ways to lift the data, and the engine needs to choose one.

Uniqueness failures are strictly less severe than existence failures. An existence failure means “this migration is impossible.” A uniqueness failure means “this migration is possible but ambiguous.” The engine handles them differently:

  • Existence failure: the engine rejects the migration and reports the error. No data is produced.
  • Uniqueness failure: the engine requests resolver data. With resolvers, the migration becomes unique and can proceed.

The pushout complement ambiguity (Worked Example 4) is actually a uniqueness failure rather than an existence failure. The migration could proceed by choosing either homeAddress or workAddress—it just can’t proceed without being told which one. The resolver provides the missing information.

In the terminology of algebraic graph transformation: existence corresponds to the gluing condition (Ehrig et al. 2006), and uniqueness corresponds to the identification condition. Both must be satisfied for a DPO rewriting step to be well-defined.

6.9 The Role of Axioms

So far we’ve focused on the theory’s signature (sorts and operations) as the trigger for existence checks. But the theory’s axioms also play a role. Axioms constrain how sorts and operations interact, and these constraints can strengthen or weaken the existence conditions.

For example, the schema theory for CQL includes the axiom that composition of morphisms is associative and unital (the category axioms). This axiom implies that the edge compatibility check is stronger for CQL than for, say, ATProto: in CQL, you must preserve not just individual edges but entire composable paths.

Similarly, if a schema theory includes an axiom stating that every vertex has a unique kind (a functionality axiom for the kind operation), then the kind consistency check can use equality rather than subtyping.

The engine reads the axioms and parameterizes the checks accordingly. In practice, most axioms affect the strictness of a check rather than its presence: the check fires because the sort exists, and the axiom determines how strictly it fires.

6.10 Putting It Together: A Decision Procedure

Given a proposed migration morphism \((\phi_V, \phi_E)\) between two schemas \(S_1\) and \(S_2\), with schema theory \(T_S\) and instance theory \(T_I\), the engine runs the following decision procedure:

  1. Read the sorts of \(T_S\). For each sort, look up the corresponding check generator.
  2. Read the axioms of \(T_S\). For each axiom, determine how it parameterizes the relevant checks.
  3. Read \(T_I\) to determine whether the reachability check applies (W-type instances) or not (functor instances).
  4. Run all applicable checks against \((\phi_V, \phi_E)\), \(S_1\), and \(S_2\).
  5. Collect results:
    • All checks pass → migration exists and is unique. Proceed.
    • Some checks fail as existence errors → migration does not exist. Report errors.
    • Some checks fail as uniqueness errors → migration exists but is ambiguous. Request resolvers.
  6. If resolvers are provided, re-run the uniqueness checks with resolver data. If they now pass, proceed.

This procedure is total: it always terminates with a definite answer. It is sound: if it says the migration exists, the lift operation will produce valid output. And it is derived: the checks it runs are determined by the theory, not by protocol-specific code.

6.11 Comparison with Other Tools

To appreciate the theory-derived approach, it’s helpful to see how other tools handle existence checking.

CQL (Schultz et al. 2017) has rigorous existence checking, but only for functor instances over categories. It checks functoriality of the schema morphism (which subsumes edge compatibility and kind consistency in the categorical setting) and relies on the properties of adhesive categories for pushout complement existence. panproto generalizes this to arbitrary schema theories.

Cambria (Litt et al. 2022) does not have explicit existence checking. It uses a lens combinator language and relies on the programmer to compose valid lenses. Invalid lenses produce runtime errors. panproto’s compile-time checks catch these errors before any data is processed.

SQL DDL (ALTER TABLE, etc.) performs some existence checking (you can’t drop a column that’s referenced by a foreign key without CASCADE), but the checks are ad hoc and specific to the relational model. panproto derives analogous checks from the theory.

Protobuf has compatibility rules (field numbers, wire types) that function as existence checks, but they’re hardcoded in the protobuf compiler. panproto derives equivalent checks from Protobuf’s schema theory.

The advantage of the theory-derived approach is not that it produces different checks—in most cases, the checks are equivalent to what you’d write by hand. The advantage is that the checks are guaranteed to be complete for the given theory. If you miss a check in a hardcoded system, you’ll discover it when a migration silently corrupts data. In panproto, the engine derives all necessary checks from the theory, so completeness is structural rather than aspirational.

6.12 Summary

  • Existence conditions determine whether a proposed migration can be validly applied to data. Not every vertex/edge map is a valid migration.
  • The conditions are derived from the theory: the engine inspects the schema theory’s sorts and axioms to determine which checks apply.
  • Six families of checks exist:
    1. Kind consistency (triggered by Kind sort)
    2. Edge compatibility (triggered by Edge sort with src/tgt)
    3. Constraint compatibility (triggered by Constraint sort)
    4. Pushout complement uniqueness (triggered by parallel edges / ThMulti)
    5. Signature coherence + simultaneity (triggered by HyperEdge sort)
    6. Reachability (triggered by W-type instance theory)
  • Existence failures are fatal: the migration cannot proceed. Uniqueness failures can be resolved with resolver data.
  • The decision procedure is total, sound, and derived—it always terminates, never produces invalid output, and never requires protocol-specific code.

# When Migrations Exist {#sec-existence}

Not every proposed vertex map and edge map constitutes a valid migration. Some schema changes are structurally incoherent---they ask the engine to do something impossible, like satisfy a tighter constraint with no new data, or resolve an ambiguous edge label with no disambiguating information. The engine must detect these situations *before* attempting to lift data, because a migration that silently produces garbage is worse than one that refuses to run.

In most migration tools, the validity checks are hardcoded: a fixed list of rules, written by hand, specific to one protocol. panproto takes a different approach. The existence conditions are *derived* from the schema theory and instance theory. The engine inspects the theory's sorts, operations, and axioms, and assembles the appropriate checks automatically. This is why the same engine can validate migrations for SQL (where you need foreign key consistency) and ATProto (where you need reachability in a tree).

This chapter explains how that derivation works and demonstrates it through four worked examples of migrations that fail.


## Theory-Derived Conditions

The engine determines which existence checks to apply by inspecting the schema theory's signature---its sorts and operations. Each sort or structural feature triggers a specific check. The checks are cumulative: a schema theory with more sorts requires more checks.

Here is the mapping from theory features to existence conditions:

### Kind Consistency

**Trigger**: the schema theory has a `Kind` sort (a sort classifying vertices by their data type).

**Condition**: for every vertex $v$ in the source schema, $\text{kind}(v)$ must be compatible with $\text{kind}(\phi_V(v))$ in the target schema. "Compatible" means the kinds are equal, or the source kind is a subtype of the target kind according to the protocol's subtyping rules.

**Intuition**: you can't map a `string` vertex to a `union{string, integer}` vertex if the protocol doesn't support implicit widening. The kinds must match (or the target must be strictly more general, in protocols that support subsumption).

**Protocols affected**: all protocols have some notion of kind. The strictness of "compatible" varies: Protobuf is strict (kinds must be equal), ATProto allows some subtyping (`string` is compatible with `union{string}`), GraphQL respects interface subtyping.

### Edge Compatibility

**Trigger**: the schema theory has an `Edge` sort with `src` and `tgt` operations (operations assigning source and target vertices to each edge).

**Condition**: the edge map must be compatible with the vertex map. Formally, for every edge $e : u \to v$ in the source schema:

$$\text{src}(\phi_E(e)) = \phi_V(\text{src}(e)) \quad \text{and} \quad \text{tgt}(\phi_E(e)) = \phi_V(\text{tgt}(e))$$

This is the naturality condition: the edge map and vertex map must form a graph homomorphism.

**Intuition**: if field `text` goes from `Post` to `Text` in v1, and you map `Post ↦ Post` and `Text ↦ Text`, then you must map `text` to an edge that also goes from `Post` to `Text`. You can't map it to an edge from `Post` to `Integer`---that would break the graph structure.

**Protocols affected**: all protocols with edges (which is all of them---edges encode fields and relationships).

### Constraint Compatibility

**Trigger**: the schema theory has a `Constraint` sort (a sort attaching invariants to vertices or edges).

**Condition**: for every constraint $c$ on a source vertex or edge, the corresponding constraint in the target must be *at least as permissive*. That is, the set of values satisfying the target constraint must be a superset of the values satisfying the source constraint.

More precisely, if $c_1$ constrains vertex $v$ in the source schema, and $c_2$ constrains $\phi_V(v)$ in the target schema, then the engine checks:

$$\text{sat}(c_2) \supseteq \text{sat}(c_1)$$

where $\text{sat}(c)$ is the set of values satisfying constraint $c$.

**Intuition**: if the source schema says `maxLength: 3000` and the target says `maxLength: 300`, then a valid source value of length 1000 would violate the target constraint. The migration can't guarantee that all source data fits in the target's tighter constraint, so it fails.

The direction here matters. We are checking whether target data can be *restricted* to the source schema. A tighter target constraint is fine (all target values already satisfy the source constraint). A *looser* target constraint is the problem: the source expects tight bounds, but the target allows values that would violate them.

Wait---let's be precise. In the *backward* direction (lifting v2 data to v1), we need v2 data to satisfy v1's constraints. So the check is: for each source (v1) constraint, every value that satisfies the target (v2) constraint must also satisfy the source constraint. This means $\text{sat}(c_2) \subseteq \text{sat}(c_1)$---the target must be *tighter or equal* to the source.

For the *forward* direction, the condition reverses. panproto checks the appropriate direction based on which lift direction you're compiling.

**Protocols affected**: ATProto, SQL, Protobuf, GraphQL, JSON Schema---every protocol with constraints.

### Pushout Complement Uniqueness

**Trigger**: the schema theory allows parallel edges (the multigraph condition from ThMulti).

**Condition**: when ancestor contraction creates a new parent-child relationship (because an intermediate node was removed), there must be a unique way to label the new edge. If the schema allows multiple edges between the same pair of vertices, the engine cannot determine which label to use without additional resolver data.

This condition comes from the theory of algebraic graph transformation. In the double-pushout (DPO) approach [@ehrig2006], a graph rewriting rule can only be applied when a *pushout complement* exists, and that complement is unique. The pushout complement existence condition---also called the *gluing condition*---ensures that the rewriting doesn't create dangling edges or ambiguous identifications.

In panproto's setting, the analogous condition arises during ancestor contraction. When a removed intermediate vertex $w$ has a parent edge $e_1 : u \to w$ and a child edge $e_2 : w \to v$, contraction needs to create a direct edge $e' : u \to v$. If the target schema has multiple edges from $\phi_V(u)$ to $\phi_V(v)$, the choice of $e'$ is ambiguous.

**Intuition**: imagine a schema where a `Person` has two edges to `Address`: `homeAddress` and `workAddress`. If a migration removes an intermediate `ContactInfo` node, the engine needs to reconnect `Person` to `Address` directly, but which label should it use? Without resolver data, the migration is ambiguous and therefore invalid.

**Protocols affected**: ATProto, GraphQL, JSON Schema (all protocols with ThMulti in the schema theory).

### Signature Coherence and Simultaneity

**Trigger**: the schema theory has a `HyperEdge` sort (edges connecting more than two vertices simultaneously).

**Condition**: when a migration maps the vertices of a hyperedge, *all* vertices in the hyperedge must be mapped, or *none* of them. You cannot partially map a hyperedge---that would create a dangling hyperedge connecting some mapped vertices and some unmapped ones.

This is the *simultaneity condition* for hypergraph morphisms, discussed by @barr1990 in the context of sketch morphisms. A sketch morphism must preserve the distinguished cones and cocones of the sketch, which requires that the underlying graph homomorphism respect the hypergraph structure.

**Intuition**: in SQL, a compound foreign key `(author_id, post_id) → (users.id, posts.id)` is a hyperedge connecting three tables simultaneously. If you map `users` and `posts` but drop the join table, the foreign key becomes incoherent. Either all participants in the hyperedge survive, or the hyperedge itself must be explicitly removed.

**Protocols affected**: SQL (the primary user of ThHyperEdge).

### Reachability

**Trigger**: the instance theory uses W-types (ThWType).

**Condition**: after applying the vertex map, the root vertex of every instance must still be reachable. More precisely, for every path from the root to a leaf in the source schema, the image of that path under the migration must still be a valid path in the target schema (possibly shortened by contraction, but never broken).

This is a global condition: it cannot be checked vertex-by-vertex. The engine performs a reachability analysis on the *target* schema, restricted to the image of the migration, starting from the image of the root vertex.

**Intuition**: if a migration removes a vertex that lies on *every* path from the root to some subtree, that entire subtree becomes unreachable. The data it contains cannot be preserved. If the subtree contains required fields, the migration is invalid.

**Protocols affected**: ATProto, GraphQL, JSON Schema (all protocols with W-type instances).


## The Check Pipeline

The engine assembles these checks into a pipeline based on the theory structure. The pipeline runs during `compile` and reports all failures at once (it does not short-circuit on the first failure, because developers need to see the full picture).

```{mermaid}
%%| label: fig-check-pipeline
%%| fig-cap: "The existence check pipeline. Each check is triggered by a theory feature."
flowchart TD
    M["Migration morphism (φ_V, φ_E)"] --> K{Kind sort?}
    K -->|yes| KC["Kind consistency check"]
    K -->|no| E
    KC --> E{Edge sort?}
    E -->|yes| EC["Edge compatibility check"]
    E -->|no| C
    EC --> C{Constraint sort?}
    C -->|yes| CC["Constraint compatibility check"]
    C -->|no| P
    CC --> P{Parallel edges?}
    P -->|yes| PC["Pushout complement uniqueness"]
    P -->|no| H
    PC --> H{HyperEdge sort?}
    H -->|yes| HC["Signature coherence + simultaneity"]
    H -->|no| R
    HC --> R{W-type instances?}
    R -->|yes| RC["Reachability check"]
    R -->|no| OK["✓ Migration exists"]
    RC --> OK
    KC -.->|fail| F["✗ Migration does not exist"]
    EC -.->|fail| F
    CC -.->|fail| F
    PC -.->|fail| F
    HC -.->|fail| F
    RC -.->|fail| F
```

The pipeline is *data-driven*: the engine reads the theory and builds the pipeline at initialization time. Adding a new sort to a theory automatically adds the corresponding check. Removing a sort removes the check. There is no hardcoded list of protocol-specific rules.


## Worked Example 1: Constraint Tightening

Let's see a migration that fails the constraint compatibility check.

```{.typescript include="../code/ts/06-existence-check.ts" snippet="constraint-tightening"}
```

The scenario: v1 allows post text up to 3000 characters (`maxLength: 3000`). v2 tightens this to 300 characters (`maxLength: 300`). The migration maps v1's `Text` vertex to v2's `Text` vertex.

**Why it fails**: We are compiling a backward migration (lifting v2 data to v1 compatibility). A v2 record has text with at most 300 characters, which is within v1's 3000-character limit. So backward migration is actually *fine*---the target (v2) constraint is tighter, which means all v2 data satisfies v1's constraint.

But now consider the *forward* direction: migrating v1 data to v2 compatibility. A v1 record could have text with 1000 characters, which violates v2's 300-character limit. The engine detects this: `sat(maxLength: 300) ⊄ sat(maxLength: 3000)`.

The engine reports:

```
ExistenceError: Constraint tightening on vertex Text
  Source constraint: maxLength ≤ 3000
  Target constraint: maxLength ≤ 300
  Direction: forward
  Reason: source data may violate target constraint
```

:::{.callout-note}
## Constraint direction matters

Constraint tightening blocks *forward* migration but permits *backward* migration. Constraint *loosening* blocks backward migration but permits forward migration. The engine tracks the migration direction and applies the check accordingly. This is why panproto compiles forward and backward migrations separately---they have different existence conditions.
:::


## Worked Example 2: Kind Inconsistency

Consider a migration that maps a `string` vertex to a `union{string, integer}` vertex.

In ATProto's schema theory, vertex kinds form a lattice. The kind `string` is not a subtype of `union{string, integer}`---rather, `string` is a *component* of the union, but the kinds themselves are different. An ATProto record anchored to a `union{string, integer}` vertex can hold either a string or an integer; a record anchored to a `string` vertex can only hold a string.

The migration maps `StringField ↦ UnionField` in the vertex map. The engine checks kind consistency:

```
kind(StringField) = string
kind(φ_V(StringField)) = kind(UnionField) = union{string, integer}
string ≠ union{string, integer}
```

The engine then consults the protocol's subtyping rules. ATProto permits widening from a concrete type to a union *containing* that type in the forward direction (a string value is a valid union value). But in the backward direction, a union value might be an integer, which is not a valid string.

So this migration:

- **Passes** kind consistency for the forward direction (string embeds into union{string, integer}).
- **Fails** kind consistency for the backward direction (union{string, integer} does not embed into string).

The engine reports:

```
ExistenceError: Kind inconsistency on vertex StringField
  Source kind: string
  Target kind: union{string, integer}
  Direction: backward
  Reason: target kind is not a subkind of source kind
```

The fix depends on your intent. If you want backward compatibility, you need a *resolver*---a function that handles the integer case (e.g., by converting it to a string representation, or by signaling an error). Resolvers are covered in [Chapter 8](08-lenses.qmd).


## Worked Example 3: Required Field Missing

This is perhaps the most common migration failure in practice. Consider v1 with a required `name` field and v2 without it. A migration maps v2's vertices into v1, but `name` has no preimage---no v2 vertex maps to v1's `Name` vertex.

In the backward direction (lifting v2 data to v1), the engine needs to produce a v1 record with a `name` field. But v2 data doesn't contain a name. If `name` is optional in v1, this is fine---the engine produces a record with `name` absent. But if `name` is required, the engine cannot produce a valid v1 record.

The check works as follows:

1. The engine identifies all v1 vertices and edges that are *not* in the image of the migration.
2. For each unmapped vertex, it checks whether the vertex is required (reachable from the root via a path of required edges).
3. If an unmapped vertex is required, the migration fails.

This check is a combination of edge compatibility and reachability. The engine uses the schema theory's constraint sort (to determine which edges are required vs. optional) and, for W-type instances, performs a reachability analysis from the root.

```
ExistenceError: Required field missing
  Unmapped vertex: Name (kind: string)
  Required path: Post → name → Name
  Direction: backward
  Reason: no source vertex maps to Name, and Name is reachable
          via a required path from the root
```

The fix is to provide a *default value* for the missing field. In panproto, defaults are specified as part of the migration morphism:

```typescript
// Adding a default makes the migration valid
const migration = {
  vertexMap: { /* ... existing maps ... */ },
  edgeMap: { /* ... existing maps ... */ },
  defaults: {
    Name: { value: "Unknown" }
  }
};
```

With the default in place, the lift operation fills in `"Unknown"` for the missing `name` field, and the migration passes.


## Worked Example 4: Pushout Complement Ambiguity

This example requires a multigraph schema---one with parallel edges between the same pair of vertices. ATProto, GraphQL, and JSON Schema all allow this (via ThMulti in the schema theory).

Consider a schema with:

- Vertices: `Person`, `ContactInfo`, `Address`
- Edges: `contact : Person → ContactInfo`, `location : ContactInfo → Address`

And a v2 schema that removes `ContactInfo`, connecting `Person` directly to `Address` via two edges:

- Vertices: `Person`, `Address`
- Edges: `homeAddress : Person → Address`, `workAddress : Person → Address`

The migration maps `Person ↦ Person` and `Address ↦ Address`, but `ContactInfo` has no preimage (it's been removed). During ancestor contraction, the engine needs to create a direct edge from `Person` to `Address` to replace the two-hop path `Person → ContactInfo → Address`.

But there are *two* edges from `Person` to `Address` in v2: `homeAddress` and `workAddress`. Which one should the contracted edge map to?

The engine cannot decide. This is the pushout complement uniqueness failure---the contraction produces an ambiguous edge.

```
ExistenceError: Pushout complement ambiguity
  Contracted path: Person → ContactInfo → Address
  Candidate edges: homeAddress, workAddress
  Direction: backward
  Reason: ancestor contraction produces ambiguous edge label;
          multiple edges from Person to Address in target schema
```

This failure has its roots in algebraic graph transformation theory. @ehrig2006 showed that DPO graph rewriting requires the pushout complement to exist and be unique. When it's not unique, the rewriting is ambiguous. @lowe1993 studied the single-pushout (SPO) approach, which resolves ambiguity by making a default choice (typically deletion), but panproto follows the DPO tradition and requires explicit disambiguation.

The fix is to provide *resolver data* in the migration morphism:

```typescript
const migration = {
  vertexMap: { Person: "Person", Address: "Address" },
  edgeMap: { /* ... */ },
  resolvers: {
    // When contracting Person → ContactInfo → Address,
    // use the homeAddress edge
    "Person→ContactInfo→Address": "homeAddress"
  }
};
```

With the resolver, the engine knows which edge to use, and the migration passes. Resolvers are discussed more fully in [Chapter 8](08-lenses.qmd).


## Why Derivation Matters

You might wonder: is the theory-derived approach actually better than a hardcoded list of checks? After all, the checks themselves are not complicated---constraint compatibility, kind consistency, and so on are individually straightforward.

The advantage is *extensibility and correctness*. Consider what happens when you add a new protocol.

**With hardcoded checks**: you study the new protocol's semantics, identify the relevant existence conditions, write the checks by hand, and hope you haven't missed any. If the protocol has unusual features (like hyperedges, or a novel subtyping relation), you need to invent new checks from scratch. The check suite grows linearly with the number of protocols, and cross-protocol consistency is a manual audit.

**With theory-derived checks**: you write the schema theory and instance theory for the new protocol. The engine reads the theory's sorts and operations, matches them to the catalogue of known check generators, and assembles the pipeline automatically. If the theory has a `Constraint` sort, the constraint check fires. If the theory has parallel edges, the pushout complement check fires. You get exactly the checks that the theory requires, no more and no less.

This is a direct consequence of the framework described by @lack2005: in adhesive categories (which include the categories of graphs and hypergraphs that panproto works with), the existence of pushout complements is decidable, and the conditions are determined by the categorical structure. panproto mechanizes this decidability.

:::{.callout-note}
## The key insight

Other tools hardcode their existence checks. panproto derives them from the theory. This is why the same engine works for SQL (where you need foreign key consistency and hyperedge simultaneity) and ATProto (where you need tree reachability and pushout complement uniqueness). The checks are not the same, but the *mechanism that produces them* is.
:::


## Existence vs. Uniqueness

There is a subtlety we've been glossing over. The checks above determine whether a migration *exists*---whether there is any valid way to lift data through the proposed morphism. But in some cases, a migration can exist without being *unique*: there might be multiple valid ways to lift the data, and the engine needs to choose one.

Uniqueness failures are strictly less severe than existence failures. An existence failure means "this migration is impossible." A uniqueness failure means "this migration is possible but ambiguous." The engine handles them differently:

- **Existence failure**: the engine rejects the migration and reports the error. No data is produced.
- **Uniqueness failure**: the engine requests resolver data. With resolvers, the migration becomes unique and can proceed.

The pushout complement ambiguity (Worked Example 4) is actually a uniqueness failure rather than an existence failure. The migration *could* proceed by choosing either `homeAddress` or `workAddress`---it just can't proceed without being told which one. The resolver provides the missing information.

In the terminology of algebraic graph transformation: existence corresponds to the *gluing condition* [@ehrig2006], and uniqueness corresponds to the *identification condition*. Both must be satisfied for a DPO rewriting step to be well-defined.


## The Role of Axioms

So far we've focused on the theory's *signature* (sorts and operations) as the trigger for existence checks. But the theory's *axioms* also play a role. Axioms constrain how sorts and operations interact, and these constraints can strengthen or weaken the existence conditions.

For example, the schema theory for CQL includes the axiom that composition of morphisms is associative and unital (the category axioms). This axiom implies that the edge compatibility check is stronger for CQL than for, say, ATProto: in CQL, you must preserve not just individual edges but entire composable paths.

Similarly, if a schema theory includes an axiom stating that every vertex has a unique kind (a functionality axiom for the `kind` operation), then the kind consistency check can use equality rather than subtyping.

The engine reads the axioms and parameterizes the checks accordingly. In practice, most axioms affect the *strictness* of a check rather than its *presence*: the check fires because the sort exists, and the axiom determines how strictly it fires.


## Putting It Together: A Decision Procedure

Given a proposed migration morphism $(\phi_V, \phi_E)$ between two schemas $S_1$ and $S_2$, with schema theory $T_S$ and instance theory $T_I$, the engine runs the following decision procedure:

1. **Read the sorts of $T_S$**. For each sort, look up the corresponding check generator.
2. **Read the axioms of $T_S$**. For each axiom, determine how it parameterizes the relevant checks.
3. **Read $T_I$** to determine whether the reachability check applies (W-type instances) or not (functor instances).
4. **Run all applicable checks** against $(\phi_V, \phi_E)$, $S_1$, and $S_2$.
5. **Collect results**:
   - All checks pass → migration exists and is unique. Proceed.
   - Some checks fail as existence errors → migration does not exist. Report errors.
   - Some checks fail as uniqueness errors → migration exists but is ambiguous. Request resolvers.
6. **If resolvers are provided**, re-run the uniqueness checks with resolver data. If they now pass, proceed.

This procedure is *total*: it always terminates with a definite answer. It is *sound*: if it says the migration exists, the lift operation will produce valid output. And it is *derived*: the checks it runs are determined by the theory, not by protocol-specific code.


## Comparison with Other Tools

To appreciate the theory-derived approach, it's helpful to see how other tools handle existence checking.

**CQL** [@schultz2017] has rigorous existence checking, but only for functor instances over categories. It checks functoriality of the schema morphism (which subsumes edge compatibility and kind consistency in the categorical setting) and relies on the properties of adhesive categories for pushout complement existence. panproto generalizes this to arbitrary schema theories.

**Cambria** [@litt2022] does not have explicit existence checking. It uses a lens combinator language and relies on the programmer to compose valid lenses. Invalid lenses produce runtime errors. panproto's compile-time checks catch these errors before any data is processed.

**SQL DDL** (ALTER TABLE, etc.) performs some existence checking (you can't drop a column that's referenced by a foreign key without CASCADE), but the checks are ad hoc and specific to the relational model. panproto derives analogous checks from the theory.

**Protobuf** has compatibility rules (field numbers, wire types) that function as existence checks, but they're hardcoded in the protobuf compiler. panproto derives equivalent checks from Protobuf's schema theory.

The advantage of the theory-derived approach is not that it produces *different* checks---in most cases, the checks are equivalent to what you'd write by hand. The advantage is that the checks are *guaranteed to be complete* for the given theory. If you miss a check in a hardcoded system, you'll discover it when a migration silently corrupts data. In panproto, the engine derives all necessary checks from the theory, so completeness is structural rather than aspirational.


## Summary

- **Existence conditions** determine whether a proposed migration can be validly applied to data. Not every vertex/edge map is a valid migration.
- The conditions are **derived from the theory**: the engine inspects the schema theory's sorts and axioms to determine which checks apply.
- Six families of checks exist:
  1. **Kind consistency** (triggered by `Kind` sort)
  2. **Edge compatibility** (triggered by `Edge` sort with `src`/`tgt`)
  3. **Constraint compatibility** (triggered by `Constraint` sort)
  4. **Pushout complement uniqueness** (triggered by parallel edges / ThMulti)
  5. **Signature coherence + simultaneity** (triggered by `HyperEdge` sort)
  6. **Reachability** (triggered by W-type instance theory)
- **Existence failures** are fatal: the migration cannot proceed. **Uniqueness failures** can be resolved with resolver data.
- The decision procedure is **total**, **sound**, and **derived**---it always terminates, never produces invalid output, and never requires protocol-specific code.

8  Lenses: Going Both Ways

Every migration in panproto is bidirectional. When you project data from a source schema to a view schema, you also get a receipt—the complement—that records everything the projection discarded. Given the complement and a (possibly modified) view, you can reconstruct the source. This round-trip property is not an implementation convenience; it is a mathematical guarantee provided by the theory of lenses.

This chapter introduces asymmetric lenses, shows how panproto implements them, and explains the combinator API that lets you build migrations from small, composable pieces.

8.1 The get/put pattern

An asymmetric lens \(\ell: S \rightleftarrows V\) consists of two operations:

  • get : S → (V, C) — Project the source \(S\) to a view \(V\), capturing the complement \(C\) (everything discarded).
  • put : (V, C) → S — Restore the source from a (possibly modified) view and the complement.

The operations satisfy two laws:

8.1.1 GetPut (round-tripping)

\[\mathtt{put}(\mathtt{get}(s)) = s\]

Getting and then putting back (without modifying the view) recovers the original source exactly. No information is lost.

8.1.2 PutGet (view consistency)

\[\pi_1(\mathtt{get}(\mathtt{put}(v, c))) = v\]

Putting a view \(v\) into the source and then getting it back returns \(v\). The view is faithfully represented in the reconstructed source.

Together, these laws say that get and put are mutual inverses up to the complement. The complement is the “slack” that allows the source to carry more information than the view without violating invertibility.

NoteWhy “lens”?

The name comes from the idea of focusing: a lens focuses on a part of a larger structure (the view), while keeping the rest (the complement) available for reconstruction. The metaphor is apt—like an optical lens, a panproto lens lets you see a specific slice of the data while the surrounding context remains intact but out of focus.

8.2 Lenses in panproto

In panproto, get is the restrict pipeline from Chapter 7 plus complement capture. put is reconstruction from the complement. The connection is direct:

  • get runs the five-step W-type restrict pipeline (or functor restrict for SQL) and simultaneously records what was removed at each step.
  • put replays the removal in reverse: re-attach fans, re-resolve edges, re-expand contracted ancestors, re-attach unreachable nodes, and restore pruned anchors.

The lens laws hold because the complement records exactly what get discarded—no more, no less. put has enough information to invert each step, and the sequential structure of the pipeline means the inversions compose correctly.

8.3 The complement

The complement is the central data structure of the lens. It is a receipt for what get threw away, organized by pipeline step.

pub struct Complement {
    /// Nodes pruned in step 1 (anchor not surviving).
    /// Maps node index to the full node (anchor, payload, children).
    pub dropped_nodes: HashMap<u32, Node>,

    /// Arcs (edges) that connected pruned nodes to surviving nodes.
    /// Each entry is (parent_idx, child_idx, edge_label).
    pub dropped_arcs: Vec<(u32, u32, Edge)>,

    /// Edge labels chosen by the resolver in step 4.
    /// Maps (new_parent, child) to the original intermediate path.
    pub contraction_choices: HashMap<(u32, u32), Vec<VertexId>>,

    /// Original parent of each contracted node (before re-parenting).
    pub original_parent: HashMap<u32, u32>,

    /// Fan children pruned in step 5 (hypergraph schemas only).
    pub pruned_fan_children: HashMap<FanId, Vec<u32>>,
}

Each field corresponds to a pipeline step:

Complement field Pipeline step What it records
dropped_nodes Step 1: anchor_surviving Nodes whose anchor was pruned
dropped_arcs Step 2: reachable_from_root Edges to unreachable nodes
contraction_choices Step 3–4: contraction + resolution The intermediate path that was collapsed
original_parent Step 3: ancestor_contraction Original parent before re-parenting
pruned_fan_children Step 5: reconstruct_fans Fan children that were removed

The complement is the difference between the source and the view. It has a precise size: it is proportional to the amount of data that the migration discards. For a migration that only removes two leaf fields from a 10-field schema, the complement is roughly 20% of the source. For a migration that projects away 90% of the schema, the complement is large.

NoteThe complement is opaque

The complement should be treated as an opaque blob by application code. Its internal structure is an implementation detail of the pipeline. The only operation you should perform on a complement is passing it to put. Inspecting or modifying the complement voids the lens laws.

8.4 The put operation

Reconstruction is the pipeline in reverse. Given a (possibly modified) view and the complement, put rebuilds the source:

pub fn put(view: &WTree, complement: &Complement) -> WTree {
    let mut tree = view.clone();

    // Step 5 inverse: restore pruned fan children
    restore_fan_children(&mut tree, &complement.pruned_fan_children);

    // Step 4 inverse: no action needed (edge labels are in the complement)

    // Step 3 inverse: re-expand contracted paths
    expand_contractions(&mut tree, &complement.original_parent,
                        &complement.contraction_choices);

    // Step 2 inverse: re-attach unreachable nodes
    reattach_unreachable(&mut tree, &complement.dropped_arcs);

    // Step 1 inverse: restore pruned nodes
    restore_nodes(&mut tree, &complement.dropped_nodes);

    tree
}

The key property: if the view has been modified (e.g., a field value was changed), the modification is preserved in the reconstructed source. The complement only fills in the missing parts; it does not overwrite the view’s content. This is what makes the lens useful for bidirectional synchronization.

8.4.1 Example: round-trip with modification

Consider our threadViewPost example from Chapter 7. The migration removes likeCount and displayName.

  1. get produces a view without those fields, plus a complement recording their values (likeCount = 42, displayName = "Alice", etc.).
  2. An application modifies the view: it changes text from "Hello world" to "Hello, updated world".
  3. put takes the modified view and the complement, producing a source where text = "Hello, updated world" but likeCount = 42 and displayName = "Alice" are restored from the complement.

The round-trip preserves both the modification (new text) and the complement (old like count and display name).

NoteWhat if the view adds new fields?

The complement only records what was removed. If the view schema has fields that the source schema does not, put cannot place them—they have nowhere to go. This situation arises when the migration is not a simple projection but a more complex transformation. panproto handles it by requiring that put only accepts views whose schema is the migration’s target schema. Schema mismatches are caught at migration compilation time, not at runtime.

8.5 Cambria-style combinators

Building migrations by hand—specifying vertex maps, edge maps, and resolver tables—is precise but tedious. For common schema changes, panproto provides combinators: small, pre-built lenses that you compose into a pipeline.

The design follows Litt et al. (2022)’s Cambria system, adapted to panproto’s categorical framework. Each combinator is a well-behaved lens (satisfies GetPut and PutGet). Composition of well-behaved lenses is a well-behaved lens. So a pipeline of combinators automatically satisfies the lens laws.

const lens = pipeline([
  renameField('displayName', 'name'),
  addField('bio', 'string', ''),
  removeField('legacyField'),
]);

8.5.1 Available combinators

Combinator Forward (get) Backward (put)
renameField(old, new) Rename field in output Rename back
addField(name, default) Add field with default value Remove the field
removeField(name) Remove the field Restore from complement
wrapInObject(field, wrapper) Nest field inside a new object Unwrap
hoistField(path) Move a nested field up one level Move back down
coerceType(field, from, to) Convert value (e.g., string → int) Convert back

Each combinator handles its own complement. removeField stores the removed value in the complement. addField stores nothing (the default is known). renameField stores nothing (the mapping is invertible). wrapInObject stores the wrapper structure. And so on.

8.5.2 Composition

Combinators compose via pipeline():

const migration = pipeline(
  renameField("userName", "handle"),
  removeField("internalId"),
  addField("version", 2),
  hoistField("profile.displayName"),
);

The pipeline applies combinators left to right for get and right to left for put. The complement is a stack: each combinator pushes its complement fragment during get, and put pops them in reverse order.

This is function composition in the lens category. If \(\ell_1: S \rightleftarrows T\) and \(\ell_2: T \rightleftarrows V\), the composite \(\ell_2 \circ \ell_1: S \rightleftarrows V\) has:

\[\mathtt{get}_{2 \circ 1}(s) = \mathtt{get}_2(\pi_1(\mathtt{get}_1(s)))\] \[\mathtt{put}_{2 \circ 1}(v, (c_1, c_2)) = \mathtt{put}_1(\mathtt{put}_2(v, c_2), c_1)\]

The complement of the composite is the pair \((c_1, c_2)\)—the complements from each stage.

8.6 Worked example: get/put in action

Here is a complete example showing the combinator API in use:

import { Panproto, renameField, addField, removeField, pipeline } from '@panproto/core';

const panproto = await Panproto.init();
const atproto = panproto.protocol('atproto');

// start snippet combinators
const lens = pipeline([
  renameField('displayName', 'name'),
  addField('bio', 'string', ''),
  removeField('legacyField'),
]);
// end snippet combinators

// start snippet get-put
// Assume schemaV1, schemaV2, and migration are already built
const inputRecord = {
  displayName: 'Alice',
  legacyField: 'old-data',
};

// Forward: project to target schema, capturing the complement
const { view, complement } = migration.get(inputRecord);
// view: { name: 'Alice', bio: '' }
// complement: Uint8Array (opaque—tracks dropped fields and resolver choices)

// Backward: restore from modified view + complement
const modifiedView = { name: 'Alice (updated)', bio: 'Hello!' };
const restored = migration.put(modifiedView, complement);
// restored.data: original structure with modifications propagated back
// end snippet get-put

The critical observation: putResult.data.secretToken is "abc123" even though the view never contained it. The complement remembered it, and put restored it. Meanwhile, putResult.data.handle is "carol" (from the modified view), not "csmith" (from the original source). The lens respected the view modification.

This is the GetPut law in action: if we had not modified the view, put(get(source)) would have returned the original source exactly. And this is the PutGet law: getting from the put result returns a view with handle = "carol".

8.7 Complement serialization

The complement needs to be stored somewhere between get and put. panproto serializes it as CBOR (Concise Binary Object Representation), which is compact and fast to parse.

For a typical protocol evolution (removing a few fields from a 20-field schema), the serialized complement is a few hundred bytes per record. For bulk migrations, complements can be batched into a single CBOR array, compressed with zstd, and stored alongside the migrated data.

The storage strategy depends on the application:

  • Ephemeral complement: Store in memory during a request-response cycle. The complement lives only as long as the request.
  • Persistent complement: Store in a database column or object store. Enables deferred round-tripping: migrate data forward today, receive modifications over time, migrate back later.
  • Streaming complement: In a pipeline (Kafka, NATS), attach the complement as a message header. Downstream consumers can round-trip without access to the original source.
NoteComplement compression

Because the complement consists mostly of dropped field values—which are often repetitive (e.g., many records with likeCount = 0 or displayName = null)—it compresses extremely well. In benchmarks on ATProto data, zstd compression reduces complement size by 85–95%.

8.8 Symmetric lenses

Asymmetric lenses assume one schema is a “view” of the other—a strict sub-structure. This works well for protocol evolution within a single protocol family (ATProto v1 to v2), where the new schema is typically a refinement of the old.

But what about cross-protocol migration? When translating between ATProto and ActivityPub, neither schema is a sub-structure of the other. ATProto has likeCount but not published; ActivityPub has published but not likeCount. An asymmetric lens in either direction would lose data.

Symmetric lenses model this situation. A symmetric lens between \(S_1\) and \(S_2\) is a span of asymmetric lenses:

\[S_1 \leftarrow C \rightarrow S_2\]

The apex \(C\) is a common “core” schema that both \(S_1\) and \(S_2\) project to. The left leg \(\ell_1: C \rightleftarrows S_1\) migrates between the core and \(S_1\). The right leg \(\ell_2: C \rightleftarrows S_2\) migrates between the core and \(S_2\). To translate from \(S_1\) to \(S_2\):

  1. get along \(\ell_1\): project \(S_1\) data to the core \(C\), capturing complement \(c_1\).
  2. put along \(\ell_2\): embed the core data into \(S_2\), using a default complement \(c_2\).

The complements \(c_1\) and \(c_2\) record what each protocol has that the other lacks. If you later need to translate back from \(S_2\) to \(S_1\), you use \(c_1\) to restore the ATProto-specific fields and \(c_2\) to discard the ActivityPub-specific ones.

This is the construction of Hofmann et al. (2011), who showed that symmetric lenses form a category where the morphisms are equivalence classes of spans. The equivalence relation identifies spans that differ only in the choice of core schema, which corresponds to the intuition that the core is an implementation detail—what matters is the round-trip behavior.

8.8.1 The core schema

Finding the right core schema is a design problem, not an algorithmic one. For ATProto-to-ActivityPub translation, the core might include:

  • Post text (both protocols have it)
  • Author identifier (both have it, with different formats)
  • Timestamp (both have it)

But not:

  • Like count (ATProto only)
  • Content warning (ActivityPub only, via summary field)

panproto provides tooling to compute a maximal core (the categorical pullback of the two schemas along their shared structure), but the result is often too large or too small for practical use. In practice, the core is hand-curated by someone who understands both protocols.

NoteSymmetric lenses and information loss

A symmetric lens always loses information in both directions. Translating from ATProto to ActivityPub loses likeCount; translating back loses published. The complements track what was lost so that round-trips within a single direction are lossless. But a full cycle \(S_1 \to S_2 \to S_1\) is only lossless if the core contains everything in \(S_1\)—which means \(S_1\) is a sub-structure of the core, and we are back to the asymmetric case.

This is a fundamental limitation, not a panproto limitation. No system can translate between genuinely different schemas without losing something.

8.9 The categorical picture

For readers comfortable with category theory, here is the precise placement of panproto’s lenses in the literature.

An asymmetric lens \(\ell: S \rightleftarrows V\) is a morphism in the lens category \(\mathbf{Lens}\). Objects are schemas; morphisms are lens pairs \((get, put)\) satisfying GetPut and PutGet. Composition is sequential pipeline composition as described in Section 8.5. Identity is the trivial lens \((id, \pi_1)\).

This is the original definition of Foster et al. (2007), who introduced lenses for bidirectional programming in the context of the Harmony file synchronizer.

The delta lens generalization of Johnson and Rosebrugh (2013) replaces the concrete complement with an abstract notion of “update propagation.” A delta lens is a functor-cofunctor pair between categories of states and updates. This connects lenses to fibrations: the lens laws become the lifting conditions for a Grothendieck fibration.

Clarke (2020) unifies the picture by showing that internal lenses in a double category correspond to functors that are simultaneously cofunctors—the “functor-cofunctor” duality that delta lenses exhibit. panproto’s complement-based lenses are a concrete instance of this abstract framework, where the complement plays the role of the cartesian lifting.

The practical implication: panproto’s lens laws are not ad hoc conditions. They are instances of general categorical coherence conditions. Any construction that satisfies the categorical conditions automatically satisfies the lens laws. This is why combinator composition preserves well-behavedness: it is functorial composition in \(\mathbf{Lens}\).

NoteDo I need to know category theory?

No. The combinator API (Section 8.5) hides all of this. You compose combinators, and the lens laws hold by construction. The categorical framework is the reason the laws hold, but you do not need to understand the reason to use the result.

The category theory matters when you are extending panproto—adding new combinators, supporting new data formats, or proving properties of migrations. For day-to-day use, get, put, and pipeline are the entire API.

8.10 Lenses and the restrict pipeline

The connection between this chapter and Chapter 7 is worth making explicit.

The restrict pipeline from Chapter 7 is get without complement capture. It is the forward direction of the lens. panproto implements get by running the restrict pipeline and simultaneously building the complement at each step:

  1. Step 1 (anchor_surviving): Record each pruned node in dropped_nodes.
  2. Step 2 (reachable_from_root): Record arcs to unreachable nodes in dropped_arcs.
  3. Step 3 (ancestor_contraction): Record original parents in original_parent.
  4. Step 4 (resolve_edge): Record intermediate paths in contraction_choices.
  5. Step 5 (reconstruct_fans): Record pruned fan children in pruned_fan_children.

The complement is built during the forward pass, not after. This is important for performance: the restrict pipeline makes a single pass over the instance tree, and complement capture adds only constant overhead per node.

The put operation replays the steps in reverse, using the complement to undo each one. The result is a reconstructed source that agrees with the original on every field except those that were modified in the view.

8.11 Further reading

The foundational paper on asymmetric lenses for bidirectional programming is Foster et al. (2007). The treatment there is in terms of string lenses (regular-expression-based transformations on strings), but the abstract framework applies directly to tree and graph transformations.

Hofmann et al. (2011) introduced symmetric lenses and the span construction. Their proof that symmetric lenses form a category resolved a long-standing question about how to compose bidirectional transformations when neither direction is “primary.”

Johnson and Rosebrugh (2013) generalized both asymmetric and symmetric lenses to the delta lens framework, connecting lenses to fibrations and indexed categories. This is the deepest theoretical treatment and the one that most directly connects to panproto’s categorical foundations.

Clarke (2020) provides the most recent synthesis, unifying lenses, functors, and cofunctors in the framework of double categories. The paper is demanding but rewarding for readers with a strong category theory background.

Litt et al. (2022) is the practical inspiration for panproto’s combinator API. Cambria demonstrated that lens combinators are a viable approach to schema evolution in real systems (specifically, collaborative editing with CRDTs). panproto extends Cambria’s approach from flat JSON schemas to recursive, polynomial-functor schemas.

# Lenses: Going Both Ways {#sec-lenses}

Every migration in panproto is bidirectional. When you project data from a source schema to a view schema, you also get a receipt—the **complement**—that records everything the projection discarded. Given the complement and a (possibly modified) view, you can reconstruct the source. This round-trip property is not an implementation convenience; it is a mathematical guarantee provided by the theory of lenses.

This chapter introduces asymmetric lenses, shows how panproto implements them, and explains the combinator API that lets you build migrations from small, composable pieces.


## The get/put pattern {#sec-get-put}

An **asymmetric lens** $\ell: S \rightleftarrows V$ consists of two operations:

- **`get : S → (V, C)`** — Project the source $S$ to a view $V$, capturing the **complement** $C$ (everything discarded).
- **`put : (V, C) → S`** — Restore the source from a (possibly modified) view and the complement.

The operations satisfy two laws:

### GetPut (round-tripping)

$$\mathtt{put}(\mathtt{get}(s)) = s$$

Getting and then putting back (without modifying the view) recovers the original source exactly. No information is lost.

### PutGet (view consistency)

$$\pi_1(\mathtt{get}(\mathtt{put}(v, c))) = v$$

Putting a view $v$ into the source and then getting it back returns $v$. The view is faithfully represented in the reconstructed source.

Together, these laws say that `get` and `put` are mutual inverses up to the complement. The complement is the "slack" that allows the source to carry more information than the view without violating invertibility.

:::{.callout-note}
## Why "lens"?

The name comes from the idea of *focusing*: a lens focuses on a part of a larger structure (the view), while keeping the rest (the complement) available for reconstruction. The metaphor is apt—like an optical lens, a panproto lens lets you see a specific slice of the data while the surrounding context remains intact but out of focus.
:::


## Lenses in panproto

In panproto, `get` is the restrict pipeline from @sec-lifting plus complement capture. `put` is reconstruction from the complement. The connection is direct:

- **`get`** runs the five-step W-type restrict pipeline (or functor restrict for SQL) and simultaneously records what was removed at each step.
- **`put`** replays the removal in reverse: re-attach fans, re-resolve edges, re-expand contracted ancestors, re-attach unreachable nodes, and restore pruned anchors.

The lens laws hold because the complement records *exactly* what `get` discarded—no more, no less. `put` has enough information to invert each step, and the sequential structure of the pipeline means the inversions compose correctly.


## The complement {#sec-complement}

The complement is the central data structure of the lens. It is a receipt for what `get` threw away, organized by pipeline step.

```rust
pub struct Complement {
    /// Nodes pruned in step 1 (anchor not surviving).
    /// Maps node index to the full node (anchor, payload, children).
    pub dropped_nodes: HashMap<u32, Node>,

    /// Arcs (edges) that connected pruned nodes to surviving nodes.
    /// Each entry is (parent_idx, child_idx, edge_label).
    pub dropped_arcs: Vec<(u32, u32, Edge)>,

    /// Edge labels chosen by the resolver in step 4.
    /// Maps (new_parent, child) to the original intermediate path.
    pub contraction_choices: HashMap<(u32, u32), Vec<VertexId>>,

    /// Original parent of each contracted node (before re-parenting).
    pub original_parent: HashMap<u32, u32>,

    /// Fan children pruned in step 5 (hypergraph schemas only).
    pub pruned_fan_children: HashMap<FanId, Vec<u32>>,
}
```

Each field corresponds to a pipeline step:

| Complement field | Pipeline step | What it records |
|------------------|---------------|-----------------|
| `dropped_nodes` | Step 1: `anchor_surviving` | Nodes whose anchor was pruned |
| `dropped_arcs` | Step 2: `reachable_from_root` | Edges to unreachable nodes |
| `contraction_choices` | Step 3–4: contraction + resolution | The intermediate path that was collapsed |
| `original_parent` | Step 3: `ancestor_contraction` | Original parent before re-parenting |
| `pruned_fan_children` | Step 5: `reconstruct_fans` | Fan children that were removed |

The complement is the *difference* between the source and the view. It has a precise size: it is proportional to the amount of data that the migration discards. For a migration that only removes two leaf fields from a 10-field schema, the complement is roughly 20% of the source. For a migration that projects away 90% of the schema, the complement is large.

:::{.callout-note}
## The complement is opaque

The complement should be treated as an opaque blob by application code. Its internal structure is an implementation detail of the pipeline. The only operation you should perform on a complement is passing it to `put`. Inspecting or modifying the complement voids the lens laws.
:::


## The `put` operation

Reconstruction is the pipeline in reverse. Given a (possibly modified) view and the complement, `put` rebuilds the source:

```rust
pub fn put(view: &WTree, complement: &Complement) -> WTree {
    let mut tree = view.clone();

    // Step 5 inverse: restore pruned fan children
    restore_fan_children(&mut tree, &complement.pruned_fan_children);

    // Step 4 inverse: no action needed (edge labels are in the complement)

    // Step 3 inverse: re-expand contracted paths
    expand_contractions(&mut tree, &complement.original_parent,
                        &complement.contraction_choices);

    // Step 2 inverse: re-attach unreachable nodes
    reattach_unreachable(&mut tree, &complement.dropped_arcs);

    // Step 1 inverse: restore pruned nodes
    restore_nodes(&mut tree, &complement.dropped_nodes);

    tree
}
```

The key property: if the view has been modified (e.g., a field value was changed), the modification is preserved in the reconstructed source. The complement only fills in the *missing* parts; it does not overwrite the view's content. This is what makes the lens useful for bidirectional synchronization.

### Example: round-trip with modification

Consider our `threadViewPost` example from @sec-lifting. The migration removes `likeCount` and `displayName`.

1. **`get`** produces a view without those fields, plus a complement recording their values (`likeCount = 42`, `displayName = "Alice"`, etc.).
2. An application modifies the view: it changes `text` from `"Hello world"` to `"Hello, updated world"`.
3. **`put`** takes the modified view and the complement, producing a source where `text = "Hello, updated world"` but `likeCount = 42` and `displayName = "Alice"` are restored from the complement.

The round-trip preserves both the modification (new text) and the complement (old like count and display name).

:::{.callout-note}
## What if the view adds new fields?

The complement only records what was *removed*. If the view schema has fields that the source schema does not, `put` cannot place them—they have nowhere to go. This situation arises when the migration is not a simple projection but a more complex transformation. panproto handles it by requiring that `put` only accepts views whose schema is the migration's target schema. Schema mismatches are caught at migration compilation time, not at runtime.
:::


## Cambria-style combinators {#sec-combinators}

Building migrations by hand—specifying vertex maps, edge maps, and resolver tables—is precise but tedious. For common schema changes, panproto provides **combinators**: small, pre-built lenses that you compose into a pipeline.

The design follows @litt2022's Cambria system, adapted to panproto's categorical framework. Each combinator is a well-behaved lens (satisfies GetPut and PutGet). Composition of well-behaved lenses is a well-behaved lens. So a pipeline of combinators automatically satisfies the lens laws.

### Available combinators

| Combinator | Forward (`get`) | Backward (`put`) |
|------------|-----------------|-------------------|
| `renameField(old, new)` | Rename field in output | Rename back |
| `addField(name, default)` | Add field with default value | Remove the field |
| `removeField(name)` | Remove the field | Restore from complement |
| `wrapInObject(field, wrapper)` | Nest field inside a new object | Unwrap |
| `hoistField(path)` | Move a nested field up one level | Move back down |
| `coerceType(field, from, to)` | Convert value (e.g., string → int) | Convert back |

Each combinator handles its own complement. `removeField` stores the removed value in the complement. `addField` stores nothing (the default is known). `renameField` stores nothing (the mapping is invertible). `wrapInObject` stores the wrapper structure. And so on.

### Composition

Combinators compose via `pipeline()`:

```typescript
const migration = pipeline(
  renameField("userName", "handle"),
  removeField("internalId"),
  addField("version", 2),
  hoistField("profile.displayName"),
);
```

The pipeline applies combinators left to right for `get` and right to left for `put`. The complement is a stack: each combinator pushes its complement fragment during `get`, and `put` pops them in reverse order.

This is function composition in the lens category. If $\ell_1: S \rightleftarrows T$ and $\ell_2: T \rightleftarrows V$, the composite $\ell_2 \circ \ell_1: S \rightleftarrows V$ has:

$$\mathtt{get}_{2 \circ 1}(s) = \mathtt{get}_2(\pi_1(\mathtt{get}_1(s)))$$
$$\mathtt{put}_{2 \circ 1}(v, (c_1, c_2)) = \mathtt{put}_1(\mathtt{put}_2(v, c_2), c_1)$$

The complement of the composite is the pair $(c_1, c_2)$—the complements from each stage.

## Complement serialization

The complement needs to be stored somewhere between `get` and `put`. panproto serializes it as CBOR (Concise Binary Object Representation), which is compact and fast to parse.

For a typical protocol evolution (removing a few fields from a 20-field schema), the serialized complement is a few hundred bytes per record. For bulk migrations, complements can be batched into a single CBOR array, compressed with zstd, and stored alongside the migrated data.

The storage strategy depends on the application:

- **Ephemeral complement:** Store in memory during a request-response cycle. The complement lives only as long as the request.
- **Persistent complement:** Store in a database column or object store. Enables deferred round-tripping: migrate data forward today, receive modifications over time, migrate back later.
- **Streaming complement:** In a pipeline (Kafka, NATS), attach the complement as a message header. Downstream consumers can round-trip without access to the original source.

:::{.callout-note}
## Complement compression

Because the complement consists mostly of dropped field values—which are often repetitive (e.g., many records with `likeCount = 0` or `displayName = null`)—it compresses extremely well. In benchmarks on ATProto data, zstd compression reduces complement size by 85–95%.
:::


## Symmetric lenses {#sec-symmetric-lenses}

Asymmetric lenses assume one schema is a "view" of the other—a strict sub-structure. This works well for protocol evolution within a single protocol family (ATProto v1 to v2), where the new schema is typically a refinement of the old.

But what about cross-protocol migration? When translating between ATProto and ActivityPub, neither schema is a sub-structure of the other. ATProto has `likeCount` but not `published`; ActivityPub has `published` but not `likeCount`. An asymmetric lens in either direction would lose data.

**Symmetric lenses** model this situation. A symmetric lens between $S_1$ and $S_2$ is a **span** of asymmetric lenses:

$$S_1 \leftarrow C \rightarrow S_2$$

The apex $C$ is a common "core" schema that both $S_1$ and $S_2$ project to. The left leg $\ell_1: C \rightleftarrows S_1$ migrates between the core and $S_1$. The right leg $\ell_2: C \rightleftarrows S_2$ migrates between the core and $S_2$. To translate from $S_1$ to $S_2$:

1. `get` along $\ell_1$: project $S_1$ data to the core $C$, capturing complement $c_1$.
2. `put` along $\ell_2$: embed the core data into $S_2$, using a default complement $c_2$.

The complements $c_1$ and $c_2$ record what each protocol has that the other lacks. If you later need to translate back from $S_2$ to $S_1$, you use $c_1$ to restore the ATProto-specific fields and $c_2$ to discard the ActivityPub-specific ones.

This is the construction of @hofmann2011, who showed that symmetric lenses form a category where the morphisms are equivalence classes of spans. The equivalence relation identifies spans that differ only in the choice of core schema, which corresponds to the intuition that the core is an implementation detail—what matters is the round-trip behavior.

### The core schema

Finding the right core schema is a design problem, not an algorithmic one. For ATProto-to-ActivityPub translation, the core might include:

- Post text (both protocols have it)
- Author identifier (both have it, with different formats)
- Timestamp (both have it)

But not:

- Like count (ATProto only)
- Content warning (ActivityPub only, via `summary` field)

panproto provides tooling to *compute* a maximal core (the categorical pullback of the two schemas along their shared structure), but the result is often too large or too small for practical use. In practice, the core is hand-curated by someone who understands both protocols.

:::{.callout-note}
## Symmetric lenses and information loss

A symmetric lens always loses information in both directions. Translating from ATProto to ActivityPub loses `likeCount`; translating back loses `published`. The complements track what was lost so that round-trips within a single direction are lossless. But a full cycle $S_1 \to S_2 \to S_1$ is only lossless if the core contains everything in $S_1$—which means $S_1$ is a sub-structure of the core, and we are back to the asymmetric case.

This is a fundamental limitation, not a panproto limitation. No system can translate between genuinely different schemas without losing something.
:::


## The categorical picture {#sec-lens-category}

For readers comfortable with category theory, here is the precise placement of panproto's lenses in the literature.

An asymmetric lens $\ell: S \rightleftarrows V$ is a morphism in the **lens category** $\mathbf{Lens}$. Objects are schemas; morphisms are lens pairs $(get, put)$ satisfying GetPut and PutGet. Composition is sequential pipeline composition as described in @sec-combinators. Identity is the trivial lens $(id, \pi_1)$.

This is the original definition of @foster2007, who introduced lenses for bidirectional programming in the context of the Harmony file synchronizer.

The **delta lens** generalization of @johnson2013 replaces the concrete complement with an abstract notion of "update propagation." A delta lens is a functor-cofunctor pair between categories of states and updates. This connects lenses to fibrations: the lens laws become the lifting conditions for a Grothendieck fibration.

@clarke2020 unifies the picture by showing that internal lenses in a double category correspond to functors that are simultaneously cofunctors—the "functor-cofunctor" duality that delta lenses exhibit. panproto's complement-based lenses are a concrete instance of this abstract framework, where the complement plays the role of the cartesian lifting.

The practical implication: panproto's lens laws are not ad hoc conditions. They are instances of general categorical coherence conditions. Any construction that satisfies the categorical conditions automatically satisfies the lens laws. This is why combinator composition preserves well-behavedness: it is functorial composition in $\mathbf{Lens}$.

:::{.callout-note}
## Do I need to know category theory?

No. The combinator API (@sec-combinators) hides all of this. You compose combinators, and the lens laws hold by construction. The categorical framework is the *reason* the laws hold, but you do not need to understand the reason to use the result.

The category theory matters when you are *extending* panproto—adding new combinators, supporting new data formats, or proving properties of migrations. For day-to-day use, `get`, `put`, and `pipeline` are the entire API.
:::


## Lenses and the restrict pipeline

The connection between this chapter and @sec-lifting is worth making explicit.

The restrict pipeline from @sec-lifting is `get` without complement capture. It is the *forward* direction of the lens. panproto implements `get` by running the restrict pipeline and simultaneously building the complement at each step:

1. **Step 1** (`anchor_surviving`): Record each pruned node in `dropped_nodes`.
2. **Step 2** (`reachable_from_root`): Record arcs to unreachable nodes in `dropped_arcs`.
3. **Step 3** (`ancestor_contraction`): Record original parents in `original_parent`.
4. **Step 4** (`resolve_edge`): Record intermediate paths in `contraction_choices`.
5. **Step 5** (`reconstruct_fans`): Record pruned fan children in `pruned_fan_children`.

The complement is built *during* the forward pass, not *after*. This is important for performance: the restrict pipeline makes a single pass over the instance tree, and complement capture adds only constant overhead per node.

The `put` operation replays the steps in reverse, using the complement to undo each one. The result is a reconstructed source that agrees with the original on every field except those that were modified in the view.


## Further reading

The foundational paper on asymmetric lenses for bidirectional programming is @foster2007. The treatment there is in terms of string lenses (regular-expression-based transformations on strings), but the abstract framework applies directly to tree and graph transformations.

@hofmann2011 introduced symmetric lenses and the span construction. Their proof that symmetric lenses form a category resolved a long-standing question about how to compose bidirectional transformations when neither direction is "primary."

@johnson2013 generalized both asymmetric and symmetric lenses to the delta lens framework, connecting lenses to fibrations and indexed categories. This is the deepest theoretical treatment and the one that most directly connects to panproto's categorical foundations.

@clarke2020 provides the most recent synthesis, unifying lenses, functors, and cofunctors in the framework of double categories. The paper is demanding but rewarding for readers with a strong category theory background.

@litt2022 is the practical inspiration for panproto's combinator API. Cambria demonstrated that lens combinators are a viable approach to schema evolution in real systems (specifically, collaborative editing with CRDTs). panproto extends Cambria's approach from flat JSON schemas to recursive, polynomial-functor schemas.

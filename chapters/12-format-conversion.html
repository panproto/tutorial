12  Format Conversion

13 Format Conversion

There is a program called pandoc that converts between document formats. You hand it a Markdown file and it gives you a PDF, an HTML page, a Word document, or a LaTeX source file. The conversion is not lossless—some Markdown constructs have no meaningful representation in Word, and some Word features have no Markdown equivalent—but for ordinary prose, pandoc gets you close enough that the conversion is practically useful.

panproto is pandoc for schemas.

Every protocol in panproto ships with both a parse_* function that reads a native schema document into the universal Schema graph and an emit_* function that writes a Schema graph back out to a native schema document. These two functions together enable a format conversion pipeline: parse from format A, obtain a universal intermediate representation, optionally migrate or transform that representation, and emit to format B.

This chapter walks through what that pipeline looks like in practice, what it can and cannot faithfully carry across format boundaries, and why some cross-format conversions are structurally more faithful than others.

13.1 A Motivating Example: Avro to Protobuf

Suppose you have an existing Apache Avro schema for an event record—a common starting point for teams migrating off a Kafka-centric data platform toward gRPC.

{
  "type": "record",
  "name": "UserSignedUp",
  "namespace": "com.example.events",
  "fields": [
    { "name": "userId",    "type": "string" },
    { "name": "email",     "type": "string" },
    { "name": "createdAt", "type": "long",   "logicalType": "timestamp-millis" },
    { "name": "referrer",  "type": ["null", "string"], "default": null }
  ]
}

The manual conversion to Protobuf is tedious but straightforward for simple schemas. For large schemas with dozens of nested record types, union types, and cross-file references, it becomes error-prone. Field numbers must be assigned. Union types (["null", "string"]) must be mapped to optional or oneof constructs. Logical types (timestamp-millis) must be mapped to well-known Protobuf types (google.protobuf.Timestamp).

With panproto, the conversion is a pipeline:

import { parse_avsc, emit_proto, AVRO_SPEC, PROTOBUF_SPEC } from "@panproto/core";

// Parse the Avro schema document into panproto's universal Schema graph.
const schema = parse_avsc(avroSource, AVRO_SPEC);

// Emit the Schema graph as a Protobuf .proto file.
const protoSource = emit_proto(schema, PROTOBUF_SPEC);

The parse_avsc call reads the Avro JSON and constructs a Schema graph: vertices for UserSignedUp, string, long, and the union type; edges for each field; constraint annotations for the logical type and the nullability of referrer. The emit_proto call walks that Schema graph and produces equivalent Protobuf IDL, assigning field numbers, mapping the timestamp logical type to google.protobuf.Timestamp, and translating the ["null", "string"] union into an optional string.

This is the pandoc analogy for schemas. pandoc’s intermediate representation is a document abstract syntax tree. panproto’s intermediate representation is the Schema graph—a multigraph or hypergraph decorated with protocol-independent constraint annotations.

13.2 The Parse → Schema → Migrate → Emit Pipeline

The general pipeline has four stages:

Parse. The parse_* function for a protocol reads its native schema notation and constructs a Schema graph. The parse step is responsible for understanding protocol-specific syntax and encoding it into the protocol-neutral graph representation. What cannot be represented in the graph is dropped or converted to a best-effort annotation.

Schema graph. The universal intermediate representation is the same Schema graph that panproto uses for everything else—diffs, migrations, lens derivation, breaking-change detection. It is not a separate format invented for conversion; it is the standard representation, which is why the rest of the panproto machinery can be applied to it between the parse and emit steps.

Migrate / transform. This stage is optional. You can apply a panproto migration to the Schema graph between parse and emit—for example, to rename a field, flatten a nested type, or restructure a hierarchy. This is the stage where the cross-format conversion acquires the full power of panproto’s migration engine. Without this stage, the pipeline is a direct format translation; with it, the pipeline can simultaneously translate and transform.

Emit. The emit_* function for a protocol walks the Schema graph and produces native schema notation. The emit step is responsible for mapping the protocol-neutral representation back to the target format’s specific constructs—assigning field numbers, choosing appropriate type names, generating the correct syntax.

Every one of panproto’s 54 built-in protocols has both a parse and an emit function. This means every pair of protocols is connected by a potential conversion path through the shared Schema graph.

13.3 Cross-Protocol Examples

The following examples illustrate the pipeline for four common conversion scenarios. The code is conceptual—it shows the API shape and the conversion intent, not all of the configuration details that a production invocation would require.

13.3.1 JSON Schema to TypeScript Interfaces

TypeScript interfaces are the natural consumer of JSON Schema definitions in a frontend or Node.js codebase. The conversion maps JSON Schema type: "object" to a TypeScript interface, properties to typed fields, and constraint keywords to comments (since TypeScript’s type system cannot express maxLength or pattern directly).

import { parse_json_schema, emit_typescript } from "@panproto/core";

const schema = parse_json_schema(jsonSchemaSource, JSON_SCHEMA_SPEC);
const tsSource = emit_typescript(schema, TYPESCRIPT_SPEC);

// Input: JSON Schema object with "properties", "required", "maxLength", etc.
// Output: TypeScript interface declarations.
//
// interface Post {
//   text: string;       // maxLength: 3000
//   createdAt: string;  // format: date-time
//   tags?: string[];    // maxItems: 8
// }
//
// Constraint annotations (maxLength, maxItems) become comments.
// TypeScript's type system has no way to enforce them structurally.

The ? on tags comes from the fact that it was not in the JSON Schema required array. The comment annotations are the emit function’s way of preserving constraint information that has no structural equivalent in the target format—a best-effort record of what was lost.

13.3.2 SQL DDL to Parquet Schema

Parquet is a columnar storage format used in data lakes. Its schema language describes flat records with named, typed columns—a structure that maps naturally from SQL table definitions. The conversion from SQL DDL to Parquet schema is particularly useful when exporting a relational database’s schema to an analytics pipeline.

import { parse_sql, emit_parquet } from "@panproto/core";

const schema = parse_sql(ddlSource, SQL_SPEC);
const parquetSchema = emit_parquet(schema, PARQUET_SPEC);

// Input: SQL DDL with CREATE TABLE statements, column types, NOT NULL constraints.
// Output: Parquet schema with equivalent message/field structure.
//
// CREATE TABLE orders (
//   id         BIGINT NOT NULL,
//   customer   VARCHAR(255) NOT NULL,
//   amount     DECIMAL(10,2),
//   created_at TIMESTAMPTZ NOT NULL
// );
//
// →
//
// message orders {
//   required int64  id;
//   required binary customer (UTF8);
//   optional fixed_len_byte_array(16) amount (DECIMAL(10,2));
//   required int96  created_at;
// }
//
// Foreign keys are dropped: Parquet has no hyperedge structure.

The critical loss in this conversion is the foreign key structure. SQL schemas are hypergraphs; Parquet schemas are simple flat records. The parse step captures the foreign keys as hyperedges in the Schema graph, but the emit step cannot represent them in Parquet—they are dropped with a warning. The column-level data (names, types, nullability) survives faithfully.

13.3.3 GraphQL SDL to OpenAPI Specification

GraphQL types map naturally to OpenAPI schema objects. The conversion is useful when you want to expose a GraphQL-backed service through a REST API that clients can consume without a GraphQL client library.

import { parse_graphql, emit_openapi } from "@panproto/core";

const schema = parse_graphql(sdlSource, GRAPHQL_SPEC);
const openApiDoc = emit_openapi(schema, OPENAPI_SPEC);

// Input: GraphQL SDL with type definitions, interface implementations, unions.
// Output: OpenAPI 3.1 spec with equivalent schema components.
//
// type User implements Node {
//   id: ID!
//   name: String!
//   email: String
//   posts: [Post!]!
// }
//
// →
//
// components:
//   schemas:
//     User:
//       type: object
//       required: [id, name, posts]
//       properties:
//         id:   { type: string }
//         name: { type: string }
//         email: { type: string, nullable: true }
//         posts:
//           type: array
//           items: { $ref: '#/components/schemas/Post' }
//
// The `Node` interface is represented via allOf composition.
// GraphQL union types become OpenAPI oneOf.
// Resolver logic and directives are not representable and are dropped.

GraphQL’s interface system—the implements Node clause—survives conversion because both GraphQL SDL and OpenAPI belong to the same theory group (see Section 13.5 below). The abstract type relationships are expressed in OpenAPI using allOf composition.

13.3.4 Protobuf to FlatBuffers

Protobuf and FlatBuffers are both binary serialization formats with schema-defined message types and typed fields. They occupy very similar structural territory, and the conversion between them is correspondingly high-fidelity.

import { parse_proto, emit_fbs } from "@panproto/core";

const schema = parse_proto(protoSource, PROTOBUF_SPEC);
const fbsSource = emit_fbs(schema, FLATBUFFERS_SPEC);

// Input: .proto file with message and enum definitions, field numbers.
// Output: .fbs FlatBuffers schema with equivalent table and enum definitions.
//
// message UserProfile {
//   string  username    = 1;
//   int32   follower_count = 2;
//   repeated string interests = 3;
// }
//
// →
//
// table UserProfile {
//   username:       string;
//   follower_count: int32;
//   interests:      [string];
// }
//
// Field numbers are dropped: FlatBuffers uses field order and vtable offsets.
// Wire encoding differences are handled automatically.
// Protobuf services (RPC definitions) have no FlatBuffers equivalent and are dropped.

The structural content—message name, field names, field types, repeated markers—survives intact. What is lost is the Protobuf-specific wire metadata: field numbers are dropped because FlatBuffers does not use them (it uses vtable offsets instead). Service definitions are also dropped. But the schema structure—the multigraph of messages and fields—converts faithfully.

13.4 Round-Trip Fidelity

Format conversion is not lossless in general. What survives a round trip—parse from format A, emit to format B, parse from format B, emit back to format A—depends on how much structural territory the two formats share.

There are two sources of information loss:

Parse loss. When parse_* reads a source document, it must represent the source format’s constructs in the universal Schema graph. Constructs that have no graph-level equivalent are dropped. SQL foreign keys that reference three tables simultaneously (a hyperedge) can be represented in the Schema graph, but a CHECK constraint containing an arbitrary SQL expression may be captured only as an opaque annotation—its semantics are not machine-processable at the schema level.

Emit loss. When emit_* writes a target document, it must represent the Schema graph’s constructs in the target format’s notation. Constructs in the Schema graph that have no target-format equivalent are dropped or approximated. Hyperedges are dropped when emitting to a format with no hyperedge concept. Constraint keywords are dropped when emitting to a format with no constraint vocabulary.

A useful heuristic is to ask whether the source and target formats are structurally comparable:

Conversion What survives What is typically lost
JSON Schema → TypeScript Field names, types, optionality Constraint keywords (maxLength, pattern)
SQL DDL → Parquet Column names, types, nullability Foreign keys, CHECK constraints, defaults
GraphQL SDL → OpenAPI Type names, field names, types, interfaces Directives, resolver-level semantics
Protobuf → FlatBuffers Message names, field names, types Field numbers, service definitions
Avro → Protobuf Record names, field names, types Logical types (approximated), union semantics

The middle column—what survives—corresponds to the structural skeleton of the schema: the graph of types and relationships. This is what the Schema graph represents faithfully. The right column—what is lost—corresponds to format-specific decorations and semantic extensions that lie outside the schema graph’s representational scope.

13.5 The Theory Group Advantage

The heuristic of “structural comparability” has a precise underpinning in panproto’s architecture. As established in ?sec-protocols, every protocol is characterized by a pair of theories: a schema theory and an instance theory. Protocols whose schema theories are closely related will have higher-fidelity conversions.

panproto’s built-in protocols cluster into five theory groups based on their schema and instance theories:

The table is simplified—actual protocols add additional component theories for protocol-specific constructs such as field numbering (Protobuf), grapheme limits (ATProto), or service definitions (gRPC). But the dominant theory combination, which determines the structural shape of schemas and the shape of data, follows this grouping.

Within-group conversion is structurally lossless at the schema level. Two Group C protocols—say, Avro and Protobuf—share the same schema theory: colimit(ThGraph, ThConstraint, ThMulti). A Schema graph that is a valid Avro schema is, structurally, also a valid Protobuf schema. The conversion between them reduces to remapping protocol-specific surface syntax: Avro’s "type": "record" becomes Protobuf’s message, Avro’s "type": ["null", "string"] becomes Protobuf’s optional string. The graph structure itself does not change.

Cross-group conversion involves a structural mismatch. Converting from Group B (SQL, hypergraph + functor) to Group C (Protobuf, multigraph + W-type) loses hyperedge structure: foreign keys, composite unique constraints, and multi-column primary keys cannot be expressed in Group C’s multigraph schema theory. Converting from Group C to Group B in the other direction loses tree-shaped instance semantics: SQL rows are flat, and the nested document structure of a Group C instance does not survive the trip to a relational table without denormalization decisions that the engine cannot make automatically.

The diagram shows that cross-group conversions always carry some structural loss. Within-group conversions—the solid double-headed arrows—are structurally lossless: the schema graph is preserved intact, and only surface syntax changes.

13.5.1 A Note on “Lossless”

“Structurally lossless at the schema level” means the graph structure of the schema survives the round trip. It does not mean the round trip produces identical source bytes, or that all protocol-specific metadata is preserved. Protobuf field numbers, for example, are not part of the multigraph structure—they are a surface feature of the wire format. A Protobuf-to-Avro-to-Protobuf round trip will not preserve field numbers; the emit step will assign fresh ones. The schema graph—the message names, field names, types, and relationships—will be intact, but the wire-format metadata will not.

If your conversion requirements include preserving wire-format metadata, you can carry it as an opaque annotation through the Schema graph. panproto’s Schema graph supports arbitrary key-value metadata on vertices and edges; the parse functions use this to preserve information that is meaningful to the source format but has no universal semantic. The emit function for the target format can then ignore this metadata, or a custom emit wrapper can read it and apply it selectively.

13.6 Comparison with pandoc

The pandoc analogy that opened this chapter is worth examining in more detail, because the structural parallel is exact in some places and breaks down in others.

What is the same. Both pandoc and panproto convert between multiple formats using a shared intermediate representation. pandoc’s IR is a document abstract syntax tree: a tree of Pandoc elements (paragraphs, headers, code blocks, spans, and so on). panproto’s IR is the Schema graph: a multigraph or hypergraph with constraint annotations. In both systems, the IR is the canonical representation, and all conversion goes through it. Neither system converts directly between pairs of formats; all paths go through the IR.

Both systems also face the same fundamental limitation: information that cannot be represented in the IR is lost. pandoc cannot round-trip Word documents with custom styles, tracked changes, or embedded macros, because these features have no equivalent in the Pandoc AST. panproto cannot round-trip SQL schemas with arbitrary CHECK expressions, because these expressions are opaque at the schema graph level.

What is different. pandoc’s IR is a fixed, hand-designed document AST. Adding a new element type to the AST requires changing pandoc’s source code. panproto’s IR is not fixed—it is the Schema graph, whose structure is determined by the protocol theories. Adding a new structural concept (a new sort of vertex or edge) means extending a theory via a colimit, not patching the IR definition.

This difference matters for extensibility. When a new schema format appears—say, a domain-specific schema language for a new database engine—panproto can accommodate it by defining a new protocol theory. The Schema graph automatically gains the new format’s structural concepts as a natural extension of the theory. pandoc, by contrast, would require an AST extension to handle document constructs with no existing element type.

The second difference is in the use of the IR. In pandoc, the IR exists solely for conversion; you interact with it by writing filters that transform the AST. In panproto, the Schema graph is the primary representation for all operations: diff, migration, existence checking, lens derivation, breaking-change detection, and conversion all operate on the same Schema graph. Format conversion is not a separate subsystem—it is the composition of parse_* and emit_* applied to the same object that the rest of the toolchain already uses.

13.7 Summary

  • panproto is to schemas what pandoc is to documents: it converts between formats using a universal intermediate representation, the Schema graph.

  • Every protocol ships with a parse_* function (source format → Schema graph) and an emit_* function (Schema graph → target format). All 54 built-in protocols support both directions.

  • The conversion pipeline is: parse_*() → Schema graph → (optional migration) → emit_*(). The optional migration step allows simultaneous format conversion and structural transformation.

  • Round-trip fidelity depends on what the source and target formats share structurally. Information that fits in the Schema graph survives; information that does not—hyperedges in a format without hyperedge support, constraint keywords in a format without a constraint vocabulary—is dropped.

  • Theory groups make fidelity precise. Protocols sharing the same schema and instance theory (same group) convert with structural losslessness. Cross-group conversions involve structural mismatches: Group B (SQL, hypergraph) to Group C (Protobuf, multigraph) loses hyperedge structure; Group C to Group E (Parquet, functor instances) faces an instance-theory mismatch.

  • panproto’s IR is not a fixed data structure but a schema graph whose shape is determined by extensible theories. This distinguishes it from pandoc’s fixed document AST: new schema formats extend the theory space, and the IR adapts accordingly.

  • Format conversion in panproto is not a separate subsystem. It is the composition of two operations—parse and emit—applied to the same Schema graph that the rest of the toolchain already uses for diffs, migrations, and lens derivation.

# Format Conversion {#sec-format-conversion}

There is a program called pandoc that converts between document formats. You hand it a Markdown file and it gives you a PDF, an HTML page, a Word document, or a LaTeX source file. The conversion is not lossless—some Markdown constructs have no meaningful representation in Word, and some Word features have no Markdown equivalent—but for ordinary prose, pandoc gets you close enough that the conversion is practically useful.

panproto is pandoc for schemas.

Every protocol in panproto ships with both a `parse_*` function that reads a native schema document into the universal Schema graph and an `emit_*` function that writes a Schema graph back out to a native schema document. These two functions together enable a **format conversion pipeline**: parse from format A, obtain a universal intermediate representation, optionally migrate or transform that representation, and emit to format B.

This chapter walks through what that pipeline looks like in practice, what it can and cannot faithfully carry across format boundaries, and why some cross-format conversions are structurally more faithful than others.


## A Motivating Example: Avro to Protobuf

Suppose you have an existing Apache Avro schema for an event record—a common starting point for teams migrating off a Kafka-centric data platform toward gRPC.

```avsc
{
  "type": "record",
  "name": "UserSignedUp",
  "namespace": "com.example.events",
  "fields": [
    { "name": "userId",    "type": "string" },
    { "name": "email",     "type": "string" },
    { "name": "createdAt", "type": "long",   "logicalType": "timestamp-millis" },
    { "name": "referrer",  "type": ["null", "string"], "default": null }
  ]
}
```

The manual conversion to Protobuf is tedious but straightforward for simple schemas. For large schemas with dozens of nested record types, union types, and cross-file references, it becomes error-prone. Field numbers must be assigned. Union types (`["null", "string"]`) must be mapped to `optional` or `oneof` constructs. Logical types (`timestamp-millis`) must be mapped to well-known Protobuf types (`google.protobuf.Timestamp`).

With panproto, the conversion is a pipeline:

```typescript
import { parse_avsc, emit_proto, AVRO_SPEC, PROTOBUF_SPEC } from "@panproto/core";

// Parse the Avro schema document into panproto's universal Schema graph.
const schema = parse_avsc(avroSource, AVRO_SPEC);

// Emit the Schema graph as a Protobuf .proto file.
const protoSource = emit_proto(schema, PROTOBUF_SPEC);
```

The `parse_avsc` call reads the Avro JSON and constructs a Schema graph: vertices for `UserSignedUp`, `string`, `long`, and the union type; edges for each field; constraint annotations for the logical type and the nullability of `referrer`. The `emit_proto` call walks that Schema graph and produces equivalent Protobuf IDL, assigning field numbers, mapping the timestamp logical type to `google.protobuf.Timestamp`, and translating the `["null", "string"]` union into an `optional string`.

This is the pandoc analogy for schemas. pandoc's intermediate representation is a document abstract syntax tree. panproto's intermediate representation is the Schema graph—a multigraph or hypergraph decorated with protocol-independent constraint annotations.


## The Parse → Schema → Migrate → Emit Pipeline

The general pipeline has four stages:

```{mermaid}
%%| label: fig-conversion-pipeline
%%| fig-cap: "The format conversion pipeline. Every protocol has both a parse and an emit function. The Schema graph is the universal intermediate representation."
graph LR
    A["Source document<br/>(format A)"]
    B["Schema graph<br/>(universal IR)"]
    C["Schema graph<br/>(optionally transformed)"]
    D["Target document<br/>(format B)"]

    A -->|"parse_*()"| B
    B -->|"migrate / transform"| C
    C -->|"emit_*()"| D
```

**Parse.** The `parse_*` function for a protocol reads its native schema notation and constructs a Schema graph. The parse step is responsible for understanding protocol-specific syntax and encoding it into the protocol-neutral graph representation. What cannot be represented in the graph is dropped or converted to a best-effort annotation.

**Schema graph.** The universal intermediate representation is the same Schema graph that panproto uses for everything else—diffs, migrations, lens derivation, breaking-change detection. It is not a separate format invented for conversion; it is the standard representation, which is why the rest of the panproto machinery can be applied to it between the parse and emit steps.

**Migrate / transform.** This stage is optional. You can apply a panproto migration to the Schema graph between parse and emit—for example, to rename a field, flatten a nested type, or restructure a hierarchy. This is the stage where the cross-format conversion acquires the full power of panproto's migration engine. Without this stage, the pipeline is a direct format translation; with it, the pipeline can simultaneously translate and transform.

**Emit.** The `emit_*` function for a protocol walks the Schema graph and produces native schema notation. The emit step is responsible for mapping the protocol-neutral representation back to the target format's specific constructs—assigning field numbers, choosing appropriate type names, generating the correct syntax.

Every one of panproto's 54 built-in protocols has both a parse and an emit function. This means every pair of protocols is connected by a potential conversion path through the shared Schema graph.

### JSON Schema to TypeScript Interfaces

TypeScript interfaces are the natural consumer of JSON Schema definitions in a frontend or Node.js codebase.
```typescript
import { parse_json_schema, emit_typescript } from "@panproto/core";

const schema = parse_json_schema(jsonSchemaSource, JSON_SCHEMA_SPEC);

// Input: JSON Schema object with "properties", "required", "maxLength", etc.
// Output: TypeScript interface declarations.
//
// interface Post {
//   text: string;       // maxLength: 3000
//   createdAt: string;  // format: date-time

// }
//
// Constraint annotations (maxLength, maxItems) become comments.
// TypeScript's type system has no way to enforce them structurally.
```

The `?` on `tags` comes from the fact that it was not in the JSON Schema `required` array. The comment annotations are the emit function's way of preserving constraint information that has no structural equivalent in the target format—a best-effort record of what was lost.

Protobuf and FlatBuffers are both binary serialization formats with schema-defined message types and typed fields. They occupy very similar structural territory, and the conversion between them is correspondingly high-fidelity.

```typescript
import { parse_proto, emit_fbs } from "@panproto/core";

const schema = parse_proto(protoSource, PROTOBUF_SPEC);
const fbsSource = emit_fbs(schema, FLATBUFFERS_SPEC);

// Output: .fbs FlatBuffers schema with equivalent table and enum definitions.

// message UserProfile {
//   string  username    = 1;
//   int32   follower_count = 2;
//   repeated string interests = 3;
// }
//
// →
//
// table UserProfile {
//   username:       string;
//   follower_count: int32;
//   interests:      [string];
// }
//
// Field numbers are dropped: FlatBuffers uses field order and vtable offsets.
// Wire encoding differences are handled automatically.
// Protobuf services (RPC definitions) have no FlatBuffers equivalent and are dropped.
```

The structural content—message name, field names, field types, repeated markers—survives intact. What is lost is the Protobuf-specific wire metadata: field numbers are dropped because FlatBuffers does not use them (it uses vtable offsets instead). Service definitions are also dropped. But the schema structure—the multigraph of messages and fields—converts faithfully.


## Round-Trip Fidelity

Format conversion is not lossless in general. What survives a round trip—parse from format A, emit to format B, parse from format B, emit back to format A—depends on how much structural territory the two formats share.

There are two sources of information loss:

**Parse loss.** When `parse_*` reads a source document, it must represent the source format's constructs in the universal Schema graph. Constructs that have no graph-level equivalent are dropped. SQL foreign keys that reference three tables simultaneously (a hyperedge) can be represented in the Schema graph, but a `CHECK` constraint containing an arbitrary SQL expression may be captured only as an opaque annotation—its semantics are not machine-processable at the schema level.

**Emit loss.** When `emit_*` writes a target document, it must represent the Schema graph's constructs in the target format's notation. Constructs in the Schema graph that have no target-format equivalent are dropped or approximated. Hyperedges are dropped when emitting to a format with no hyperedge concept. Constraint keywords are dropped when emitting to a format with no constraint vocabulary.

A useful heuristic is to ask whether the source and target formats are **structurally comparable**:

| Conversion | What survives | What is typically lost |
|---|---|---|
| JSON Schema → TypeScript | Field names, types, optionality | Constraint keywords (`maxLength`, `pattern`) |
| SQL DDL → Parquet | Column names, types, nullability | Foreign keys, `CHECK` constraints, defaults |
| GraphQL SDL → OpenAPI | Type names, field names, types, interfaces | Directives, resolver-level semantics |
| Protobuf → FlatBuffers | Message names, field names, types | Field numbers, service definitions |
| Avro → Protobuf | Record names, field names, types | Logical types (approximated), union semantics |

The middle column—what survives—corresponds to the structural skeleton of the schema: the graph of types and relationships. This is what the Schema graph represents faithfully. The right column—what is lost—corresponds to format-specific decorations and semantic extensions that lie outside the schema graph's representational scope.


## The Theory Group Advantage {#sec-theory-groups}

The heuristic of "structural comparability" has a precise underpinning in panproto's architecture. As established in @sec-protocols, every protocol is characterized by a pair of theories: a schema theory and an instance theory. Protocols whose schema theories are closely related will have higher-fidelity conversions.

panproto's built-in protocols cluster into five **theory groups** based on their schema and instance theories:

| Group | Schema theory family | Instance theory | Representative protocols |
|-------|---------------------|-----------------|--------------------------|
| **A** | ThCategory | ThFunctor | CQL |
| **B** | ThHypergraph + ThConstraint | ThFunctor | SQL DDL, Cassandra, DynamoDB |
| **C** | ThGraph + ThConstraint + ThMulti | ThWType | JSON Schema, ATProto, Avro, Protobuf, Thrift, MsgPack, Cap'n Proto, FlatBuffers |
| **D** | ThGraph + ThConstraint + ThMulti + ThInterface | ThWType | GraphQL SDL, OpenAPI, AsyncAPI |
| **E** | ThGraph + ThConstraint + ThMulti | ThFunctor | Parquet, Arrow, DataFrame |

: Theory groups for selected protocols. Protocols in the same group share the same schema and instance theories, enabling structurally lossless conversion. {#tbl-theory-groups}

The table is simplified—actual protocols add additional component theories for protocol-specific constructs such as field numbering (Protobuf), grapheme limits (ATProto), or service definitions (gRPC). But the dominant theory combination, which determines the structural shape of schemas and the shape of data, follows this grouping.

**Within-group conversion** is structurally lossless at the schema level. Two Group C protocols—say, Avro and Protobuf—share the same schema theory: `colimit(ThGraph, ThConstraint, ThMulti)`. A Schema graph that is a valid Avro schema is, structurally, also a valid Protobuf schema. The conversion between them reduces to remapping protocol-specific surface syntax: Avro's `"type": "record"` becomes Protobuf's `message`, Avro's `"type": ["null", "string"]` becomes Protobuf's `optional string`. The graph structure itself does not change.

**Cross-group conversion** involves a structural mismatch. Converting from Group B (SQL, hypergraph + functor) to Group C (Protobuf, multigraph + W-type) loses hyperedge structure: foreign keys, composite unique constraints, and multi-column primary keys cannot be expressed in Group C's multigraph schema theory. Converting from Group C to Group B in the other direction loses tree-shaped instance semantics: SQL rows are flat, and the nested document structure of a Group C instance does not survive the trip to a relational table without denormalization decisions that the engine cannot make automatically.

```{mermaid}

%%| fig-cap: "Theory groups and conversion fidelity. Solid arrows indicate structurally lossless conversion (within-group). Dashed arrows indicate conversion with structural loss (cross-group)."
graph LR
    A["Group A<br/>CQL"]
    B["Group B<br/>SQL, Cassandra, DynamoDB"]
```

### A Note on "Lossless"

"Structurally lossless at the schema level" means the graph structure of the schema survives the round trip. It does not mean the round trip produces identical source bytes, or that all protocol-specific metadata is preserved. Protobuf field numbers, for example, are not part of the multigraph structure—they are a surface feature of the wire format. A Protobuf-to-Avro-to-Protobuf round trip will not preserve field numbers; the emit step will assign fresh ones. The schema graph—the message names, field names, types, and relationships—will be intact, but the wire-format metadata will not.

If your conversion requirements include preserving wire-format metadata, you can carry it as an opaque annotation through the Schema graph. panproto's Schema graph supports arbitrary key-value metadata on vertices and edges; the parse functions use this to preserve information that is meaningful to the source format but has no universal semantic. The emit function for the target format can then ignore this metadata, or a custom emit wrapper can read it and apply it selectively.


## Comparison with pandoc

The pandoc analogy that opened this chapter is worth examining in more detail, because the structural parallel is exact in some places and breaks down in others.

**What is the same.** Both pandoc and panproto convert between multiple formats using a shared intermediate representation. pandoc's IR is a document abstract syntax tree: a tree of Pandoc elements (paragraphs, headers, code blocks, spans, and so on). panproto's IR is the Schema graph: a multigraph or hypergraph with constraint annotations. In both systems, the IR is the canonical representation, and all conversion goes through it. Neither system converts directly between pairs of formats; all paths go through the IR.

Both systems also face the same fundamental limitation: information that cannot be represented in the IR is lost. pandoc cannot round-trip Word documents with custom styles, tracked changes, or embedded macros, because these features have no equivalent in the Pandoc AST. panproto cannot round-trip SQL schemas with arbitrary `CHECK` expressions, because these expressions are opaque at the schema graph level.

**What is different.** pandoc's IR is a fixed, hand-designed document AST. Adding a new element type to the AST requires changing pandoc's source code. panproto's IR is not fixed—it is the Schema graph, whose structure is determined by the protocol theories. Adding a new structural concept (a new sort of vertex or edge) means extending a theory via a colimit, not patching the IR definition.

This difference matters for extensibility. When a new schema format appears—say, a domain-specific schema language for a new database engine—panproto can accommodate it by defining a new protocol theory. The Schema graph automatically gains the new format's structural concepts as a natural extension of the theory. pandoc, by contrast, would require an AST extension to handle document constructs with no existing element type.

The second difference is in the use of the IR. In pandoc, the IR exists solely for conversion; you interact with it by writing filters that transform the AST. In panproto, the Schema graph is the primary representation for all operations: diff, migration, existence checking, lens derivation, breaking-change detection, and conversion all operate on the same Schema graph. Format conversion is not a separate subsystem—it is the composition of `parse_*` and `emit_*` applied to the same object that the rest of the toolchain already uses.

| | pandoc | panproto |
|---|---|---|
| Converts between | Document formats | Schema formats |
| Intermediate representation | Pandoc document AST | Schema graph |
| IR defined by | Fixed Haskell data types | Protocol theories (extensible) |
| IR used for | Conversion only | Conversion, diff, migration, lenses, ... |
| Loss mechanism | AST has no element for X | Schema graph has no vertex/edge for X |
| Format-specific extensions | Custom AST metadata | Theory-specific constraint annotations |

: Structural comparison between pandoc and panproto. {#tbl-pandoc-comparison}


## Summary

- panproto is to schemas what pandoc is to documents: it converts between formats using a universal intermediate representation, the Schema graph.

- Every protocol ships with a `parse_*` function (source format → Schema graph) and an `emit_*` function (Schema graph → target format). All 54 built-in protocols support both directions.

- The **conversion pipeline** is: `parse_*()` → Schema graph → (optional migration) → `emit_*()`. The optional migration step allows simultaneous format conversion and structural transformation.

- **Round-trip fidelity** depends on what the source and target formats share structurally. Information that fits in the Schema graph survives; information that does not—hyperedges in a format without hyperedge support, constraint keywords in a format without a constraint vocabulary—is dropped.

- **Theory groups** make fidelity precise. Protocols sharing the same schema and instance theory (same group) convert with structural losslessness. Cross-group conversions involve structural mismatches: Group B (SQL, hypergraph) to Group C (Protobuf, multigraph) loses hyperedge structure; Group C to Group E (Parquet, functor instances) faces an instance-theory mismatch.

- panproto's IR is not a fixed data structure but a schema graph whose shape is determined by extensible theories. This distinguishes it from pandoc's fixed document AST: new schema formats extend the theory space, and the IR adapts accordingly.

- Format conversion in panproto is not a separate subsystem. It is the composition of two operations—parse and emit—applied to the same Schema graph that the rest of the toolchain already uses for diffs, migrations, and lens derivation.

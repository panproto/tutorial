13  Schematic Version Control

Schemas evolve. Fields get added, types change, constraints tighten. When multiple teams develop against the same schema, their changes diverge and eventually need merging. The schema CLI provides git-style version control for this process, except that instead of tracking text it tracks mathematical structure.

13.1 The Git Analogy

Git concept panproto equivalent
Blob (file content) Schema (graph structure)
Diff/patch Migration (graph morphism)
Three-way merge Colimit (pushout of schemas)
Content-addressing (SHA) Content-addressing (blake3)
Commit DAG Schema evolution DAG
Branch Named pointer to a schema version
Conflict Structural incompatibility

The important difference: git merges text and hopes the result compiles. panproto merges structure and guarantees data integrity.

13.2 Quick Start

schema init
schema add schema.json
schema commit -m "initial ATProto schema"

schema branch add-verification
schema checkout add-verification
schema add schema-v2.json
schema commit -m "add verification status field"

schema checkout main
schema merge add-verification

13.3 The Object Store

Every schema, migration, and commit is identified by a blake3 hash of its canonical form. Two schemas with the same structure produce the same hash, even if they were created independently by different teams on different continents. Identity comes from structure, not naming.

.panproto/
  objects/<hex[0..2]>/<hex[2..]>   # content-addressed objects
  refs/heads/main                   # branch pointers
  refs/tags/v1.0                    # tag pointers
  HEAD                              # current branch
  logs/                             # reflog (audit trail)

13.4 Command Reference

13.4.1 schema init

Creates a .panproto/ directory with the object store, refs, and HEAD pointing to main.

In git, init creates .git/ for tracking arbitrary files. Here it creates .panproto/ for tracking a single schema and its evolution. There is no working tree in the git sense; the repository tracks one schema at a time, not a directory of files.

schema init
schema init my-project

13.4.2 schema add

Stages a schema for the next commit. Loads the schema JSON, computes a structural diff against the current HEAD’s schema, auto-derives a migration (the graph morphism mapping surviving vertices and edges), and validates the migration via the existence checker.

In git, add stages file contents for commit. Here it stages a schema graph and simultaneously computes and validates the migration morphism that transforms the old schema into the new one. If the diff involves vertex renames or edge contractions that cannot be auto-derived, you can supply an explicit migration file with --migration.

schema add schema.json
schema add schema.json --migration explicit-migration.json

13.4.3 schema commit

Creates a commit from the staged schema. The commit object stores the schema’s content hash, the migration’s content hash, parent commit IDs, and metadata. Advances the current branch ref and appends a reflog entry.

In git, a commit snapshots the entire working tree. Here a commit snapshots a single schema graph and the morphism that produced it from its parent. Every commit carries enough information to migrate instance data forward or backward.

schema commit -m "add verification status field"

13.4.4 schema status

Shows the current HEAD (branch name or detached commit ID) and whether anything is staged.

13.4.5 schema log

Walks the commit DAG from HEAD backwards, printing each commit’s ID, author, timestamp, message, and schema hash. For merge commits, shows all parent IDs.

In git, log shows file-level changes. Here it shows schema-level evolution: which vertices and edges were added, removed, or modified at each step.

schema log
schema log -n 5

13.4.6 schema diff

Computes a structural diff between two schema files. Reports added and removed vertices and edges, kind changes, constraint changes, variant changes, ordering changes, and usage mode changes.

Git’s diff operates on lines of text. This diff operates on the schema graph: it knows that adding a vertex is different from adding an edge, that tightening a constraint is different from relaxing one, and that removing a coproduct variant is a type error while removing an optional field is merely lossy.

schema diff old-schema.json new-schema.json

13.4.7 schema show

Inspects any object in the store by ref name or object ID. For commits, shows the schema ID, parent IDs, migration ID, protocol, author, and message. For schemas, shows vertex and edge counts. For migrations, shows the vertex and edge mapping counts.

Git’s show displays file contents and diffs. This show displays the internal structure of content-addressed schema objects.

schema show main
schema show v1.0

13.4.8 schema branch

Creates, lists, or deletes branches. Branches are lightweight pointers to commit IDs, exactly as in git. With no arguments, lists all branches and marks the current one.

schema branch
schema branch feature
schema branch -d feature

13.4.9 schema tag

Creates, lists, or deletes tags. Tags are immutable pointers to commits, typically used for schema releases.

schema tag v1.0
schema tag -d v1.0

13.4.10 schema checkout

Switches HEAD to the named branch, or detaches HEAD at a specific commit. In git, checkout also updates the working tree; here it updates which schema version HEAD points to.

schema checkout feature

13.4.11 schema merge

Merges a branch into the current branch. Finds the merge base (lowest common ancestor), computes diffs from the base to both tips, and applies all non-conflicting changes.

This is where panproto diverges most from git. Git’s merge operates on lines of text and produces conflict markers when the same lines were edited. panproto’s merge operates on the schema graph and produces typed conflicts: BothModifiedVertex (both branches changed the same vertex’s kind differently), BothModifiedConstraint (both changed the same constraint value), DeleteModifyVertex (one branch deleted a vertex the other modified), RemovedVariant (a coproduct variant was dropped, making existing data ill-typed), or LinearityTightened (an edge’s usage mode became more restrictive). If there are no conflicts, the merge is auto-committed. If there are, the conflicts are reported with their structural types.

Fast-forward merges work as in git: if one side is an ancestor of the other, the ref is simply moved forward.

schema merge feature

13.4.12 schema rebase

Replays the current branch’s commits onto another branch. Finds the merge base, collects all commits from the base to HEAD, then replays each one on top of the target via three-way merge. If any step conflicts, the rebase stops with an error.

The mechanics are the same as git rebase, but each replay step uses structural three-way merge rather than textual merge. This means a rebase can detect schema-level conflicts (like incompatible kind changes) that git rebase would miss.

schema rebase main

13.4.13 schema cherry-pick

Applies a single commit’s schema change to the current branch. Extracts the diff between the commit and its parent, then three-way merges it onto HEAD using the parent as the base.

Same idea as git cherry-pick, but the “diff” being applied is a schema graph morphism, not a text patch.

schema cherry-pick abc1234...

13.4.14 schema reset

Moves HEAD to a different commit. Three modes, matching git:

  • --soft: moves the ref only; the index is unchanged
  • --mixed (default): moves the ref and clears the index
  • --hard: moves the ref, clears the index, and overwrites the working schema

All modes append a reflog entry, so the old position is recoverable.

schema reset --soft v1.0
schema reset --hard HEAD~3

13.4.15 schema stash

Saves the current staged schema to a stash stack and clears the index. pop restores the most recent stash. list shows all entries. drop discards the top entry.

In git, stash saves uncommitted working tree changes. Here it saves the staged schema so you can switch branches without committing.

schema stash push -m "work in progress"
schema stash pop
schema stash list

13.4.16 schema reflog

Shows the history of mutations to a ref. Every commit, merge, checkout, reset, rebase, and cherry-pick appends an entry recording the old and new ref values. This is the recovery mechanism: if a reset or rebase moves HEAD past a commit you need, the reflog still has the old ID.

Works identically to git reflog.

schema reflog
schema reflog main

13.4.17 schema bisect

Binary search for the commit that introduced a breaking change. Given a known-good and known-bad commit, bisect finds the path between them and presents the midpoint for testing. You report whether the midpoint is good or bad, and it narrows the range. Converges in O(log n) steps.

Git bisect tests whether code compiles or a test passes. Here you can test whether a schema change introduced a breaking change, a validation failure, or any other property you care about.

schema bisect v1.0 HEAD

13.4.18 schema blame

Shows which commit introduced a specific schema element: a vertex, an edge, or a constraint. Walks the DAG backwards from HEAD, comparing each commit’s schema to its parent’s, and returns the commit where the element first appeared or was last modified.

Git blame operates on lines of text in a file. This blame operates on structural elements of a schema graph. You can ask “who added the user_status vertex?” or “when was the maxLength constraint on text changed?”

schema blame --element-type vertex user_status
schema blame --element-type constraint "text:maxLength"

13.4.19 schema lift

Migrates instance data from one schema version to another. Finds the path between two refs in the DAG, composes all migrations along the path into a single morphism, and applies the functorial lift to the record.

This has no git equivalent. Git tracks file history but cannot transform file contents based on structural changes. panproto can, because migrations are graph morphisms with a well-defined action on instance data. The commutativity guarantee ensures the result is the same regardless of which path you take through the DAG.

schema lift --migration mig.json --src-schema v1.json --tgt-schema v2.json record.json

13.4.20 schema gc

Marks all objects reachable from branches, tags, stash, and HEAD, then deletes everything else. Safe to run at any time.

Works like git gc but simpler: there are no packfiles or loose object optimization. Objects are either reachable or deleted.

schema gc

10  Building Your Own Protocol

So far we have worked with protocols that panproto ships out of the box—76 protocols across 9 categories—serialization, data schemas, APIs, databases, type systems, web/document formats, data science, domain-specific, and configuration. But panproto is not a fixed catalog. It is a toolkit for building schema languages, and the mechanism for doing so is the colimit of theories.

This chapter shows how to define a new protocol from scratch by composing reusable component theories, and how the full machinery—migrations, existence checking, diff, lifting—comes along for free.

10.1 Colimits (Pushouts) of Theories

A colimit is the categorical way to glue two structures together along a shared part. In panproto, the structures are GAT theories and the shared part is a smaller theory that both include.

Concretely, given two theories \(T_1\) and \(T_2\) that each contain (via theory morphisms) a common sub-theory \(S\), the colimit \(T_1 +_S T_2\) produces a new theory that:

  1. Contains all sorts and operations from both \(T_1\) and \(T_2\).
  2. Identifies anything that came from the shared base \(S\), so nothing is duplicated.
  3. Preserves all equations from both sides.

The Rust API exposes this as a single function:

pub fn colimit(t1: &Theory, t2: &Theory, shared: &Theory) -> Result<Theory, GatError>;

10.1.1 A Concrete Example

Consider two theories:

  • ThGraph with sorts {Vertex, Edge} and operations {src: Edge → Vertex, tgt: Edge → Vertex}.
  • ThConstraint with sorts {Vertex, Constraint} and operations {target: Constraint → Vertex}.

Both theories contain the sort Vertex. The shared base ThVertex is simply the theory with the single sort {Vertex} and no operations.

The colimit ThConstrainedGraph = colimit(ThGraph, ThConstraint, ThVertex) merges them:

  • Sorts: {Vertex, Edge, Constraint} — Vertex appears once, not twice.
  • Operations: {src, tgt, target} — all operations are retained.

The colimit legs are theory morphisms from each component into the result. These morphisms are what let panproto track the provenance of each sort and operation—essential for computing migrations later.

Note

Colimits of GATs are computed by taking a coequalizer in the category of presentations. The implementation follows the algorithm described in Lynch et al. (2024), which handles the subtleties of dependent sorts and parametric operations that arise in full GATs (not just single-sorted algebraic theories).

10.2 Building a Custom Protocol

Let us walk through defining a protocol for TOML configuration files. A TOML config has hierarchical sections, each containing key-value pairs with typed values.

10.2.1 Defining the Protocol

We compose component theories to describe the shape of TOML schemas:

import { Panproto } from '@panproto/core';

const panproto = await Panproto.init();

// start snippet define-protocol
const tomlConfig = panproto.defineProtocol({
  name: 'toml-config',
  schemaTheory: 'ThConstrainedGraph',
  instanceTheory: 'ThWType',
  edgeRules: [
    { edgeKind: 'key', srcKinds: ['section'], tgtKinds: [] },
    { edgeKind: 'subsection', srcKinds: ['section'], tgtKinds: ['section'] },
  ],
  objKinds: ['section'],
  constraintSorts: ['required', 'deprecated'],
});
// end snippet define-protocol

// start snippet use-protocol
const configSchema = tomlConfig.schema()
  .vertex('root', 'section')
  .vertex('root.database', 'section')
  .vertex('root.database.host', 'string')
  .vertex('root.database.port', 'integer')
  .edge('root', 'root.database', 'subsection', { name: 'database' })
  .edge('root.database', 'root.database.host', 'key', { name: 'host' })
  .edge('root.database', 'root.database.port', 'key', { name: 'port' })
  .constraint('root.database.host', 'required', 'true')
  .build();
// end snippet use-protocol

The protocol definition tells panproto what sorts of things a TOML schema can contain (sections, keys, value types) and how they relate (keys belong to sections, sections can nest). Under the hood, this constructs a GAT theory via colimits of the component theories we selected.

10.2.2 Using the Protocol

With the protocol defined, we can build schemas within it:

import { Panproto } from '@panproto/core';

const panproto = await Panproto.init();

// start snippet define-protocol
const tomlConfig = panproto.defineProtocol({
  name: 'toml-config',
  schemaTheory: 'ThConstrainedGraph',
  instanceTheory: 'ThWType',
  edgeRules: [
    { edgeKind: 'key', srcKinds: ['section'], tgtKinds: [] },
    { edgeKind: 'subsection', srcKinds: ['section'], tgtKinds: ['section'] },
  ],
  objKinds: ['section'],
  constraintSorts: ['required', 'deprecated'],
});
// end snippet define-protocol

// start snippet use-protocol
const configSchema = tomlConfig.schema()
  .vertex('root', 'section')
  .vertex('root.database', 'section')
  .vertex('root.database.host', 'string')
  .vertex('root.database.port', 'integer')
  .edge('root', 'root.database', 'subsection', { name: 'database' })
  .edge('root.database', 'root.database.host', 'key', { name: 'host' })
  .edge('root.database', 'root.database.port', 'key', { name: 'port' })
  .constraint('root.database.host', 'required', 'true')
  .build();
// end snippet use-protocol

10.2.3 Everything Works Immediately

Here is the key payoff: because the protocol is defined as a GAT theory, every piece of panproto’s machinery operates on it without any additional code.

Diff detects what changed between two versions of a TOML schema:

const report = panproto.diff(configV1, configV2);
// Reports: added key "timeout" in section "server",
//          renamed section "db" → "database"

Existence checking tells you whether a migration between two schema versions can be constructed:

const existence = panproto.checkExistence(configV1, configV2, migration);
// Reports whether the proposed migration handles all changes

Lifting migrates actual TOML data through the schema change:

const newConfig = migration.lift(oldConfig);
// Moves values into renamed sections, supplies defaults for new keys

Lens decomposition gives you bidirectional access:

const view = migration.get(oldConfig);    // forward view
const restored = migration.put(view, complement);  // round-trip

None of this required writing TOML-specific migration logic. The theory determined the protocol’s structure; panproto’s generic engine did the rest.

10.3 Theory Library

panproto ships with a library of reusable component theories designed to be composed via colimits:

Theory What it provides
ThGraph Vertices and directed edges (src, tgt)
ThConstraint Constraint annotations on vertices
ThMulti Multiple edge types between the same vertices
ThHypergraph Hyperedges connecting arbitrary sets of vertices
ThInterface Input/output interface boundaries
ThWType W-type structure for recursive/inductive data
ThFunctor Functorial mappings between graph-like structures
ThMeta Meta-level annotations (documentation, deprecation)
ThFlat Flat key-value records (no nesting)

Every protocol panproto ships is defined as a colimit of theories from this library. The library is not closed; you can define your own component theories and use them in colimits alongside the built-in ones.

Note

The compositional nature of colimits means that if two protocols share a component theory, panproto can automatically construct a partial translation between them. This is the foundation for cross-protocol migration, which we touched on in Chapter 5 and which becomes fully systematic once you see protocols as colimits.

10.3.1 All Protocols as Colimits

Every one of panproto’s 76 built-in protocols falls into one of six theory groups, determined by which component theories its colimit requires. The table below summarizes the groups; the instance theory column names the theory used to represent data (instances) for schemas in that group.

Group membership determines which generic algorithms apply and how cross-group migrations are constructed—for instance, migrating from a Group C protocol (Protobuf) to a Group A protocol (JSON Schema) requires a theory morphism that embeds the simpler ThSimpleGraph into the richer colimit(ThGraph, ThConstraint, ThMulti). The migration morphism itself remains the same kind of graph homomorphism described throughout this tutorial.

See Chapter 18 for the complete catalog of all 76 protocols with their theory group assignments.

10.4 Summary

Defining a new protocol in panproto is not an act of implementing a plugin or writing an adapter. It is an act of specifying a theory, typically by composing existing theories via colimits. The colimit merges shared structure, the theory morphisms track provenance, and the generic engine provides migrations, diffs, existence checking, and lifting—all derived from the mathematics.

This is what it means for panproto to be protocol-parametric: the protocol is a parameter, and changing it requires no changes to the engine.

10.4.1 Further Reading

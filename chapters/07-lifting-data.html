7  Lifting Data

The previous chapter established when a migration exists. This chapter is about how it works: the restrict operation that transforms data from one schema to another. This is the algorithmic heart of panproto.

There are two flavors of restriction, corresponding to the two kinds of schema that panproto supports. W-type restrict handles tree-structured data (JSON, CBOR, protocol buffers) by walking a five-step pipeline over the instance tree. Functor restrict handles tabular data (SQL) by precomposition along a schema morphism. Both are grounded in the same categorical idea—pulling data back along a map—but the concrete algorithms differ because trees and tables have different structure.

We will spend most of our time on the W-type pipeline, since it is the more intricate of the two and the one that handles the recursive, nested data formats that dominate modern protocol design.

7.1 A worked example: threadViewPost

To make the pipeline concrete, we need a schema that exercises every step. The ATProto threadViewPost type is ideal: a post that contains an array of replies, each of which is itself a post. The recursion means the instance is a tree of arbitrary depth.

Suppose version 1 of the schema looks like this (simplified):

threadViewPost
├── post: postView
│   ├── uri: string
│   ├── author: profileViewBasic
│   │   ├── did: string
│   │   ├── handle: string
│   │   └── displayName: string
│   ├── text: string
│   └── likeCount: integer
└── replies: array<threadViewPost>

And version 2 drops likeCount from postView and drops displayName from profileViewBasic. A migration \(m: S_2 \to S_1\) exists (both changes are field removals, which are always backward-compatible). The migration’s vertex map tells us which vertices in \(S_1\) survive in \(S_2\).

Now consider a concrete instance—a thread with two replies, one of which has a nested reply:

root (threadViewPost)
├── post (postView)
│   ├── uri = "at://did:plc:abc/post/1"
│   ├── author (profileViewBasic)
│   │   ├── did = "did:plc:abc"
│   │   ├── handle = "alice.bsky.social"
│   │   └── displayName = "Alice"
│   ├── text = "Hello world"
│   └── likeCount = 42
└── replies
    ├── reply-1 (threadViewPost)
    │   ├── post (postView)
    │   │   ├── uri = "at://did:plc:def/post/2"
    │   │   ├── author (profileViewBasic)
    │   │   │   ├── did = "did:plc:def"
    │   │   │   ├── handle = "bob.bsky.social"
    │   │   │   └── displayName = "Bob"
    │   │   ├── text = "Nice post"
    │   │   └── likeCount = 7
    │   └── replies
    │       └── reply-1-1 (threadViewPost)
    │           └── ...
    └── reply-2 (threadViewPost)
        └── ...

This is the instance we will push through the five-step pipeline.

7.2 W-type restrict: the five-step pipeline

The W-type restrict algorithm transforms an instance tree in five sequential steps. Each step has a clear algebraic meaning, and together they implement the restriction of a polynomial functor algebra along a morphism of polynomial functors. The correspondence to the theory is not decorative—it is what guarantees correctness.

7.2.1 Step 1: anchor_surviving

The first step is the simplest: walk every node in the instance and check whether its anchor vertex (the schema vertex that the node is an instance of) survives the migration’s vertex map. If the anchor vertex is not in the image of the vertex map, the node is marked for removal.

/// Mark each node as surviving or pruned based on whether
/// its anchor vertex is in the migration's vertex map.
fn anchor_surviving(
    instance: &WTree,
    vertex_map: &HashMap<VertexId, VertexId>,
) -> Vec<bool> {
    instance.nodes.iter().map(|node| {
        vertex_map.values().any(|v| *v == node.anchor)
    }).collect()
}

This is signature restriction on polynomial algebras: given a polynomial \(P = \sum_{i \in I} y^{B_i}\) and a sub-signature \(J \subseteq I\), the restricted polynomial is \(P|_J = \sum_{j \in J} y^{B_j}\). An algebra (instance) over \(P\) restricts to an algebra over \(P|_J\) by discarding the operations whose index falls outside \(J\) (Abbott et al. 2005).

In our example, the vertices likeCount and displayName are not in the image of the vertex map (they were dropped in version 2). So every node anchored at likeCount or displayName is marked for removal:

  • root.post.likeCount — pruned
  • root.post.author.displayName — pruned
  • reply-1.post.likeCount — pruned
  • reply-1.post.author.displayName — pruned
  • (and so on for every nested reply)

Everything else survives.

NoteWhy “anchor”?

In the W-type formalization, each node in an instance tree sits over a vertex in the schema polynomial. The schema vertex is the node’s anchor because it determines the node’s shape: what children it can have, what edges connect it to its parent, and what data it carries. When we say a node’s anchor survives, we mean the schema still has a place for that kind of node.

7.2.2 Step 2: reachable_from_root

After marking survivors, some surviving nodes may be stranded: they are still marked as surviving, but they can only be reached from the root through pruned nodes. A sub-tree of a W-type is not necessarily a W-type—it may be disconnected.

Step 2 runs a breadth-first search from the root, traversing only through surviving nodes. Any surviving node not visited by the BFS is marked as unreachable and will not appear in the output.

/// BFS from root through surviving nodes. Returns the set of
/// node indices that are both surviving and reachable.
fn reachable_from_root(
    instance: &WTree,
    surviving: &[bool],
) -> HashSet<u32> {
    let mut visited = HashSet::new();
    let mut queue = VecDeque::new();

    if surviving[instance.root as usize] {
        queue.push_back(instance.root);
    }

    while let Some(idx) = queue.pop_front() {
        if !visited.insert(idx) {
            continue;
        }
        for child in instance.children(idx) {
            if surviving[child as usize] {
                queue.push_back(child);
            }
        }
    }

    visited
}

This is standard connected-component extraction. In our example it has no effect, because removing likeCount and displayName nodes does not disconnect anything—they are leaf nodes. But consider a different migration that removes postView while keeping text. The text nodes would survive step 1 but become unreachable in step 2, because their parent (postView) was pruned and no path from the root reaches them through survivors alone.

NoteWhen does step 2 matter?

Step 2 is a no-op whenever every pruned node is a leaf or when the surviving nodes form a connected sub-tree. It becomes essential when intermediate structural nodes are removed—for example, when an entire wrapper object is dropped from a schema but some of its children are “hoisted” to a higher level in a later step.

7.2.3 Step 3: ancestor_contraction

This is the tree surgery step. When intermediate nodes are pruned, children that survived steps 1 and 2 need to be re-parented to their nearest surviving ancestor. The result is a new tree where every edge connects two surviving nodes.

/// For each surviving node, find its nearest surviving ancestor.
/// Returns a map from node index to new parent index.
fn ancestor_contraction(
    instance: &WTree,
    reachable: &HashSet<u32>,
) -> HashMap<u32, u32> {
    let mut new_parent = HashMap::new();

    for &idx in reachable {
        if idx == instance.root {
            continue;
        }
        // Walk up the original tree until we find a surviving ancestor
        let mut ancestor = instance.parent(idx);
        while !reachable.contains(&ancestor) {
            ancestor = instance.parent(ancestor);
        }
        new_parent.insert(idx, ancestor);
    }

    new_parent
}

In algebraic graph transformation terms, this is computing the pushout complement (Ehrig et al. 2006, sec. 3.1). We have a span \(L \hookleftarrow K \hookrightarrow R\) where \(L\) is the original tree, \(K\) is the interface (surviving nodes with their original connectivity), and \(R\) is the contracted tree. The pushout complement identifies how to glue \(R\) into the surrounding context.

In our threadViewPost example, the pruned nodes (likeCount, displayName) are all leaves, so no contraction occurs. But imagine a migration that removes the postView wrapper while keeping uri, author, and text as direct children of threadViewPost. Then postView nodes are pruned, and their surviving children are re-parented to the threadViewPost node:

Before:                          After:
threadViewPost                   threadViewPost
├── postView (removed)    →     ├── uri
│   ├── uri                      ├── author
│   ├── author                   │   ├── did
│   │   ├── did                  │   └── handle
│   │   ├── handle               ├── text
│   │   └── displayName          └── replies
│   ├── text                         └── ...
│   └── likeCount
└── replies
    └── ...
NoteContraction preserves depth order

A node’s depth in the contracted tree is always less than or equal to its depth in the original tree. Contraction never pushes nodes deeper. This is a consequence of the fact that we only re-parent to ancestors, never to siblings or descendants.

7.2.4 Step 4: resolve_edge

After contraction, each re-parented child has a new parent, but the edge connecting them needs a label from the target schema. In the original tree, the edge from A to C went through B: A --(e1)--> B --(e2)--> C. After removing B, we need a single edge label for the direct A --> C connection.

In a simple graph (at most one edge between any two vertices), this is trivial: there is only one choice. But schemas are often multigraphs—multiple edges can exist between the same pair of vertices (think of an object with two fields of the same type). In that case, the choice is ambiguous.

The resolver table provides the deterministic choice. It is part of the migration specification and maps each contracted path (source_vertex, target_vertex, intermediate_path) to a specific edge in the target schema.

/// Given a contracted parent-child pair, look up the correct
/// edge label from the resolver table.
fn resolve_edge(
    parent_anchor: VertexId,
    child_anchor: VertexId,
    contracted_path: &[VertexId],
    resolver: &ResolverTable,
) -> EdgeId {
    resolver
        .lookup(parent_anchor, child_anchor, contracted_path)
        .expect("migration guarantees a resolution exists")
}

This is exactly the pushout complement selection problem (Löwe 1993). When the pushout complement is not unique (which happens precisely when the interface graph has parallel edges), an additional choice is required to determine which complement to use. The resolver table encodes this choice. The existence checker from Chapter 6 guarantees that a valid resolution exists for every contracted path, so resolve_edge never fails on a well-formed migration.

NoteWhen is resolution trivial?

For JSON-like schemas (which are trees, not multigraphs), resolution is always trivial: there is at most one edge between any pair of vertices in the schema. The resolver table is only needed for schemas where the same type appears in multiple positions under the same parent—protocol buffers with repeated message types, for instance.

7.2.5 Step 5: reconstruct_fans

The final step handles hypergraph schemas—schemas where an edge can connect a vertex to multiple children simultaneously. In practice, this means SQL: a table row is a hyperedge connecting a primary key to all its column values.

When some columns of a table are pruned, the surviving columns must be collected into a new, smaller hyperedge (a new “fan”). This is limit cone construction: the surviving columns form a sub-diagram, and the new fan is the limit of that sub-diagram in the category of fans (Barr and Wells 1990, Ch. 4).

/// Reconstruct fans (hyperedges) after pruning some children.
/// Each surviving fan keeps only its surviving children and
/// receives a new fan ID from the target schema.
fn reconstruct_fans(
    instance: &WTree,
    reachable: &HashSet<u32>,
    fan_map: &HashMap<FanId, FanId>,
) -> Vec<Fan> {
    instance.fans.iter().filter_map(|fan| {
        let new_children: Vec<u32> = fan.children.iter()
            .copied()
            .filter(|c| reachable.contains(c))
            .collect();

        if new_children.is_empty() {
            return None;
        }

        let new_fan_id = fan_map.get(&fan.id)?;
        Some(Fan {
            id: *new_fan_id,
            parent: fan.parent,
            children: new_children,
        })
    }).collect()
}

For pure tree schemas (JSON, CBOR, protocol buffers), step 5 is a no-op: every edge connects exactly one parent to exactly one child, so there are no fans to reconstruct. The step exists because panproto’s data model is general enough to handle both trees and hypergraphs uniformly.

NoteThe five steps in summary
Step Operation Algebraic meaning When it matters
1 anchor_surviving Signature restriction Always
2 reachable_from_root Connected component Intermediate nodes pruned
3 ancestor_contraction Pushout complement Children of pruned nodes survive
4 resolve_edge Complement selection Multigraph schemas
5 reconstruct_fans Limit cone Hypergraph (SQL) schemas

For a typical JSON schema migration (adding or removing fields), only step 1 does meaningful work. Steps 2–5 are structural no-ops. This is by design: the common case should be fast.

7.3 The pipeline in sequence

Let us trace our threadViewPost example through all five steps.

Input: An instance tree with nodes for threadViewPost, postView, profileViewBasic, uri, author, did, handle, displayName, text, likeCount, and replies, recursively nested to depth 3.

Step 1 — anchor_surviving: Mark likeCount and displayName nodes as pruned. All other nodes survive. In a thread with 3 posts, this marks 6 nodes for removal (one likeCount and one displayName per post).

Step 2 — reachable_from_root: BFS from root. All surviving nodes are reachable (the pruned nodes are leaves). No additional nodes are removed.

Step 3 — ancestor_contraction: No contraction needed. Every surviving node’s parent is also surviving.

Step 4 — resolve_edge: No re-parenting occurred, so no edges need resolution. All original edges between surviving nodes carry over directly.

Step 5 — reconstruct_fans: The schema is a tree, not a hypergraph. No fans to reconstruct.

Output: The original instance tree, minus the likeCount and displayName leaf nodes. Every other node, edge, and value is preserved exactly.

This is the common case for protocol evolution: field removal produces a simple, fast restriction that touches only the affected leaves.

7.4 Functor restrict (\(\Delta_F\) precomposition)

For tabular data (SQL schemas), restriction takes a different form. Given a schema morphism \(F: S_1 \to S_2\) and an instance \(I: S_2 \to \mathbf{Set}\), the restricted instance is the precomposition \(\Delta_F(I) = I \circ F : S_1 \to \mathbf{Set}\).

In concrete terms: if \(F\) drops a table from the schema, the rows for that table vanish from the instance. If \(F\) drops a column, that column disappears from every row. If \(F\) renames a table, the rows are relabeled. The operation is purely structural—no data is computed, only rearranged.

/// Functor restriction: precompose an instance along a schema morphism.
fn restrict_functor<F, I>(morphism: &F, instance: &I) -> RestrictedInstance
where
    F: SchemaMorphism,
    I: Instance,
{
    let mut result = RestrictedInstance::new(morphism.source());
    for table in morphism.source().tables() {
        let target_table = morphism.map_table(table);
        for row in instance.rows(target_table) {
            let restricted_row = row.project(morphism.column_map(table));
            result.insert(table, restricted_row);
        }
    }
    result
}

Precomposition is always total: if \(F\) is a well-typed schema morphism and \(I\) is a well-typed instance, then \(I \circ F\) is a well-typed instance over \(S_1\). There are no existence conditions to check—the operation cannot fail (Spivak 2012).

This simplicity is exactly why functor restriction is the right abstraction for SQL migrations. The complexity is not in the restriction itself but in constructing the morphism \(F\), which is the job of the migration specification.

7.4.1 Adjoints: \(\Sigma_F\) and \(\Pi_F\)

The restriction functor \(\Delta_F\) sits in the middle of an adjoint triple:

\[\Sigma_F \dashv \Delta_F \dashv \Pi_F\]

The left adjoint \(\Sigma_F\) (pushforward, or left Kan extension) computes the “best approximation” of an \(S_1\)-instance as an \(S_2\)-instance by taking colimits. In SQL terms, this is roughly UNION across fibers. The right adjoint \(\Pi_F\) (right Kan extension) computes the “most conservative approximation” by taking limits—roughly JOIN across fibers (Mac Lane 1998, Ch. X).

Both adjoints are computationally more expensive than \(\Delta_F\) and are outside panproto’s current scope. The key point is that \(\Delta_F\) alone gives us everything we need for the forward direction of a migration: projecting existing data onto a new schema.

NoteWhy not \(\Sigma_F\)?

The pushforward \(\Sigma_F\) would let us add data to an instance (e.g., populate a new required column with default values). This is sometimes called “data migration” in the database literature. panproto deliberately limits itself to \(\Delta_F\) because restriction is lossless and reversible (via the complement—see Chapter 8), while \(\Sigma_F\) requires inventing data and is generally not reversible. When panproto needs to populate defaults, it does so through the lens complement mechanism rather than through \(\Sigma_F\).

7.5 Performance

The restrict pipeline targets >1 million records per second for simple projections (field additions and removals) running in WASM. Several design decisions make this possible.

7.5.1 Compiled migrations

When a migration is loaded, panproto pre-computes everything it can:

  • Surviving sets: The set of schema vertices that survive is computed once and stored as a bitset. Step 1 becomes a single bitset lookup per node.
  • Remap tables: Edge resolution (step 4) is pre-computed into a flat lookup table indexed by (source_anchor, child_anchor). No hash map lookups at runtime.
  • Fan maps: Fan reconstruction (step 5) is pre-computed into a mapping from old fan IDs to new fan IDs with pre-allocated child slots.

The migration specification is effectively compiled into a small program that the restrict pipeline executes.

7.5.2 Arena allocation

Instance trees are allocated in a bumpalo arena. This has two benefits:

  1. Allocation is O(1): Bump allocation is a pointer increment. No free-list management, no fragmentation.
  2. Deallocation is O(1): The entire arena is freed at once when the instance is no longer needed. No tree of destructors to walk.

For the restrict pipeline specifically, the output tree is allocated in a fresh arena. Surviving nodes are written sequentially into the output arena, giving excellent cache locality.

7.5.3 Zero-copy fast path

For the common case—no contraction (step 3 is a no-op), no fan reconstruction (step 5 is a no-op), and trivial edge resolution (step 4 is a no-op)—the pipeline reduces to:

  1. Walk the input tree.
  2. For each node, check the surviving bitset.
  3. If surviving, copy the node to the output arena with a rewritten anchor field.

The node’s payload (the actual data—strings, integers, booleans) is not copied. Only the structural metadata (anchor, parent pointer, child list) is written. The payload is referenced by pointer into the input arena. This is safe because the input arena outlives the restrict operation.

In benchmarks on an M2 MacBook Air, this fast path processes ~3.2 million nodes per second for a threadViewPost-like schema with 8 fields per post. The WASM target is slightly slower (~1.8M nodes/sec) due to bounds checking and the lack of SIMD, but still well above the 1M target.

7.5.4 When it gets expensive

The pipeline slows down when:

  • Deep contraction chains (step 3): Walking up to find the nearest surviving ancestor is O(depth). For deeply nested schemas with many pruned intermediate levels, this can be significant. panproto mitigates this by pre-computing a “skip ancestor” table during migration compilation.
  • Ambiguous resolution (step 4): If the resolver table is large (many parallel edges in the schema), the lookup has higher constant factors. This is rare in practice.
  • Large fans (step 5): Reconstructing fans with hundreds of children (wide SQL tables) requires scanning and filtering each fan. The cost is linear in fan width.

For all three cases, the pre-computation done during migration compilation amortizes the per-instance cost.

7.6 The view from 10,000 feet

The restrict operation is the point where theory meets bytes. The categorical framework—polynomial functors, W-types, pushout complements—is not an academic exercise. Each piece of theory corresponds to a concrete algorithmic step:

Theory Algorithm Purpose
Signature restriction (Abbott et al. 2005) anchor_surviving Decide which nodes to keep
Connected component reachable_from_root Remove disconnected survivors
Pushout complement (Ehrig et al. 2006) ancestor_contraction Re-parent orphaned nodes
Complement selection (Löwe 1993) resolve_edge Choose edge labels after contraction
Limit cone (Barr and Wells 1990) reconstruct_fans Reconstruct hyperedges
Precomposition (Spivak 2012) \(\Delta_F\) Restrict tabular instances

The W-type pipeline handles the case that matters most for modern protocols: recursive, tree-structured data encoded as JSON, CBOR, or protocol buffers. The functor restrict handles the SQL case. Together, they cover the data formats that panproto targets.

The algorithmic contribution of panproto is not any single step—each step is individually well-understood. The contribution is the pipeline: the specific sequence of steps, the compilation strategy that makes the common case fast, and the correctness argument that flows from the categorical semantics. Because each step has a precise algebraic meaning, the pipeline composes correctly by construction. There are no edge cases to discover by fuzzing (though we fuzz anyway). The edge cases are classified by the theory: they correspond to the conditions under which each step is non-trivial.

The next chapter shows how to run the pipeline in both directions—not just forward (source to view) but backward (view back to source)—using the lens framework. The complement captured during get is exactly the information discarded by the restrict pipeline, and put uses it to reconstruct the original instance.

7.7 Further reading

# Lifting Data {#sec-lifting}

The previous chapter established *when* a migration exists. This chapter is about *how* it works: the `restrict` operation that transforms data from one schema to another. This is the algorithmic heart of panproto.

There are two flavors of restriction, corresponding to the two kinds of schema that panproto supports. **W-type restrict** handles tree-structured data (JSON, CBOR, protocol buffers) by walking a five-step pipeline over the instance tree. **Functor restrict** handles tabular data (SQL) by precomposition along a schema morphism. Both are grounded in the same categorical idea—pulling data back along a map—but the concrete algorithms differ because trees and tables have different structure.

We will spend most of our time on the W-type pipeline, since it is the more intricate of the two and the one that handles the recursive, nested data formats that dominate modern protocol design.


## A worked example: `threadViewPost`

To make the pipeline concrete, we need a schema that exercises every step. The ATProto `threadViewPost` type is ideal: a post that contains an array of replies, each of which is itself a post. The recursion means the instance is a tree of arbitrary depth.

Suppose version 1 of the schema looks like this (simplified):

```
threadViewPost
├── post: postView
│   ├── uri: string
│   ├── author: profileViewBasic
│   │   ├── did: string
│   │   ├── handle: string
│   │   └── displayName: string
│   ├── text: string
│   └── likeCount: integer
└── replies: array<threadViewPost>
```

And version 2 drops `likeCount` from `postView` and drops `displayName` from `profileViewBasic`. A migration $m: S_2 \to S_1$ exists (both changes are field removals, which are always backward-compatible). The migration's vertex map tells us which vertices in $S_1$ survive in $S_2$.

Now consider a concrete instance—a thread with two replies, one of which has a nested reply:

```
root (threadViewPost)
├── post (postView)
│   ├── uri = "at://did:plc:abc/post/1"
│   ├── author (profileViewBasic)
│   │   ├── did = "did:plc:abc"
│   │   ├── handle = "alice.bsky.social"
│   │   └── displayName = "Alice"
│   ├── text = "Hello world"
│   └── likeCount = 42
└── replies
    ├── reply-1 (threadViewPost)
    │   ├── post (postView)
    │   │   ├── uri = "at://did:plc:def/post/2"
    │   │   ├── author (profileViewBasic)
    │   │   │   ├── did = "did:plc:def"
    │   │   │   ├── handle = "bob.bsky.social"
    │   │   │   └── displayName = "Bob"
    │   │   ├── text = "Nice post"
    │   │   └── likeCount = 7
    │   └── replies
    │       └── reply-1-1 (threadViewPost)
    │           └── ...
    └── reply-2 (threadViewPost)
        └── ...
```

This is the instance we will push through the five-step pipeline.


## W-type restrict: the five-step pipeline

The W-type restrict algorithm transforms an instance tree in five sequential steps. Each step has a clear algebraic meaning, and together they implement the restriction of a polynomial functor algebra along a morphism of polynomial functors. The correspondence to the theory is not decorative—it is what guarantees correctness.

### Step 1: `anchor_surviving` {#sec-step-anchor}

The first step is the simplest: walk every node in the instance and check whether its **anchor vertex** (the schema vertex that the node is an instance of) survives the migration's vertex map. If the anchor vertex is not in the image of the vertex map, the node is marked for removal.

```rust
/// Mark each node as surviving or pruned based on whether
/// its anchor vertex is in the migration's vertex map.
fn anchor_surviving(
    instance: &WTree,
    vertex_map: &HashMap<VertexId, VertexId>,
) -> Vec<bool> {
    instance.nodes.iter().map(|node| {
        vertex_map.values().any(|v| *v == node.anchor)
    }).collect()
}
```

This is signature restriction on polynomial algebras: given a polynomial $P = \sum_{i \in I} y^{B_i}$ and a sub-signature $J \subseteq I$, the restricted polynomial is $P|_J = \sum_{j \in J} y^{B_j}$. An algebra (instance) over $P$ restricts to an algebra over $P|_J$ by discarding the operations whose index falls outside $J$ [@abbott2005].

In our example, the vertices `likeCount` and `displayName` are not in the image of the vertex map (they were dropped in version 2). So every node anchored at `likeCount` or `displayName` is marked for removal:

- `root.post.likeCount` — pruned
- `root.post.author.displayName` — pruned
- `reply-1.post.likeCount` — pruned
- `reply-1.post.author.displayName` — pruned
- (and so on for every nested reply)

Everything else survives.

:::{.callout-note}
## Why "anchor"?

In the W-type formalization, each node in an instance tree sits over a vertex in the schema polynomial. The schema vertex is the node's *anchor* because it determines the node's shape: what children it can have, what edges connect it to its parent, and what data it carries. When we say a node's anchor survives, we mean the schema still has a place for that kind of node.
:::


### Step 2: `reachable_from_root` {#sec-step-reachable}

After marking survivors, some surviving nodes may be stranded: they are still marked as surviving, but they can only be reached from the root through pruned nodes. A sub-tree of a W-type is not necessarily a W-type—it may be disconnected.

Step 2 runs a breadth-first search from the root, traversing only through surviving nodes. Any surviving node not visited by the BFS is marked as unreachable and will not appear in the output.

```rust
/// BFS from root through surviving nodes. Returns the set of
/// node indices that are both surviving and reachable.
fn reachable_from_root(
    instance: &WTree,
    surviving: &[bool],
) -> HashSet<u32> {
    let mut visited = HashSet::new();
    let mut queue = VecDeque::new();

    if surviving[instance.root as usize] {
        queue.push_back(instance.root);
    }

    while let Some(idx) = queue.pop_front() {
        if !visited.insert(idx) {
            continue;
        }
        for child in instance.children(idx) {
            if surviving[child as usize] {
                queue.push_back(child);
            }
        }
    }

    visited
}
```

This is standard connected-component extraction. In our example it has no effect, because removing `likeCount` and `displayName` nodes does not disconnect anything—they are leaf nodes. But consider a different migration that removes `postView` while keeping `text`. The `text` nodes would survive step 1 but become unreachable in step 2, because their parent (`postView`) was pruned and no path from the root reaches them through survivors alone.

:::{.callout-note}
## When does step 2 matter?

Step 2 is a no-op whenever every pruned node is a leaf or when the surviving nodes form a connected sub-tree. It becomes essential when intermediate structural nodes are removed—for example, when an entire wrapper object is dropped from a schema but some of its children are "hoisted" to a higher level in a later step.
:::


### Step 3: `ancestor_contraction` {#sec-step-contraction}

This is the tree surgery step. When intermediate nodes are pruned, children that survived steps 1 and 2 need to be re-parented to their nearest surviving ancestor. The result is a new tree where every edge connects two surviving nodes.

```rust
/// For each surviving node, find its nearest surviving ancestor.
/// Returns a map from node index to new parent index.
fn ancestor_contraction(
    instance: &WTree,
    reachable: &HashSet<u32>,
) -> HashMap<u32, u32> {
    let mut new_parent = HashMap::new();

    for &idx in reachable {
        if idx == instance.root {
            continue;
        }
        // Walk up the original tree until we find a surviving ancestor
        let mut ancestor = instance.parent(idx);
        while !reachable.contains(&ancestor) {
            ancestor = instance.parent(ancestor);
        }
        new_parent.insert(idx, ancestor);
    }

    new_parent
}
```

In algebraic graph transformation terms, this is computing the pushout complement [@ehrig2006, §3.1]. We have a span $L \hookleftarrow K \hookrightarrow R$ where $L$ is the original tree, $K$ is the interface (surviving nodes with their original connectivity), and $R$ is the contracted tree. The pushout complement identifies how to glue $R$ into the surrounding context.

Here is the key visual intuition:

```{mermaid}
%%| label: fig-contraction-before
%%| fig-cap: "Before contraction: node B is marked for removal, but its child C survives."
graph TD
    A["A (root)"] --> B["B (removed)"]
    B --> C["C (surviving)"]
    B --> D["D (removed)"]
    A --> E["E (surviving)"]
```

After contraction, C is re-parented to A:

```{mermaid}
%%| label: fig-contraction-after
%%| fig-cap: "After contraction: C is now a direct child of A. B and D are gone."
graph TD
    A["A (root)"] --> C["C (re-parented)"]
    A --> E["E (surviving)"]
```

In our `threadViewPost` example, the pruned nodes (`likeCount`, `displayName`) are all leaves, so no contraction occurs. But imagine a migration that removes the `postView` wrapper while keeping `uri`, `author`, and `text` as direct children of `threadViewPost`. Then `postView` nodes are pruned, and their surviving children are re-parented to the `threadViewPost` node:

```
Before:                          After:
threadViewPost                   threadViewPost
├── postView (removed)    →     ├── uri
│   ├── uri                      ├── author
│   ├── author                   │   ├── did
│   │   ├── did                  │   └── handle
│   │   ├── handle               ├── text
│   │   └── displayName          └── replies
│   ├── text                         └── ...
│   └── likeCount
└── replies
    └── ...
```

:::{.callout-note}
## Contraction preserves depth order

A node's depth in the contracted tree is always less than or equal to its depth in the original tree. Contraction never pushes nodes deeper. This is a consequence of the fact that we only re-parent to *ancestors*, never to siblings or descendants.
:::


### Step 4: `resolve_edge` {#sec-step-resolve}

After contraction, each re-parented child has a new parent, but the edge connecting them needs a label from the target schema. In the original tree, the edge from A to C went through B: `A --(e1)--> B --(e2)--> C`. After removing B, we need a single edge label for the direct `A --> C` connection.

In a simple graph (at most one edge between any two vertices), this is trivial: there is only one choice. But schemas are often multigraphs—multiple edges can exist between the same pair of vertices (think of an object with two fields of the same type). In that case, the choice is ambiguous.

The **resolver table** provides the deterministic choice. It is part of the migration specification and maps each contracted path `(source_vertex, target_vertex, intermediate_path)` to a specific edge in the target schema.

```rust
/// Given a contracted parent-child pair, look up the correct
/// edge label from the resolver table.
fn resolve_edge(
    parent_anchor: VertexId,
    child_anchor: VertexId,
    contracted_path: &[VertexId],
    resolver: &ResolverTable,
) -> EdgeId {
    resolver
        .lookup(parent_anchor, child_anchor, contracted_path)
        .expect("migration guarantees a resolution exists")
}
```

This is exactly the pushout complement selection problem [@lowe1993]. When the pushout complement is not unique (which happens precisely when the interface graph has parallel edges), an additional choice is required to determine which complement to use. The resolver table encodes this choice. The existence checker from @sec-existence guarantees that a valid resolution exists for every contracted path, so `resolve_edge` never fails on a well-formed migration.

:::{.callout-note}
## When is resolution trivial?

For JSON-like schemas (which are trees, not multigraphs), resolution is always trivial: there is at most one edge between any pair of vertices in the schema. The resolver table is only needed for schemas where the same type appears in multiple positions under the same parent—protocol buffers with repeated message types, for instance.
:::


### Step 5: `reconstruct_fans` {#sec-step-fans}

The final step handles **hypergraph schemas**—schemas where an edge can connect a vertex to multiple children simultaneously. In practice, this means SQL: a table row is a hyperedge connecting a primary key to all its column values.

When some columns of a table are pruned, the surviving columns must be collected into a new, smaller hyperedge (a new "fan"). This is limit cone construction: the surviving columns form a sub-diagram, and the new fan is the limit of that sub-diagram in the category of fans [@barr1990, Ch. 4].

```rust
/// Reconstruct fans (hyperedges) after pruning some children.
/// Each surviving fan keeps only its surviving children and
/// receives a new fan ID from the target schema.
fn reconstruct_fans(
    instance: &WTree,
    reachable: &HashSet<u32>,
    fan_map: &HashMap<FanId, FanId>,
) -> Vec<Fan> {
    instance.fans.iter().filter_map(|fan| {
        let new_children: Vec<u32> = fan.children.iter()
            .copied()
            .filter(|c| reachable.contains(c))
            .collect();

        if new_children.is_empty() {
            return None;
        }

        let new_fan_id = fan_map.get(&fan.id)?;
        Some(Fan {
            id: *new_fan_id,
            parent: fan.parent,
            children: new_children,
        })
    }).collect()
}
```

For pure tree schemas (JSON, CBOR, protocol buffers), step 5 is a no-op: every edge connects exactly one parent to exactly one child, so there are no fans to reconstruct. The step exists because panproto's data model is general enough to handle both trees and hypergraphs uniformly.

:::{.callout-note}
## The five steps in summary

| Step | Operation | Algebraic meaning | When it matters |
|------|-----------|-------------------|-----------------|
| 1 | `anchor_surviving` | Signature restriction | Always |
| 2 | `reachable_from_root` | Connected component | Intermediate nodes pruned |
| 3 | `ancestor_contraction` | Pushout complement | Children of pruned nodes survive |
| 4 | `resolve_edge` | Complement selection | Multigraph schemas |
| 5 | `reconstruct_fans` | Limit cone | Hypergraph (SQL) schemas |

For a typical JSON schema migration (adding or removing fields), only step 1 does meaningful work. Steps 2–5 are structural no-ops. This is by design: the common case should be fast.
:::


## The pipeline in sequence

Let us trace our `threadViewPost` example through all five steps.

**Input:** An instance tree with nodes for `threadViewPost`, `postView`, `profileViewBasic`, `uri`, `author`, `did`, `handle`, `displayName`, `text`, `likeCount`, and `replies`, recursively nested to depth 3.

**Step 1 — `anchor_surviving`:** Mark `likeCount` and `displayName` nodes as pruned. All other nodes survive. In a thread with 3 posts, this marks 6 nodes for removal (one `likeCount` and one `displayName` per post).

**Step 2 — `reachable_from_root`:** BFS from root. All surviving nodes are reachable (the pruned nodes are leaves). No additional nodes are removed.

**Step 3 — `ancestor_contraction`:** No contraction needed. Every surviving node's parent is also surviving.

**Step 4 — `resolve_edge`:** No re-parenting occurred, so no edges need resolution. All original edges between surviving nodes carry over directly.

**Step 5 — `reconstruct_fans`:** The schema is a tree, not a hypergraph. No fans to reconstruct.

**Output:** The original instance tree, minus the `likeCount` and `displayName` leaf nodes. Every other node, edge, and value is preserved exactly.

This is the common case for protocol evolution: field removal produces a simple, fast restriction that touches only the affected leaves.


## Functor restrict ($\Delta_F$ precomposition) {#sec-functor-restrict}

For tabular data (SQL schemas), restriction takes a different form. Given a schema morphism $F: S_1 \to S_2$ and an instance $I: S_2 \to \mathbf{Set}$, the restricted instance is the precomposition $\Delta_F(I) = I \circ F : S_1 \to \mathbf{Set}$.

In concrete terms: if $F$ drops a table from the schema, the rows for that table vanish from the instance. If $F$ drops a column, that column disappears from every row. If $F$ renames a table, the rows are relabeled. The operation is purely structural—no data is computed, only rearranged.

```rust
/// Functor restriction: precompose an instance along a schema morphism.
fn restrict_functor<F, I>(morphism: &F, instance: &I) -> RestrictedInstance
where
    F: SchemaMorphism,
    I: Instance,
{
    let mut result = RestrictedInstance::new(morphism.source());
    for table in morphism.source().tables() {
        let target_table = morphism.map_table(table);
        for row in instance.rows(target_table) {
            let restricted_row = row.project(morphism.column_map(table));
            result.insert(table, restricted_row);
        }
    }
    result
}
```

Precomposition is always total: if $F$ is a well-typed schema morphism and $I$ is a well-typed instance, then $I \circ F$ is a well-typed instance over $S_1$. There are no existence conditions to check—the operation cannot fail [@spivak2012].

This simplicity is exactly why functor restriction is the right abstraction for SQL migrations. The complexity is not in the restriction itself but in constructing the morphism $F$, which is the job of the migration specification.

### Adjoints: $\Sigma_F$ and $\Pi_F$

The restriction functor $\Delta_F$ sits in the middle of an adjoint triple:

$$\Sigma_F \dashv \Delta_F \dashv \Pi_F$$

The left adjoint $\Sigma_F$ (pushforward, or left Kan extension) computes the "best approximation" of an $S_1$-instance as an $S_2$-instance by taking colimits. In SQL terms, this is roughly `UNION` across fibers. The right adjoint $\Pi_F$ (right Kan extension) computes the "most conservative approximation" by taking limits—roughly `JOIN` across fibers [@maclane1998, Ch. X].

Both adjoints are computationally more expensive than $\Delta_F$ and are outside panproto's current scope. The key point is that $\Delta_F$ alone gives us everything we need for the forward direction of a migration: projecting existing data onto a new schema.

:::{.callout-note}
## Why not $\Sigma_F$?

The pushforward $\Sigma_F$ would let us *add* data to an instance (e.g., populate a new required column with default values). This is sometimes called "data migration" in the database literature. panproto deliberately limits itself to $\Delta_F$ because restriction is lossless and reversible (via the complement—see @sec-lenses), while $\Sigma_F$ requires inventing data and is generally not reversible. When panproto needs to populate defaults, it does so through the lens complement mechanism rather than through $\Sigma_F$.
:::


## Performance {#sec-restrict-perf}

The restrict pipeline targets **>1 million records per second** for simple projections (field additions and removals) running in WASM. Several design decisions make this possible.

### Compiled migrations

When a migration is loaded, panproto pre-computes everything it can:

- **Surviving sets:** The set of schema vertices that survive is computed once and stored as a bitset. Step 1 becomes a single bitset lookup per node.
- **Remap tables:** Edge resolution (step 4) is pre-computed into a flat lookup table indexed by `(source_anchor, child_anchor)`. No hash map lookups at runtime.
- **Fan maps:** Fan reconstruction (step 5) is pre-computed into a mapping from old fan IDs to new fan IDs with pre-allocated child slots.

The migration specification is effectively compiled into a small program that the restrict pipeline executes.

### Arena allocation

Instance trees are allocated in a `bumpalo` arena. This has two benefits:

1. **Allocation is O(1):** Bump allocation is a pointer increment. No free-list management, no fragmentation.
2. **Deallocation is O(1):** The entire arena is freed at once when the instance is no longer needed. No tree of destructors to walk.

For the restrict pipeline specifically, the output tree is allocated in a fresh arena. Surviving nodes are written sequentially into the output arena, giving excellent cache locality.

### Zero-copy fast path

For the common case—no contraction (step 3 is a no-op), no fan reconstruction (step 5 is a no-op), and trivial edge resolution (step 4 is a no-op)—the pipeline reduces to:

1. Walk the input tree.
2. For each node, check the surviving bitset.
3. If surviving, copy the node to the output arena with a rewritten anchor field.

The node's payload (the actual data—strings, integers, booleans) is not copied. Only the structural metadata (anchor, parent pointer, child list) is written. The payload is referenced by pointer into the input arena. This is safe because the input arena outlives the restrict operation.

In benchmarks on an M2 MacBook Air, this fast path processes ~3.2 million nodes per second for a `threadViewPost`-like schema with 8 fields per post. The WASM target is slightly slower (~1.8M nodes/sec) due to bounds checking and the lack of SIMD, but still well above the 1M target.

### When it gets expensive

The pipeline slows down when:

- **Deep contraction chains** (step 3): Walking up to find the nearest surviving ancestor is O(depth). For deeply nested schemas with many pruned intermediate levels, this can be significant. panproto mitigates this by pre-computing a "skip ancestor" table during migration compilation.
- **Ambiguous resolution** (step 4): If the resolver table is large (many parallel edges in the schema), the lookup has higher constant factors. This is rare in practice.
- **Large fans** (step 5): Reconstructing fans with hundreds of children (wide SQL tables) requires scanning and filtering each fan. The cost is linear in fan width.

For all three cases, the pre-computation done during migration compilation amortizes the per-instance cost.


## The view from 10,000 feet {#sec-restrict-overview}

The restrict operation is the point where theory meets bytes. The categorical framework—polynomial functors, W-types, pushout complements—is not an academic exercise. Each piece of theory corresponds to a concrete algorithmic step:

| Theory | Algorithm | Purpose |
|--------|-----------|---------|
| Signature restriction [@abbott2005] | `anchor_surviving` | Decide which nodes to keep |
| Connected component | `reachable_from_root` | Remove disconnected survivors |
| Pushout complement [@ehrig2006] | `ancestor_contraction` | Re-parent orphaned nodes |
| Complement selection [@lowe1993] | `resolve_edge` | Choose edge labels after contraction |
| Limit cone [@barr1990] | `reconstruct_fans` | Reconstruct hyperedges |
| Precomposition [@spivak2012] | $\Delta_F$ | Restrict tabular instances |

The W-type pipeline handles the case that matters most for modern protocols: recursive, tree-structured data encoded as JSON, CBOR, or protocol buffers. The functor restrict handles the SQL case. Together, they cover the data formats that panproto targets.

The algorithmic contribution of panproto is not any single step—each step is individually well-understood. The contribution is the *pipeline*: the specific sequence of steps, the compilation strategy that makes the common case fast, and the correctness argument that flows from the categorical semantics. Because each step has a precise algebraic meaning, the pipeline composes correctly by construction. There are no edge cases to discover by fuzzing (though we fuzz anyway). The edge cases are *classified* by the theory: they correspond to the conditions under which each step is non-trivial.

The next chapter shows how to run the pipeline in both directions—not just forward (source to view) but backward (view back to source)—using the lens framework. The complement captured during `get` is exactly the information discarded by the restrict pipeline, and `put` uses it to reconstruct the original instance.


## Further reading

The W-type perspective on inductive data types originates with @gambino2004, who showed that W-types in locally cartesian closed categories correspond to initial algebras of polynomial functors. The container / interaction structure perspective of @abbott2005 gives the fibrational view that panproto uses for signature restriction.

The graph transformation approach to steps 3 and 4 follows @ehrig2006, particularly the treatment of pushout complements in adhesive categories. The non-uniqueness of pushout complements in the presence of parallel edges was first identified by @lowe1993 in the context of graph grammars. @drewes1997 extends this to hypergraph grammars, which is the foundation for step 5.

The functorial data migration framework ($\Delta_F$, $\Sigma_F$, $\Pi_F$) is due to @spivak2012. The general theory of Kan extensions that underlies the adjoint triple is covered in @maclane1998, Chapter X.

9  Breaking Changes

Schema evolution is a fact of life. Protocols grow new fields, deprecate old ones, tighten constraints, and occasionally restructure entirely. The question is not whether a schema will change but how to classify the change: is it safe to deploy, or will it break existing consumers?

panproto computes a structural diff between two schemas and classifies each change against the protocol’s theory-derived rules. The result is a compatibility verdict: fully compatible, backward compatible, or breaking. This chapter explains the classification, walks through concrete examples, and shows how to integrate the check into CI.

9.1 The diff

panproto’s diff algorithm operates on the schema’s graph structure—vertices, edges, and their attributes—rather than on a textual representation. Two schemas that are textually different but structurally identical (e.g., fields listed in a different order) produce an empty diff.

const oldSchema = atproto.schema()
  .vertex('post', 'record', { nsid: 'app.bsky.feed.post' })
  .vertex('post:body', 'object')
  .vertex('post:body.text', 'string')
  .edge('post', 'post:body', 'record-schema')
  .edge('post:body', 'post:body.text', 'prop', { name: 'text' })
  .constraint('post:body.text', 'maxLength', '3000')
  .build();

const newSchema = atproto.schema()
  .vertex('post', 'record', { nsid: 'app.bsky.feed.post' })
  .vertex('post:body', 'object')
  .vertex('post:body.text', 'string')
  .vertex('post:body.tags', 'array')
  .edge('post', 'post:body', 'record-schema')
  .edge('post:body', 'post:body.text', 'prop', { name: 'text' })
  .edge('post:body', 'post:body.tags', 'prop', { name: 'tags' })
  .constraint('post:body.text', 'maxLength', '3000')
  .build();

const report = panproto.diff(oldSchema, newSchema);
// report.compatibility: 'fully-compatible'
// report.changes: [{ kind: 'vertex-added', id: 'post:body.tags' }, ...]

The diff produces a list of change descriptors, each identifying:

  • What changed: a vertex, an edge, a constraint, or a structural relationship.
  • How it changed: added, removed, modified, or moved.
  • Where it changed: the path from the root to the affected element.

The change descriptors are the input to the compatibility classifier.

9.1.1 Structural diffing vs. textual diffing

A textual diff (like git diff) operates on serialized representations and is sensitive to formatting, ordering, and encoding choices. A structural diff operates on the mathematical object (the labeled graph) and is invariant to these surface-level differences.

This matters because two schema files can look very different textually while representing the same graph. Reordering fields in a JSON Schema, renaming internal $defs, or switching between inline and referenced definitions can all produce large textual diffs with zero structural changes. panproto’s diff sees through these differences.

NoteCanonicalization

Before diffing, panproto canonicalizes both schemas: it resolves all references, normalizes naming, sorts vertices and edges by a deterministic order, and strips metadata that does not affect the schema’s mathematical structure. The diff operates on canonical forms, ensuring that equivalent schemas always produce an empty diff.

9.2 Compatibility levels

panproto classifies schema changes into three levels. The classification is theory-derived: it follows directly from the existence conditions for migrations (Chapter 6) rather than from ad hoc rules.

9.2.1 Fully compatible

Both forward and backward migrations exist.

  • Old data can be read by new code (forward migration: old schema → new schema).
  • New data can be read by old code (backward migration: new schema → old schema).

This is the gold standard. Deploying a fully compatible change requires no coordination: old and new consumers can coexist indefinitely.

Examples: Renaming a field (with a migration that maps the old name to the new name). Adding an optional field with a default. Reordering fields.

9.2.2 Backward compatible

A forward migration exists, but no backward migration.

  • Old data can be read by new code.
  • New data may not be readable by old code.

This is acceptable for rolling deployments where consumers update before producers, or where old consumers can gracefully handle unknown data (e.g., by ignoring unrecognized fields).

Examples: Removing a field (old data is fine without it; new data lacks what old code expects). Adding a required field (old data can be given a default; new data has a field old code does not know about).

9.2.3 Breaking

No valid forward migration exists.

  • Existing data cannot be transformed to conform to the new schema.

This is the red line. A breaking change means that some existing data will be rejected or misinterpreted by code targeting the new schema. Deploying a breaking change without a data migration plan will cause failures.

Examples: Tightening a constraint below existing values. Changing a field’s type incompatibly. Removing a vertex that other vertices depend on.

NoteThe classification is decidable

The compatibility classification reduces to checking migration existence in both directions. As established in Chapter 6, migration existence is decidable for the class of schemas that panproto supports. Therefore, the compatibility classification is also decidable. There is no approximation or heuristic involved: panproto gives a definitive answer.

9.3 The connection to existence conditions

The compatibility classifier does not maintain its own set of rules. It delegates entirely to the migration existence checker from Chapter 6.

Given schemas \(S_\text{old}\) and \(S_\text{new}\):

  1. Check forward existence: Does a migration \(m: S_\text{old} \to S_\text{new}\) exist? If yes, old data can be transformed to the new schema.
  2. Check backward existence: Does a migration \(m': S_\text{new} \to S_\text{old}\) exist? If yes, new data can be transformed to the old schema.

The classification follows:

Forward exists Backward exists Classification
Yes Yes Fully compatible
Yes No Backward compatible
No Yes Forward compatible only
No No Breaking

The “forward compatible only” case (new data can be read by old code, but old data cannot be read by new code) is theoretically possible but rare in practice. It would arise from changes like loosening a constraint: old code can handle the wider range, but new code might receive data that was valid under the old, tighter constraint. panproto reports this case but treats it as breaking for safety, since the typical deployment assumption is that old data must be readable by new code.

NoteWhy not just check the diff?

Some schema registries classify changes by pattern-matching on the diff: “adding a field is backward compatible,” “removing a field is breaking,” and so on. This works for simple cases but fails for composed changes. Adding an optional field and tightening a constraint on an unrelated field is not simply the conjunction of “backward compatible” and “breaking”—the interaction between changes can produce surprising results.

panproto avoids this problem by not classifying individual changes at all. It classifies the migration as a whole by checking existence. The diff is useful for reporting what changed (so humans can understand the result), but it does not drive the classification.

9.4 Examples

Let us walk through four concrete changes and see how panproto classifies them.

9.4.1 Example 1: Adding an optional field

Change: Add an optional labels field (type: array<string>, default: []) to postView.

Forward migration (\(S_\text{old} \to S_\text{new}\)): Exists. Old data lacks labels, but the default value [] fills it in. The migration adds a leaf node with the default value at every postView instance.

Backward migration (\(S_\text{new} \to S_\text{old}\)): Exists. New data has labels, which the backward migration discards (it is a field removal, which the restrict pipeline handles in step 1).

Classification: Fully compatible.

Both directions work. Old consumers ignore the new field (it is not in their schema). New consumers see the default value for old data. This is the most common kind of protocol evolution.

Schema diff:
  + postView.labels: array<string> (optional, default: [])

Compatibility: FULLY COMPATIBLE
  Forward migration:  ✓ exists (add default)
  Backward migration: ✓ exists (drop field)

9.4.2 Example 2: Removing a field

Change: Remove the likeCount field from postView.

Forward migration (\(S_\text{old} \to S_\text{new}\)): Exists. Old data has likeCount, which the forward migration discards. The restrict pipeline handles this in step 1.

Backward migration (\(S_\text{new} \to S_\text{old}\)): Does not exist. New data lacks likeCount, and the old schema requires it (it is not optional). There is no way to conjure a valid likeCount value from nothing.

Classification: Backward compatible.

Old data can be read by new code (just drop likeCount). But new data cannot be read by old code (the required field is missing). Consumers must be updated before producers stop sending likeCount.

Schema diff:
  - postView.likeCount: integer

Compatibility: BACKWARD COMPATIBLE
  Forward migration:  ✓ exists (drop field)
  Backward migration: ✗ no valid migration (required field missing)
NoteWhat if the removed field were optional?

If likeCount had been optional in the old schema, then the backward migration would exist: new data simply omits the field, which is valid because the old schema does not require it. The classification would be fully compatible.

This is why marking fields as optional from the start is good protocol hygiene. An optional field can be added or removed without breaking compatibility in either direction.

9.4.3 Example 3: Tightening a constraint

Change: Reduce the maximum length of postView.text from 3000 characters to 300 characters.

Forward migration (\(S_\text{old} \to S_\text{new}\)): Does not exist. Old data may contain text values up to 3000 characters, which violate the new 300-character limit. The migration would need to truncate text, but truncation is not a valid structural transformation—it changes the data’s meaning.

Backward migration (\(S_\text{new} \to S_\text{old}\)): Exists. New data has text values up to 300 characters, which satisfy the old 3000-character limit (300 ≤ 3000).

Classification: Breaking.

Existing data may violate the new constraint. There is no structural migration that can make 3000-character posts fit in 300 characters without semantic loss.

Schema diff:
  ~ postView.text: maxLength 3000 → 300

Compatibility: BREAKING
  Forward migration:  ✗ constraint violation (existing data may exceed new limit)
  Backward migration: ✓ exists (new constraint is stricter, old is satisfied)

This is a case where the diff alone might mislead. “Changing a maxLength” sounds like a minor modification. But the direction of the change matters: tightening is breaking, loosening is (backward) compatible.

9.4.4 Example 4: Changing a vertex kind

Change: Change postView.likeCount from integer to string.

Forward migration (\(S_\text{old} \to S_\text{new}\)): Does not exist. Old data has integer values (42), which are not valid strings in the target schema. While some type coercions are possible (integer to string via toString()), panproto does not perform semantic coercions during structural migration—the types must be structurally compatible.

Backward migration (\(S_\text{new} \to S_\text{old}\)): Does not exist. New data has string values ("42" or "many"), which are not valid integers.

Classification: Breaking.

Neither direction works. The types are structurally incompatible.

Schema diff:
  ~ postView.likeCount: integer → string

Compatibility: BREAKING
  Forward migration:  ✗ type mismatch (integer ≠ string)
  Backward migration: ✗ type mismatch (string ≠ integer)
NoteCoercions and combinators

If you want to change a type with a known coercion (e.g., integer → string via formatting), you can use the coerceType combinator from Section 8.5 to build a manual migration. The combinator includes both directions (format and parse), so it satisfies the lens laws. But panproto will not infer this coercion automatically—you must explicitly declare it. The compatibility checker reports the structural classification; manual migrations can override it.

9.5 Composed changes

Real schema evolutions rarely consist of a single change. A new version might simultaneously add fields, remove fields, tighten one constraint, and loosen another. The compatibility of the composed change is not the “worst” individual classification—it is determined by checking migration existence for the composed schemas.

Consider a version bump that:

  1. Adds an optional labels field (fully compatible individually)
  2. Removes likeCount (backward compatible individually)
  3. Tightens text maxLength from 3000 to 300 (breaking individually)

The composed change is breaking, because the forward migration does not exist (the constraint tightening prevents it). But this is not simply because “one change is breaking.” It is because the existence checker, applied to the full old and new schemas, finds no valid forward migration.

Now consider a different composition:

  1. Adds an optional labels field (fully compatible)
  2. Removes likeCount (backward compatible)
  3. Loosens text maxLength from 300 to 3000 (backward compatible: new constraint is weaker)

The composed change is backward compatible: the forward migration exists (drop likeCount, add labels with default, existing text satisfies the loosened constraint), but the backward migration does not (new data may have text up to 3000 characters, which exceeds the old 300-character limit).

The point: composition of compatibility levels is not a lattice operation. It depends on the specific changes and how they interact. Checking migration existence on the composed schemas is the only reliable method.

9.6 The diff report

panproto produces a human-readable diff report that combines the structural diff with the compatibility classification. The report is designed for code review: a developer should be able to read it and understand both what changed and whether it is safe.

schema diff v1/schema.json v2/schema.json

Schema: app.bsky.feed.defs#threadViewPost

Changes:
  + postView.labels: array<string> (optional, default: [])
  - postView.likeCount: integer
  ~ postView.text: maxLength 300 → 3000
  + profileViewBasic.avatarUrl: string (optional)

Compatibility: BACKWARD COMPATIBLE

  Forward migration:  ✓ exists
    - labels: filled with default []
    - likeCount: dropped
    - text: constraint loosened (valid)
    - avatarUrl: filled with default null

  Backward migration: ✗ does not exist
    - likeCount: required field missing in new schema
    - text: new data may exceed old maxLength (3000 > 300)

Recommendation: Safe to deploy if consumers update before producers.

The report lists every change, explains why each direction does or does not work, and gives a deployment recommendation.

9.7 CI integration

Schema compatibility checks belong in CI, not in code review checklists. panproto provides a CLI command that exits with a non-zero status for breaking changes, making it easy to integrate into any CI system.

9.7.1 GitHub Actions

name: Schema Compatibility Check

on:
  pull_request:
    paths:
      - 'schemas/**'

jobs:
  check-compat:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # Need history to get the base schema

      - name: Install panproto
        run: cargo install panproto-cli

      - name: Get base schema
        run: git show origin/main:schemas/schema.json > /tmp/old-schema.json

      - name: Check compatibility
        run: |
          schema diff /tmp/old-schema.json schemas/schema.json

      - name: Block breaking changes
        run: |
          schema check \
            --src /tmp/old-schema.json \
            --tgt schemas/schema.json \
            --level backward-compatible

The schema check command accepts a --level flag:

  • --level fully-compatible: Fail unless the change is fully compatible in both directions.
  • --level backward-compatible: Fail unless at least a forward migration exists (the default).
  • --level breaking: Never fail (only report). Useful for auditing.

9.7.2 CLI usage

For local development, the same commands work without CI:

# Show the diff and classification
schema diff old-schema.json new-schema.json

# Check against a specific compatibility level (exit code 0 = pass, 1 = fail)
schema check --src old-schema.json --tgt new-schema.json --level backward-compatible

# Output as JSON for programmatic consumption
schema diff old-schema.json new-schema.json --format json

9.7.3 Schema registry integration

For teams that use a schema registry, panproto can check compatibility against the latest registered version:

# Check against the latest version in the registry
schema check \
  --src "registry://app.bsky.feed.defs#threadViewPost@latest" \
  --tgt schemas/schema.json

This integrates with the registry’s version history, so you always check against what is actually deployed rather than what is on the main branch.

NoteBlocking vs. warning

Not every team wants to block on backward compatibility. Some protocols (internal APIs with synchronized deployments) can tolerate breaking changes as long as they are intentional. panproto supports this with the --level flag: set it to breaking to get reports without blocking, or add an --allow-breaking flag to specific PRs via a label-based CI override.

The important thing is that the check runs and the result is visible. Even if you do not block, knowing that a change is breaking before it ships is valuable.

9.8 The graph-theoretic foundation

The compatibility classification rests on two theoretical results.

9.8.1 Existence as a graph property

A migration from \(S_\text{old}\) to \(S_\text{new}\) exists if and only if certain graph-theoretic conditions are satisfied: the vertex map must be a valid graph homomorphism, the edge map must respect the boundary operators, and the constraint maps must be monotone in the appropriate direction. These conditions are decidable for finite graphs, which is why panproto can give definitive compatibility verdicts (Ehrig et al. 2006).

The key insight from algebraic graph transformation theory is that the existence of a pushout complement (which underlies the restrict pipeline’s step 3) can be characterized purely in terms of the gluing condition: a condition on the interface between the part being removed and the part being kept. When the gluing condition is satisfied, the pushout complement exists and is unique (for adhesive categories). When it is not satisfied, no valid migration exists.

9.8.2 Adhesive categories and decidability

The schemas that panproto supports form an adhesive category (Lack and Sobociński 2005). In an adhesive category, pushouts along monomorphisms exist and have good properties: they are stable under pullback, and van Kampen squares exist. The practical consequence is that the existence of pushout complements—and therefore the existence of migrations—is decidable by checking a finite set of local conditions.

This is not true for arbitrary categories. In a general category, checking whether a pushout complement exists can be undecidable. The restriction to adhesive categories is what makes panproto’s compatibility checker sound and complete: it never gives a false positive (declaring a change compatible when it is not) and never gives a false negative (declaring a change breaking when a valid migration exists).

NoteSoundness and completeness

Sound: If panproto says a forward migration exists, it does. You can trust a “fully compatible” or “backward compatible” verdict.

Complete: If panproto says no migration exists, none does. You cannot work around a “breaking” verdict by being clever with the migration specification (though you can use manual combinators to define a non-structural migration that panproto does not reason about).

Soundness and completeness are properties of the existence checker, not of the diff. The diff is a reporting tool; the existence checker is the decision procedure.

9.9 Versioning strategies

The compatibility classification informs but does not dictate versioning strategy. Different protocols adopt different conventions.

9.9.1 Semantic versioning

Map compatibility levels to semver:

  • Fully compatible → patch or minor version bump
  • Backward compatible → minor version bump
  • Breaking → major version bump

This is the most common approach and works well for protocols with external consumers who cannot be forced to update.

9.9.2 Chronological versioning

Some protocols (ATProto’s Lexicon) use date-based or sequential version numbers without semver semantics. The compatibility level is still useful for deployment decisions, but it does not drive the version number.

9.9.3 Continuous evolution

Some internal protocols avoid versioning entirely, relying on the lens mechanism (Chapter 8) to keep old and new consumers working simultaneously. In this model, every change is deployed with its migration, and the compatibility level determines whether old consumers need any changes:

  • Fully compatible: Deploy freely.
  • Backward compatible: Deploy new consumers first, then new producers.
  • Breaking: Deploy a data migration, then new consumers, then new producers.

The compatibility checker ensures that the deployment plan matches the actual compatibility of the change.

9.10 Putting it all together

The breaking changes workflow in panproto connects every concept from the preceding chapters:

  1. Schemas (Chapters 1–2) define the structure being evolved.
  2. Theories (Chapters 3–4) constrain which evolutions are valid.
  3. Existence conditions (Chapter 6) determine whether a migration is possible.
  4. The restrict pipeline (Chapter 7) implements the forward migration.
  5. Lenses (Chapter 8) make the migration bidirectional.
  6. The compatibility classifier (this chapter) decides whether the change is safe to deploy.

The classifier is the user-facing endpoint of the theory. A developer does not need to understand polynomial functors or pushout complements to use schema check. They see a clear verdict—fully compatible, backward compatible, or breaking—backed by a proof-strength guarantee from the underlying mathematics.

This is the design philosophy of panproto: the theory is deep, but the interface is simple. The complexity lives in the library, not in the developer’s head.

9.11 Further reading

(2006), particularly the treatment of pushout complements and the gluing condition in the context of typed attributed graph grammars. The decidability of the compatibility classification relies on the adhesive category framework of Lack and Sobociński (2005), which provides the categorical conditions under which pushout complement existence is decidable.

# Breaking Changes {#sec-breaking}

Schema evolution is a fact of life. Protocols grow new fields, deprecate old ones, tighten constraints, and occasionally restructure entirely. The question is not *whether* a schema will change but *how to classify the change*: is it safe to deploy, or will it break existing consumers?

panproto computes a structural diff between two schemas and classifies each change against the protocol's theory-derived rules. The result is a compatibility verdict: fully compatible, backward compatible, or breaking. This chapter explains the classification, walks through concrete examples, and shows how to integrate the check into CI.

## The connection to existence conditions

The compatibility classifier does not maintain its own set of rules. It delegates entirely to the migration existence checker from @sec-existence.

Given schemas $S_\text{old}$ and $S_\text{new}$:

1. **Check forward existence:** Does a migration $m: S_\text{old} \to S_\text{new}$ exist? If yes, old data can be transformed to the new schema.
2. **Check backward existence:** Does a migration $m': S_\text{new} \to S_\text{old}$ exist? If yes, new data can be transformed to the old schema.

The classification follows:

| Forward exists | Backward exists | Classification |
|:-:|:-:|:--|
| Yes | Yes | Fully compatible |
| Yes | No | Backward compatible |
| No | Yes | Forward compatible only |
| No | No | Breaking |

The "forward compatible only" case (new data can be read by old code, but old data cannot be read by new code) is theoretically possible but rare in practice. ## Examples {#sec-compat-examples}

Let us walk through four concrete changes and see how panproto classifies them.

### Example 1: Adding an optional field

**Change:** Add an optional `labels` field (type: `array<string>`, default: `[]`) to `postView`.

**Forward migration ($S_\text{old} \to S_\text{new}$):** Exists. Old data lacks `labels`, but the default value `[]` fills it in. The migration adds a leaf node with the default value at every `postView` instance.

**Backward migration ($S_\text{new} \to S_\text{old}$):** Exists. New data has `labels`, which the backward migration discards (it is a field removal, which the restrict pipeline handles in step 1).

**Classification: Fully compatible.**

Both directions work. Old consumers ignore the new field (it is not in their schema). New consumers see the default value for old data. This is the most common kind of protocol evolution.

```
Schema diff:
  + postView.labels: array<string> (optional, default: [])

Compatibility: FULLY COMPATIBLE
  Forward migration:  ✓ exists (add default)
  Backward migration: ✓ exists (drop field)
```


### Example 2: Removing a field

**Change:** Remove the `likeCount` field from `postView`.

**Forward migration ($S_\text{old} \to S_\text{new}$):** Exists. Old data has `likeCount`, which the forward migration discards. The restrict pipeline handles this in step 1.

**Backward migration ($S_\text{new} \to S_\text{old}$):** Does not exist. New data lacks `likeCount`, and the old schema requires it (it is not optional). There is no way to conjure a valid `likeCount` value from nothing.

**Classification: Backward compatible.**

Old data can be read by new code (just drop `likeCount`). But new data cannot be read by old code (the required field is missing). Consumers must be updated before producers stop sending `likeCount`.

```
Schema diff:
  - postView.likeCount: integer

Compatibility: BACKWARD COMPATIBLE
  Forward migration:  ✓ exists (drop field)
  Backward migration: ✗ no valid migration (required field missing)
```

:::{.callout-note}
## What if the removed field were optional?

If `likeCount` had been optional in the old schema, then the backward migration would exist: new data simply omits the field, which is valid because the old schema does not require it. The classification would be fully compatible.

This is why marking fields as optional from the start is good protocol hygiene. An optional field can be added or removed without breaking compatibility in either direction.
:::


### Example 3: Tightening a constraint

**Change:** Reduce the maximum length of `postView.text` from 3000 characters to 300 characters.

**Forward migration ($S_\text{old} \to S_\text{new}$):** Does not exist. Old data may contain `text` values up to 3000 characters, which violate the new 300-character limit. The migration would need to truncate text, but truncation is not a valid structural transformation—it changes the data's meaning.

**Backward migration ($S_\text{new} \to S_\text{old}$):** Exists. New data has `text` values up to 300 characters, which satisfy the old 3000-character limit (300 ≤ 3000).

**Classification: Breaking.**

Existing data may violate the new constraint. There is no structural migration that can make 3000-character posts fit in 300 characters without semantic loss.

```
Schema diff:
  ~ postView.text: maxLength 3000 → 300

Compatibility: BREAKING
  Forward migration:  ✗ constraint violation (existing data may exceed new limit)
  Backward migration: ✓ exists (new constraint is stricter, old is satisfied)
```

This is a case where the diff alone might mislead. "Changing a maxLength" sounds like a minor modification. But the *direction* of the change matters: tightening is breaking, loosening is (backward) compatible.


### Example 4: Changing a vertex kind

**Change:** Change `postView.likeCount` from `integer` to `string`.

**Forward migration ($S_\text{old} \to S_\text{new}$):** Does not exist. Old data has integer values (`42`), which are not valid strings in the target schema. While some type coercions are possible (integer to string via `toString()`), panproto does not perform semantic coercions during structural migration—the types must be structurally compatible.

**Backward migration ($S_\text{new} \to S_\text{old}$):** Does not exist. New data has string values (`"42"` or `"many"`), which are not valid integers.

**Classification: Breaking.**

Neither direction works. The types are structurally incompatible.

```
Schema diff:
  ~ postView.likeCount: integer → string

Compatibility: BREAKING
  Forward migration:  ✗ type mismatch (integer ≠ string)
  Backward migration: ✗ type mismatch (string ≠ integer)
```

:::{.callout-note}
## Coercions and combinators

If you *want* to change a type with a known coercion (e.g., integer → string via formatting), you can use the `coerceType` combinator from @sec-combinators to build a manual migration. The combinator includes both directions (format and parse), so it satisfies the lens laws. But panproto will not infer this coercion automatically—you must explicitly declare it. The compatibility checker reports the *structural* classification; manual migrations can override it.
:::


## Composed changes {#sec-composed}

Real schema evolutions rarely consist of a single change. A new version might simultaneously add fields, remove fields, tighten one constraint, and loosen another. The compatibility of the composed change is not the "worst" individual classification—it is determined by checking migration existence for the composed schemas.

Consider a version bump that:

1. Adds an optional `labels` field (fully compatible individually)
2. Removes `likeCount` (backward compatible individually)
3. Tightens `text` maxLength from 3000 to 300 (breaking individually)

The composed change is **breaking**, because the forward migration does not exist (the constraint tightening prevents it). But this is not simply because "one change is breaking." It is because the existence checker, applied to the full old and new schemas, finds no valid forward migration.

Now consider a different composition:

1. Adds an optional `labels` field (fully compatible)
2. Removes `likeCount` (backward compatible)
3. *Loosens* `text` maxLength from 300 to 3000 (backward compatible: new constraint is weaker)

The composed change is **backward compatible**: the forward migration exists (drop `likeCount`, add `labels` with default, existing text satisfies the loosened constraint), but the backward migration does not (new data may have text up to 3000 characters, which exceeds the old 300-character limit).

The point: composition of compatibility levels is not a lattice operation. It depends on the specific changes and how they interact. Checking migration existence on the composed schemas is the only reliable method.


## The diff report {#sec-diff-report}

panproto produces a human-readable diff report that combines the structural diff with the compatibility classification. The report is designed for code review: a developer should be able to read it and understand both *what* changed and *whether it is safe*.

```
schema diff v1/schema.json v2/schema.json

Schema: app.bsky.feed.defs#threadViewPost

Changes:
  + postView.labels: array<string> (optional, default: [])
  - postView.likeCount: integer
  ~ postView.text: maxLength 300 → 3000
  + profileViewBasic.avatarUrl: string (optional)

Compatibility: BACKWARD COMPATIBLE

  Forward migration:  ✓ exists
    - labels: filled with default []
    - likeCount: dropped
    - text: constraint loosened (valid)
    - avatarUrl: filled with default null

  Backward migration: ✗ does not exist
    - likeCount: required field missing in new schema
    - text: new data may exceed old maxLength (3000 > 300)

Recommendation: Safe to deploy if consumers update before producers.
```

The report lists every change, explains why each direction does or does not work, and gives a deployment recommendation.


## CI integration {#sec-ci}

Schema compatibility checks belong in CI, not in code review checklists. panproto provides a CLI command that exits with a non-zero status for breaking changes, making it easy to integrate into any CI system.

### GitHub Actions

```yaml
name: Schema Compatibility Check

on:
  pull_request:
    paths:
      - 'schemas/**'

jobs:
  check-compat:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # Need history to get the base schema

      - name: Install panproto
        run: cargo install panproto-cli

      - name: Get base schema
        run: git show origin/main:schemas/schema.json > /tmp/old-schema.json

      - name: Check compatibility
        run: |
          schema diff /tmp/old-schema.json schemas/schema.json

      - name: Block breaking changes
        run: |
          schema check \
            --src /tmp/old-schema.json \
            --tgt schemas/schema.json \
            --level backward-compatible
```

The `schema check` command accepts a `--level` flag:

- `--level fully-compatible`: Fail unless the change is fully compatible in both directions.
- `--level backward-compatible`: Fail unless at least a forward migration exists (the default).
- `--level breaking`: Never fail (only report). Useful for auditing.

### CLI usage

For local development, the same commands work without CI:

```bash
# Show the diff and classification
schema diff old-schema.json new-schema.json

# Check against a specific compatibility level (exit code 0 = pass, 1 = fail)
schema check --src old-schema.json --tgt new-schema.json --level backward-compatible

# Output as JSON for programmatic consumption
schema diff old-schema.json new-schema.json --format json
```

### Schema registry integration

For teams that use a schema registry, panproto can check compatibility against the latest registered version:

```bash
# Check against the latest version in the registry
schema check \
  --src "registry://app.bsky.feed.defs#threadViewPost@latest" \
  --tgt schemas/schema.json
```

This integrates with the registry's version history, so you always check against what is actually deployed rather than what is on the main branch.

:::{.callout-note}
## Blocking vs. warning

Not every team wants to block on backward compatibility. Some protocols (internal APIs with synchronized deployments) can tolerate breaking changes as long as they are intentional. panproto supports this with the `--level` flag: set it to `breaking` to get reports without blocking, or add an `--allow-breaking` flag to specific PRs via a label-based CI override.

The important thing is that the check runs and the result is visible. Even if you do not block, knowing that a change is breaking before it ships is valuable.
:::


## The graph-theoretic foundation {#sec-graph-theory}

The compatibility classification rests on two theoretical results.

### Existence as a graph property

A migration from $S_\text{old}$ to $S_\text{new}$ exists if and only if certain graph-theoretic conditions are satisfied: the vertex map must be a valid graph homomorphism, the edge map must respect the boundary operators, and the constraint maps must be monotone in the appropriate direction. These conditions are decidable for finite graphs, which is why panproto can give definitive compatibility verdicts [@ehrig2006].

The key insight from algebraic graph transformation theory is that the existence of a pushout complement (which underlies the restrict pipeline's step 3) can be characterized purely in terms of the **gluing condition**: a condition on the interface between the part being removed and the part being kept. When the gluing condition is satisfied, the pushout complement exists and is unique (for adhesive categories). When it is not satisfied, no valid migration exists.

### Adhesive categories and decidability

The schemas that panproto supports form an **adhesive category** [@lack2005]. In an adhesive category, pushouts along monomorphisms exist and have good properties: they are stable under pullback, and van Kampen squares exist. The practical consequence is that the existence of pushout complements—and therefore the existence of migrations—is decidable by checking a finite set of local conditions.

This is not true for arbitrary categories. In a general category, checking whether a pushout complement exists can be undecidable. The restriction to adhesive categories is what makes panproto's compatibility checker sound and complete: it never gives a false positive (declaring a change compatible when it is not) and never gives a false negative (declaring a change breaking when a valid migration exists).

:::{.callout-note}
## Soundness and completeness

**Sound:** If panproto says a forward migration exists, it does. You can trust a "fully compatible" or "backward compatible" verdict.

**Complete:** If panproto says no migration exists, none does. You cannot work around a "breaking" verdict by being clever with the migration specification (though you can use manual combinators to define a non-structural migration that panproto does not reason about).

Soundness and completeness are properties of the *existence checker*, not of the diff. The diff is a reporting tool; the existence checker is the decision procedure.
:::


## Versioning strategies {#sec-versioning}

The compatibility classification informs but does not dictate versioning strategy. Different protocols adopt different conventions.

### Semantic versioning

Map compatibility levels to semver:

- **Fully compatible** → patch or minor version bump
- **Backward compatible** → minor version bump
- **Breaking** → major version bump

This is the most common approach and works well for protocols with external consumers who cannot be forced to update.

### Chronological versioning

Some protocols (ATProto's Lexicon) use date-based or sequential version numbers without semver semantics. The compatibility level is still useful for deployment decisions, but it does not drive the version number.

### Continuous evolution

Some internal protocols avoid versioning entirely, relying on the lens mechanism (@sec-lenses) to keep old and new consumers working simultaneously. In this model, every change is deployed with its migration, and the compatibility level determines whether old consumers need any changes:

- **Fully compatible:** Deploy freely.
- **Backward compatible:** Deploy new consumers first, then new producers.
- **Breaking:** Deploy a data migration, then new consumers, then new producers.

The compatibility checker ensures that the deployment plan matches the actual compatibility of the change.


## Putting it all together

The breaking changes workflow in panproto connects every concept from the preceding chapters:

1. **Schemas** (Chapters 1--2) define the structure being evolved.
2. **Theories** (Chapters 3--4) constrain which evolutions are valid.
3. **Existence conditions** (@sec-existence) determine whether a migration is possible.
4. **The restrict pipeline** (@sec-lifting) implements the forward migration.
5. **Lenses** (@sec-lenses) make the migration bidirectional.
6. **The compatibility classifier** (this chapter) decides whether the change is safe to deploy.

The classifier is the user-facing endpoint of the theory. A developer does not need to understand polynomial functors or pushout complements to use `schema check`. They see a clear verdict—fully compatible, backward compatible, or breaking—backed by a proof-strength guarantee from the underlying mathematics.

This is the design philosophy of panproto: the theory is deep, but the interface is simple. The complexity lives in the library, not in the developer's head.


## Further reading

The graph-theoretic foundation for schema compatibility draws on @ehrig2006, particularly the treatment of pushout complements and the gluing condition in the context of typed attributed graph grammars. The decidability of the compatibility classification relies on the adhesive category framework of @lack2005, which provides the categorical conditions under which pushout complement existence is decidable.

For the practical aspects of schema compatibility checking in production systems, the Confluent Schema Registry documentation provides useful context on how compatibility levels are used in event-streaming architectures. panproto's classification is more fine-grained (because it reasons about structural migrations rather than syntactic patterns), but the deployment strategies are similar.
